Part 1 explained the fan-out + supervisor pattern in the abstract. This is the version with code: a code reviewer where three specialized agents look at the same file at once and a fourth merges their findings into a scored report.

The architecture, end to end

The whole flow fits in one line: you paste code (or a GitHub URL), and the system extracts metadata, fires three reviewers in parallel, and a supervisor consolidates everything into a 0-100 scored report, all streamed live. None of this needs a database: it's a single Next.js route emitting SSE events.

The real stack: Next.js 16.2.5 (App Router), AI SDK v6 with generateObject, Zod 4 for structured output, and the models over Vercel's AI Gateway: claude-sonnet-4.6 for the three specialists and the cheaper claude-haiku-4.5 for the supervisor.

Figure 1 · The full flow. Input and metadata, three specialists in parallel, a supervisor that consolidates, and a report, all emitted live over a single SSE endpoint.

How I built it, step by step

The models, in one place. Everything goes through Vercel's AI Gateway with a single key. Two models: the good one to find problems, the cheap one to consolidate.

import { createGateway } from '@ai-sdk/gateway';

const gateway = createGateway({ apiKey: process.env.AI_GATEWAY_API_KEY! });

// The 3 specialists: best reasoning to find problems
export const REVIEW_MODEL = gateway('anthropic/claude-sonnet-4.6');
// The supervisor: synthesis + scoring, cheaper
export const SUPERVISOR_MODEL = gateway('anthropic/claude-haiku-4.5');

One specialist = one role + one schema. The three agents are the same function with a different system prompt. The security one looks for injection, secrets, XSS; it returns a list of findings that Zod validates. Notice the prompt caps the length (title max 10 words) so the output is comparable across agents.

const SYSTEM = `You are a security code reviewer. Analyze the code and find
vulnerabilities: SQL/command injection, XSS, hardcoded secrets,
insecure deserialization, missing validation at boundaries.
If the code is secure, return an empty array.
Be concise: title max 10 words, description max 50.`;

export async function runSecurityAgent(code: string, meta: CodeMetadata) {
  const { object } = await generateObject({
    model: REVIEW_MODEL,
    schema: FindingArraySchema,        // Zod: forces the output shape
    system: SYSTEM,
    prompt: `Language: ${meta.lang}

Code:
${code}`,
  });
  return object;                        // Finding[] — already validated
}

The Zod schemas are the contract between agents. Without a schema, the supervisor would have to parse free text from three different agents. With generateObject + Zod, each specialist returns exactly the same shape, and the supervisor receives data, not prose.

export const FindingSchema = z.object({
  severity: z.enum(['critical', 'high', 'medium', 'low', 'info']),
  category: z.enum(['security', 'performance', 'maintainability']),
  title: z.string(),
  line: z.number().int().positive().optional(),
  description: z.string(),
  suggestion: z.string(),
  codeExample: z.object({ before: z.string(), after: z.string() }).optional(),
});

export const ReviewReportSchema = z.object({
  score: z.number().int().min(0).max(100),
  summary: z.string(),
  recommendation: z.string(),
  findings: z.array(FindingSchema),
});

The orchestration: the real fan-out. Here's the heart. The three agents fire with Promise.all, so they run at the same time and total time is the slowest one's, not the sum. A detail that looks minor and isn't: I announce all three as "running" before the await, so the live timeline shows real parallelism, not a queue.

// All 3 announced BEFORE awaiting — otherwise the timeline lies
send({ type: 'agent', name: 'security', status: 'running' });
send({ type: 'agent', name: 'performance', status: 'running' });
send({ type: 'agent', name: 'maintainability', status: 'running' });

const [sec, perf, maint] = await Promise.all([
  runSecurityAgent(code, meta).then((f) => { send({ type: 'agent', name: 'security', status: 'done', findings: f }); return f; }),
  runPerformanceAgent(code, meta).then((f) => { send({ type: 'agent', name: 'performance', status: 'done', findings: f }); return f; }),
  runMaintainabilityAgent(code, meta).then((f) => { send({ type: 'agent', name: 'maintainability', status: 'done', findings: f }); return f; }),
]);

// The supervisor gets the 3 lists together and returns ONE report
const report = await runSupervisorAgent([...sec, ...perf, ...maint], meta);

The supervisor consolidates. A single call with the cheap model. It gets all the raw findings and returns a ReviewReport: it deduplicates, reprioritizes (security > performance > maintainability at equal severity), writes the summary, and assigns the score.

export async function runSupervisorAgent(findings: Finding[], meta: CodeMetadata) {
  const { object } = await generateObject({
    model: SUPERVISOR_MODEL,
    schema: ReviewReportSchema,
    system: SYSTEM,   // "deduplicate, reprioritize, summarize, score 0-100"
    prompt: `Raw findings from 3 agents:
${JSON.stringify(findings, null, 2)}`,
  });
  return object;
}

The transport: SSE, no queue. The route opens a ReadableStream and hands orchestrate a send function that pushes each event as data: {...}. The browser reads it with EventSource. No Redis, no job queue: one request that keeps spitting out progress. A rate limit of 3 per hour per IP keeps my bill from running away.

export async function POST(req: Request) {
  const ip = req.headers.get('x-forwarded-for')?.split(',')[0]?.trim() ?? 'unknown';
  if (!rateLimit(ip, { max: 3, windowMs: 3_600_000 }).allowed)
    return new Response('Rate limit exceeded.', { status: 429 });

  const stream = new ReadableStream({
    async start(controller) {
      const send = (e: object) => controller.enqueue(
        new TextEncoder().encode(`data: ${JSON.stringify(e)}\n\n`));
      try { await orchestrate(send, body); }
      finally { controller.close(); }
    },
  });
  return new Response(stream, { headers: { 'Content-Type': 'text/event-stream' } });
}

What broke, and how I fixed it

The honest part. None of this worked on the first try; several broke silently, which is the worst thing about multi-agent systems.

Bug · The supervisor deduplicates too much

On the SQL-injection example, the three agents produced 16 raw findings and the final report showed a single one. It worked, but it wasted the whole multi-agent story. The cause is in the supervisor's prompt: "deduplicate findings that cover the same underlying issue" is too broad and collapses distinct-but-similar findings. The fix is five lines: tighten "deduplicate" to "exact same issue only" and require at least one finding per non-empty category.

Bug · The timeline lied with sequential await

The first version awaited the agents one after another and only then updated the UI. The result: the timeline showed "security… then performance… then maintainability," as if they were in a line, even though the goal was parallel. The fix was to emit the three running events before the Promise.all. Parallelism has to be seen, not just happen.

Bug · The rate limit resets on cold start

The 3-per-hour limit lives in an in-memory Map. In serverless, each cold start wipes it, so a determined user can get past it. For a portfolio demo that's fine: it's a cost brake, not a real quota. A serious version would use Upstash or KV, and I left it noted instead of pretending it was solved.

What I'd do differently

Stream findings as they arrive. Today the UI shows that each agent finished, but the detail only appears when the supervisor closes. Each agent already emits its findings; I didn't paint them the moment they land, which would make the wait feel shorter.

Fix the deduplication before showing it off. It's the flaw that most hurts the demo, because it hides exactly the work of the three agents. It's a five-line change I should have made before publishing.

A routing step for the trivial cases. A ten-line snippet doesn't need three agents and a supervisor. A lightweight router that decides "this is small, one agent is enough" would save three of every four calls on the easy cases.

The pattern, generalized

Take away the word "code" and what's left is the pattern from Part 1: specialists in parallel + a supervisor that consolidates. Three things made it work here and recur everywhere: each agent has a single role (and so performs better than a prompt that does everything), Zod turns the outputs into a contract (the supervisor gets data, not prose to guess), and the Promise.all buys parallelism almost for free. The cost is real (four calls per run, and coordination one agent didn't have), but when the problem splits into points of view, it pays for itself.

If you have a due diligence, a support triage, or a moderation where several experts look at the same thing and someone has to draw a conclusion, this is the shape. The code reviewer is just the excuse to show it with real code.
