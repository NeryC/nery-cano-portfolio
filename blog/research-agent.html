How an AI agent uses tools in a loop, and the cleanest way to let it decide when it's done: a tool with no executor. Explained from zero, then applied to a real research agent in Part 2.

Every post about "autonomous LLM agents" in 2024 had roughly the same shape: a hand-written loop that calls the model, runs whatever tool it asked for, and checks some condition to decide whether to stop.

let messages = initialMessages;
for (let i = 0; i < MAX_STEPS; i++) {
  const result = await streamText({ model, messages, tools });
  messages = [...messages, ...result.responseMessages];
  if (result.text.includes('FINAL ANSWER')) break;   // ← the fragile part
}

It works. The fragile part is that last line: deciding when the agent is done by scanning its text for a magic phrase. This guide builds up to a cleaner answer, one that most agent frameworks now bake in, and by the end you'll know exactly why it beats the hand-rolled loop.

Six terms, first

Six words carry the whole idea. Define them once and the rest reads clean.

LLM. The model (Claude, GPT). It predicts text, and it can be told about tools it may call.

Tool. A function you expose to the model with a name, a description, and a typed input schema. The model can ask to call it.

Execute. The function that runs when a tool is called. Most tools have one; the trick in this guide is a tool that has none.

Agentic loop. The model calls a tool, sees the result, calls another, and so on, until it decides it's finished. Not one prompt: a loop of tool calls.

Step. One turn of that loop (one tool call). Agents cap the number of steps as a safety net.

Terminal tool. The tool whose call means "I'm done." The heart of the pattern.

What an agentic loop is, in one paragraph

An agentic loop is what turns a text predictor into something that acts. You give the model a set of tools and a goal. It picks a tool, you run it, you hand the result back, and it picks again. The loop repeats until the model signals it's done. Everything interesting about agents (research, coding, support) is a variation on this one shape. The only hard question in it is the one the 2024 loops answered badly: how does the model say "I'm done", and how does your code know?

The problem: how does the loop know when to stop?

In the hand-rolled loop, "stop" is a guess about the model's text. Two common versions, both brittle:

  • Match a magic string. Break when the output contains "FINAL ANSWER". Fails when the model rephrases, or writes the phrase while still thinking out loud.
  • Check for no tool calls. Break when the model's last message called no tools. Fails when the model calls a tool and writes prose that looks final in the same turn.

Both share a root flaw: the stop signal lives in free text, so you're parsing prose to make a control-flow decision. And once you decide it stopped, you still have a second problem, pulling the final answer back out: scan the messages, slice the JSON, parse it, and handle the case where the model wrote prose instead. Every one of those steps is a place to break.

The anatomy: a tool that ends the loop

The fix is to make "I'm done" a tool call like any other. You define a terminal tool, say finalAnswer, with a typed schema and no executor. The agent framework knows what to do: when the model calls a tool that has no execute, there's nothing to run, so the loop stops. The tool's call is the result.

Now both problems disappear at once. Termination is decided by which function the model called, a structured field, not by parsing text. And the final answer is the tool's validated input, already typed, with no extraction step.

Figure 1 · The agentic loop. Work tools have executors and feed their result back into the loop. The terminal tool has none, so calling it ends the loop with a typed payload.

The pieces, in detail

A tool with an executor, and one without

In the AI SDK, a normal tool pairs a schema with an execute that does the work:

export const searchWebTool = tool({
  description: 'Search the web for relevant pages on a topic.',
  inputSchema: z.object({ query: z.string().describe('The search query') }),
  execute: async ({ query }) => {
    const results = await searchExa(query, 5);
    return results.map(r => ({ title: r.title, url: r.url, highlights: r.highlights }));
  },
});

The terminal tool is the same shape with the executor removed:

export const finalAnswerTool = tool({
  description: 'Submit the final answer to the user with structured citations.',
  inputSchema: z.object({
    answer: z.string(),
    sources: z.array(z.object({ url: z.url(), title: z.string() })),
  }),
  // ← intentionally no execute
});

No execute. Just a schema. When the model calls it, the loop stops and the validated { answer, sources } is what your code receives.

Pattern · Terminal tool

Use a no-execute tool any time you want the model to choose between "keep working" and "I'm done". The decision is the same mechanism as picking any other tool, far more reliable than parsing free text, and the validated input becomes your typed return value for free.

Why this beats the hand-rolled loop

Termination is structural, not heuristic. "When do we stop?" is answered by which function the model called, a structured field, not a string you match against. No rephrasing or think-out-loud can fool it.

The output is already typed. When the loop stops you get the terminal tool's input, parsed and validated against the schema. There is no separate "extract the answer" step and no malformed-JSON case to handle.

The loop is declarative. An agent described as ({ model, tools, stopWhen }) reads as a complete statement of what it does. No hidden loop body, no exit conditions scattered across helpers. Come back in a year and you understand it in seconds.

The safety net: a step cap

One risk remains: the model might never call the terminal tool (it gets stuck searching in circles). So you cap the loop. In the AI SDK that's stopWhen: stepCountIs(8): after 8 tool-call iterations the framework halts the agent regardless. It bounds both cost and the risk of an infinite loop. The terminal tool is the normal exit; the step cap is the emergency brake.

The minimal agent, in code

Put it together and a whole agent is one declaration. Two work tools, one terminal tool, one step cap:

import { ToolLoopAgent, stepCountIs } from "ai";

export const agent = new ToolLoopAgent({
  model: "anthropic/claude-sonnet-4.6",
  instructions: SYSTEM_PROMPT,
  tools: {
    searchWeb: searchWebTool,     // has execute → runs, feeds back
    readPage: readPageTool,       // has execute → runs, feeds back
    finalAnswer: finalAnswerTool, // no execute → ends the loop
  },
  stopWhen: stepCountIs(8),
});

const result = await agent.stream({ messages });
// result.finalToolCall.args is typed as { answer, sources }

No while, no for, no "did it say it's done?" parsing, no regex over the response. The model chooses between "search", "read", and "finish" with the same mechanism it uses for any tool, and the loop ends the instant it picks "finish".

The pattern, generalized

The no-execute terminal tool goes well beyond research agents. Anywhere an LLM has to decide when it's done, this is the cleanest signal:

  • Coding agents: submitDiff({ files, summary }) is the terminal tool. The agent reads, edits, runs tests, and submits. The loop ends with a typed diff, not a regex over a transcript.
  • Support agents: escalateToHuman({ summary }) and resolveTicket({ answer }) are both terminal. The agent's job is to pick one. The loop ends with a structured action.
  • Extraction agents: emitRecord({ ...fields }) is the terminal tool. The agent crawls, asks follow-ups through other tools, and emits the validated record when it has enough.

The signal that you want this pattern is simple: you catch yourself writing code that reads the model's prose to decide what to do next. Turn that decision into a tool call and the guesswork disappears. That is still one agent, though; coordinating several is a different problem, and Multi-agent systems from zero covers it.
