Loop engineering: the five levers of a reliable agent loop

Somewhere in early 2026 the lever that gets more out of an AI model stopped being the prompt. It became the loop the model runs inside. This is what loop engineering is, the five dials it comes down to, and how to tell which one your flaky agent is missing.

For most of 2023 and 2024, the smartest thing you could do with an LLM was learn to talk to it. That worked because models answered in one shot. Then they started running in loops, calling tools, acting over many steps, and the prompt stopped being the bottleneck.

Solomon Hykes, who founded Docker, put the new danger in one line the whole field now quotes: "an AI agent is an LLM wrecking its environment in a loop." The reframe hiding in that joke is the important part. A more powerful model in a badly designed loop is more dangerous, not less, because it executes its bad ideas more competently. When the model is strong, the loop is where the engineering lives.

Six terms, first

Six words carry the whole idea. Define them once and the rest reads clean.

Agent loop. An LLM running tools in a cycle (perceive, reason, act, observe) until a goal is reached or a limit hits. The one feature that separates an agent from a single model call.

Tool / ACI. A function the agent can call to do something the model alone can't. Anthropic calls the whole tool surface the agent-computer interface, the agent's equivalent of a UI.

Stopping condition. The rule that decides when the loop quits: success, a max-iteration cap, a budget, or no-progress detection.

Context. The set of tokens the model sees on a given turn. Finite, and it degrades as it fills (context rot).

Verification. How the loop proves it did the thing, instead of declaring victory. Usually a test suite or a grader.

Guardrail. What stops the loop doing harm: a sandbox, scoped credentials, a budget cap, a human in the review path.

What loop engineering is, in one paragraph

Loop engineering is the discipline of designing the loop an agent runs inside, instead of perfecting a single instruction. Simon Willison named it in September 2025: "my preferred definition of an LLM agent is something that runs tools in a loop to achieve a goal. The art of using them well is to carefully design the tools and loop for them to use." The unit of work moved from "write a good prompt" to "design a good loop", and the loop decomposes into a handful of dials you turn: the tools the agent can call, when it's allowed to stop, what it keeps in context, how it checks its own work, and the guardrails that keep a runaway loop from doing damage.

Why the loop became the lever, not the prompt

The clearest way to place loop engineering is as the top floor of a three-storey building. Anthropic frames the middle floor, context engineering, as "the natural progression of prompt engineering", and the same logic carries up one more step.

Figure 1 · Three nested layers. Prompt engineering optimizes one call; context engineering optimizes what the model sees each turn; loop engineering optimizes the machinery that decides whether the agent ever reaches a good state at all.

The relationship isn't competitive, it's nested. The thing most likely to break an agent now isn't a badly worded instruction, it's a loop with no off-switch, no memory strategy, or no way to check its own work. When an agent misbehaves in production, the fix is almost always a change to one of these dials, not a reworded prompt.

The anatomy: one cycle, five dials

At the center is the cycle every agent runs: it perceives the state, reasons about what to do, acts with a tool, and observes the result, over and over. Loop engineering is everything you build around that cycle. Read across the people defining the field (Willison, Anthropic, LangChain, Thoughtworks) and the same five levers show up every time.

Figure 2 · The five levers. The model runs the cycle in the middle. You engineer the five dials around it: tools, stopping conditions, context, verification, guardrails. Watch the token travel the loop.

The five levers, in detail

1 · Tools (the agent-computer interface)

What the agent can do. Anthropic spent more time optimizing tools than the prompt on their SWE-bench work and coined ACI (agent-computer interface) as the agent's equivalent of a UI. The common failure is a bloated tool set: "if a human engineer can't definitively say which tool to use in a given situation, an AI agent can't be expected to do better." For coding agents, Willison prefers plain shell commands over MCP because "coding agents are really good at running shell commands", with an AGENTS.md that documents the few extra tools they'll need.

2 · Stopping conditions

When the loop is allowed to quit: on success, on a max-iteration cap, on a budget limit, or when it detects it's making no progress. A loop with no termination logic either declares victory early or never stops. This is the lever the research-agent guide is entirely about: a no-execute terminal tool turns "I'm done" into a structured tool call instead of a string you parse.

3 · Context management

What survives across turns. Anthropic's key finding is context rot: as the token count grows, recall degrades, so context is a finite resource with an "attention budget" you have to spend carefully. The techniques: compaction (summarize near the window limit and restart), structured note-taking (a NOTES.md the agent keeps outside its context), and sub-agents that each burn tens of thousands of tokens but return a tight 1,000 to 2,000 token summary. The guiding rule: find the smallest set of high-signal tokens that gets the outcome.

4 · Verification

How the loop proves it did the thing, instead of taking its own word for it. This is the single most repeated lever. Willison: a coding agent's value "is massively amplified by a good, cleanly passing test suite." Anthropic's harness for long-running agents uses a JSON feature list of 200+ end-to-end features, each marked passes: false, so the agent can't mark a feature done without proving it. A loop that acts without verifying is a loop that ships its own hallucinations.

5 · Guardrails

What stops the loop doing harm. Sandboxes, tightly scoped credentials, budgets, and a human in the path for anything irreversible. Willison ran a coding agent against a dedicated Fly.io org with a $5 budget so a runaway loop couldn't spend real money, and locks network access to a list of trusted hosts to block exfiltration. The stronger the model and the more open the "YOLO mode", the more this lever carries.

Diagnostic · which lever is missing?

Agent gives confident wrong answers → verification. Loops forever and burns tokens → stopping conditions. Forgets what it was doing on long tasks → context. Can talk about the task but can't do it → tools. Did something destructive → guardrails. When an agent is flaky, the fix is one of these five, almost never the prompt.

A second lens: the four-loop stack

LangChain (building on swyx's "loopcraft") frames the same craft as loops that stack. Loop 1 is the agent (model + tools). Loop 2 wraps it in verification (a grader feeds failures back). Loop 3 is event-driven: webhooks and cron trigger the agent without a human. Loop 4 is hill-climbing: you read the traces and rewrite the harness itself. The value compounds at loops 3 and 4, and a human sits in the path at every level. The five levers are the dials; the four loops are the altitudes.

The minimal loop, with the levers visible

Strip it to the smallest thing that shows all five. This is the shape underneath every framework:

let context = [systemPrompt, task];           // lever 3: context
for (let step = 0; step < MAX_STEPS; step++) {  // lever 2: stopping (cap)
  const action = await model.decide(context, TOOLS); // lever 1: tools (the ACI)
  if (action.type === "finish") break;          // lever 2: stopping (terminal tool)

  const result = await sandbox.run(action);     // lever 5: guardrails (sandboxed)
  const ok = await verify(result);              // lever 4: verification
  context = compact([...context, action, result, ok]); // lever 3 again: keep it tight
}

Every framework (ToolLoopAgent, LangGraph, Claude Code) is this loop with the dials turned to different settings. Loop engineering is choosing those settings on purpose.

The pattern, generalized

Loop engineering is the runtime sibling of harness engineering. The harness view designs the repo an agent works in (guides that orient it before it acts, sensors that catch it after). The loop view designs the runtime cycle a single agent executes (the five levers). Same equation underneath, Agent = Model + Harness, at two altitudes, and they cross-reference cleanly: a test suite is a sensor at the repo level and the verification lever at the runtime level.

The signal that you're doing loop engineering, whether you call it that or not: you catch yourself asking not "what should I tell the model" but "what can it call, when does it stop, what does it remember, how does it check itself, and what stops it doing damage". Those five questions are the job.
