The five levers of Part 1, turned on a real production system: AdminRent, a multi-tenant SaaS for managing residential communities in Paraguay. Here a missing guardrail leaks a neighbor's data and a missing verification double-counts a fine, so every lever is a decision with teeth.

Part 1 laid out loop engineering as five dials: tools, stopping, context, verification, guardrails. This is what each one looks like when the agent is working on a live Flask backend that is the source of truth for people's rent, fines, and access control.

A note on scope. My harness-engineering post ran an experiment on this same SaaS: give an agent five real commits, write an AGENTS.md, measure the delta. This post is the other lens: the runtime loop that AGENTS.md encodes, read through the five levers. Same system, different altitude.

The system, and where the loop runs

AdminRent is a multi-repo system: a Flask + SQLAlchemy + Postgres backend (the source of truth for data, RBAC, and business rules), a Next/React frontend, a Flutter app, and a Node SIFEN billing microservice. A coding agent works one task in one repo, unless the task touches the API contract, in which case backend, generated types, and frontend move as a single unit under human review.

Figure 1 · The five levers as a gate. The agent acts on the repos, but every action passes the five levers first, and anything irreversible ends at a human.

The five levers, on real code

1 · Tools: a fast, honest inner loop

The most important tool an agent has here is the test command, and the design choice that matters is speed. The suite runs on in-memory SQLite, no Postgres required, so the agent can loop on it dozens of times a minute:

pytest -q                 # runs on sqlite in-memory; no Postgres needed
python -m compileall -q app  # syntax gate, faster than a full run

The AGENTS.md also documents the environment traps the agent would otherwise rediscover by hand, like the two dependency pins that need overriding only in the local Python 3.14 venv while production runs on python:3.12-slim. That is the tools lever: a clean interface plus the few notes that stop the agent flailing.

2 · Stopping conditions: stop and ask

The sharpest stopping rule in this codebase isn't a max-iteration cap, it's a semantic one. Permissions are product decisions, so the loop is told to halt rather than guess:

If a permission decision is NOT in the roles table: STOP and ask the human.
Don't invent the policy even if it seems consistent with the principles.

This is a stopping condition fused with a guardrail. Layered on top is an autonomy ladder that scopes when the agent may act at all: always without asking (add or fix tests, update docs, refactor within a layer), ask first (a new endpoint, an uncovered permission decision, a schema change), and never without a human (production migrations, PII, billing). The loop's default is to stop early, not to push through.

3 · Context: one map, detail on demand

Each repo carries its own AGENTS.md; a root one is the system map. That is Anthropic's "right altitude" in practice: "this file is the map; the detail lives in docs/ and in the code." The agent gets a tight, high-signal brief up front and pulls specifics just-in-time, instead of drowning in a dump of the whole codebase. The context lever here is a discipline of what to put in the map versus what to let the agent fetch.

4 · Verification: the sensor gate

Verification is the lever with the most surface area, because the system is the source of truth for money. The rule is written into the conventions: a new endpoint requires a permission test, at least one role that gets 200 and one that gets 403. For a unit-scoped resource, that includes proving an owner of another unit gets 403. Here is a real one, cross-tenant isolation on amenity endpoints:

def test_admin_a_cannot_read_other_residential_amenities(app, db, ...):
    rid_b, _ = _second_residential_with_amenity(app, db)   # a second tenant
    resp = client.get(f"/api/residentials/{rid_b}/amenities", headers=headers)
    assert resp.status_code == 403      # admin of A has no role in B
    assert resp.get_json()["error"] == "Forbidden"

Above the unit tests sits a multi-sensor gate: pytest and tsc catch broken shapes, contracts-fresh (in pre-commit and CI on both repos) catches a schema changed without regenerating types, and an e2e cross-repo job boots the real backend and runs Playwright as the "does the system work together" gate. Verified end to end: rename a field in the backend and tsc breaks in the frontend. That is verification as a closed loop, not a vibe.

5 · Guardrails: the invariant that can't bend

On a multi-tenant SaaS the guardrail lever is where a bad loop does real damage, so it's the most explicit. Invariant #1, no exceptions: every query filters by residential_id. On top of it sits an RBAC table (who sees what across admin, owner, tenant, portero) that the backend enforces with require_role(...):

require_role("admin")                              # admin-only
require_role("admin", "owner", "tenant")           # residents, no portero
require_role("admin", "owner", "tenant", "portero")  # shared read incl. portero
require_role("admin", "portero")                   # gatehouse functions

The portero (gatekeeper) role is the trap the AGENTS.md calls out by name: it's lateral, it inherits nothing, and it must be included explicitly when its job needs the data. Forgetting it, or excluding it as a "lesser role", is the easiest mistake to make and almost always wrong. A hard boundary list names what the agent must never touch without a human: migrations/, the PII_DENYLIST, RBAC and auth, SIFEN signing keys, and never a direct push to main.

Takeaway

On a toy agent the five levers are good practice. On a production multi-tenant SaaS they are the difference between an agent that helps and one that leaks a resident's financial data across tenants. The stakes don't change the levers, they change how much each one is allowed to bend: here, the guardrail lever bends to zero.

Where a lever earned its place

1
portero silently excluded: a guardrail catch
Left to its own judgment on an access-log endpoint, the agent wrote require_role("admin", "owner", "tenant"), reasoning that the gatekeeper is a "minor" role. Wrong: the portero needs that data to identify residents. The fix wasn't a smarter model, it was the guardrail lever writing the rule down (portero is lateral, decide explicitly) plus the stopping condition (not in the table? stop and ask).
2
A fine counted twice: a verification catch
A fine already hits net_balance through its mirror transaction (type="fine", balance_impact=-amount). An intuitive "also subtract pending fines from the balance" change double-counts them. No prompt phrasing prevents this; a characterization test on the account statement does. The financial invariant lives in AGENTS.md precisely because it's the kind of thing that reads as correct and is not.
3
Fail-closed on sensitive fields: a default that had to be a guardrail
Private fields (identities in general views, target_unit_ids, per-person amounts) hide by default and appear only behind an explicit admin flag. A new endpoint that forgets the flag must leave the field hidden, not exposed. So the default of a parameter like include_targets is False, never True. The guardrail is the direction the default fails.

What I'd do differently

  1. My map drifted, in the one place no sensor could reach. Writing this, I found the system-level AGENTS.md, the map that sits above all the repos, still said "autonomy level 1", while the machine had run at level 2 since July 6. The in-repo docs were fine: adminrent-mono/AGENTS.md already said level 2. The stale one was the coordinating map that lives a directory above the repo, versioned by no git at all. The gardener never caught it for a structural reason: it checks out a repo and works inside it, and a doc that governs every repo but belongs to none is outside every sensor's field of view. That's the harness thesis turned on me: the rule no sensor can reach is the one that rots. The fix isn't a smarter gardener: a doc that coordinates repos it lives in none of needs its own check, or it shouldn't carry a claim (like an autonomy level) that drifts out from under it.
  2. The steering loop is manual on purpose. The gardener automates the doc-drift half of loop 4: a weekly Claude Code run that regenerates schemas, checks docs against code, and opens a PR. But deciding to add a new guardian test when a sensor trips twice stays a human call, logged one line at a time in SENSOR-LOG.md. On a system that bills real people, the choice to trust a class of change should cost a human a moment.
  3. Observability is a log, not a dashboard. SENSOR-LOG.md is the instrument: one line per real sensor catch, measured across a six-week window to decide whether the jump to level 2 was earned. A dashboard would be instrumentation nobody reads; the log is the cheapest thing that answers the one question that matters.

The pattern, generalized

Read AdminRent through LangChain's four-loop stack and it lines up. Loop 1 is the agent in a repo. Loop 2 is the sensor gate (pytest, tsc, contracts-fresh, e2e) feeding failures back. Loop 3 is event-driven: pre-commit hooks and CI fire without a human. Loop 4 is split: the gardener automates the doc-drift half (a weekly Claude Code run that opens a PR), while the steering half, a guardian test or an AGENTS.md line when a sensor trips twice, stays a logged human decision. The value compounds at 3 and 4, and a human stays in the path at every level, by design, because the system bills real people.

The transferable rule isn't "copy this AGENTS.md". It's that on any system with real stakes, the five levers stop being style and become safety: the more a wrong action costs, the less the guardrail and verification levers are allowed to bend. Start there, and let each piece earn its place.
