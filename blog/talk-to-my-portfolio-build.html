An MCP server over my CV, my projects and my 164 Platzi certificates, embeddable in any page with one line of HTML. Here is the real code, the decisions, and the six bugs that broke production before any of it worked.

Part 1 explains what MCP is and why it exists. This is the other half: what happens when you take it to production on a serverless runtime that forgives nothing.

The idea was easy to state. A recruiter opens my portfolio and, instead of reading a list of bullets, asks what they came for: "has he worked with RAG?", "what did he build in TypeScript?", "is he free on Thursday?". And the answers come from verifiable data, with a link to the diploma, not from a model inventing adjectives about me.

What was not easy: making that same server work for the chat on my site and for Claude Desktop and for anyone else's agent, without writing the logic twice. That is exactly what MCP solves, and it is why this project exists.

The architecture, end to end

It is a pnpm monorepo. The rule that organizes everything: the MCP server does not know Next.js exists. It is a plain TypeScript package, with its own package.json and its own binary, that could be published to npm and run over stdio. Next.js is merely one of its two consumers.

Figure 1 · The architecture. One package defines the MCP server; the HTTP route publishes it to the world and the demo agent consumes it without leaving the process.

The four packages, and what each one does:

PackageWhat it isDepends on
packages/sharedThe data (JSON + Markdown), its Zod schemas, and the cached loadersnothing
packages/mcp-serverThe server: tools, resources, prompts, Cal.com and Resend integrationsshared
apps/webNext.js 16.2.6: the MCP route, the agent route, the chat, and the widget pagemcp-server, shared
packages/widgetThe embeddable floating button, in Preact, bundled with esbuildnothing

The two routes in apps/web consume the same server by different paths, and that asymmetry is deliberate. /api/mcp publishes it over Streamable HTTP for the world. /api/agent consumes it internally, never touching the network. Bug 2 explains why.

How I built it, step by step

The server: 9 tools

I used the SDK's low-level API (Server + setRequestHandler) rather than McpServer, because I wanted every tool to be a plain object I could test without standing up a server. A tool is its schema, its description and its function:

// packages/mcp-server/src/tools/types.ts
export interface Tool<S extends ZodTypeAny = ZodTypeAny> {
  name: string;
  description: string;
  inputSchema: S;
  execute: (args: z.infer<S>) => Promise<unknown>;
}

And the registry is a list. Nine tools, three families: read the portfolio, search it, book a call.

// packages/mcp-server/src/server.ts
const tools: readonly AnyTool[] = [
  listProjectsTool, getProjectTool, searchByTechTool,
  getSkillsTool, getExperienceTool, searchCoursesTool,
  getCourseTool, getAvailabilityTool, bookCallTool,
];

server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: tools.map((t) => ({
    name: t.name,
    description: t.description,
    inputSchema: zodToJsonSchema(t.inputSchema),
  })),
}));

The zodToJsonSchema on that line is the bridge between two worlds: I write Zod because it gives me types inside execute, and the protocol wants JSON Schema. Bug 4 is born right there.

I wrote each tool description as a prompt, because that is what it is. This is the one for listProjects, verbatim:

description:
  "List Nery's portfolio projects. Use when the recruiter asks 'what have you built?' " +
  "or wants an overview. Optionally filter by tag (e.g. 'rag', 'ai', 'react').",

It names the reader ("the recruiter"), gives examples of the question that triggers it, and spells out the optional filter. With a description along the lines of "lists projects", the model called it for everything.

The resources: 4 shapes in the code, 171 on the wire

Here is the most uncomfortable correction in this post. For months my README said "resources (4 URIs)" and the old post said "4 resources". It is false, and I found out re-reading my own code to write this.

There are four URI shapes in the code. But listResources() does not return shapes: it returns the expanded list, one entry per thing that exists.

// packages/mcp-server/src/resources/index.ts
const resources: Resource[] = [
  { uri: "portfolio://cv/full", name: "Full CV (markdown)", mimeType: "text/markdown", ... },
];
for (const p of projects) {
  resources.push({ uri: `portfolio://projects/${p.slug}/readme`, ... });
  resources.push({ uri: `portfolio://projects/${p.slug}/case-study`, ... });
}
for (const c of courses) {
  resources.push({ uri: `portfolio://courses/${c.slug}/certificate`, ... });
}

The real arithmetic: 1 CV + 2 × 3 projects + 164 courses = 171 resources. With a ResourceTemplate the number on the wire would be different, but I expand by hand, so 171 is what a client receives when it asks for the catalog. The lesson is not that 171 is wrong: it is that I put a number in a README without running the arithmetic, and it survived months of copy-paste.

Takeaway

A number in a README is a claim with no test behind it. The three numbers in this project (9 tools, 3 prompts, 171 resources) come from counting the data structure, not from my memory; the first two matched, the third did not. If a number matters enough to publish, it matters enough to assert.

The prompts: borrowed intelligence, with a plan B

All three prompts (pitch-for-role, compare-with-jd, tech-deep-dive) use sampling: instead of calling Anthropic with my key, they ask the client to run the completion on its model. If the client has no sampling, the prompt still has to work. That means writing each one twice.

compare-with-jd is the fullest, since it uses both client capabilities. If the job description you pasted is too short, it asks for more via elicitation:

// packages/mcp-server/src/prompts/compare-with-jd.ts
if (parsed.jobDescription.trim().length < MIN_JD_CHARS) {
  if (opts.clientSupportsElicitation && opts.elicitationBridge) {
    const r = await opts.elicitationBridge({
      message: "The JD you pasted is very short. Could you paste the full role description?",
      requestedSchema: {
        type: "object",
        properties: { jobDescription: { type: "string", minLength: MIN_JD_CHARS } },
        required: ["jobDescription"],
      },
    });
    if (r.action !== "accept" || typeof r.content?.jobDescription !== "string") {
      return { description: "compare-with-jd (elicitation cancelled)", messages: [ /* ... */ ] };
    }
    parsed = { jobDescription: r.content.jobDescription };
  } else {
    // Client cannot ask: hand back a message that makes the model ask instead.
    return { description: "compare-with-jd (no elicitation support)", messages: [ /* ... */ ] };
  }
}

Note the r.action !== "accept". A person can accept, decline or cancel, and all three are legitimate answers you have to handle. Closing the dialog is not an error.

And with no sampling, the fallback is not an apology message: it is a deterministic keyword comparison against my project tags, my course skills and my experience stack. Worse than an LLM, far better than nothing.

Client capabilities are discovered during the handshake, so the bridges get reconfigured every time someone connects:

// SDK 1.29.0 exposes `oninitialized` (lowercase i) as a public callback.
if (typeof (server as any).oninitialized !== "undefined") {
  (server as any).oninitialized = configure;
}

That as any with a comment next to it is the kind of thing I would rather show than hide. The callback exists, the types did not declare it, and the alternative was worse.

The demo agent: MCP without a network

The /demo chat runs an AI SDK v6 ToolLoopAgent that discovers tools over MCP at startup and maps them to SDK tools. The schema conversion is the part that matters:

// apps/web/lib/agent.ts
import { ToolLoopAgent, jsonSchema, stepCountIs, tool } from "ai";

const aiTools = Object.fromEntries(
  discovered.map((t) => [
    t.name,
    tool({
      description: t.description,
      inputSchema: jsonSchema(t.inputSchema as Parameters<typeof jsonSchema>[0]) as never,
      execute: async (args) => { /* calls the MCP server and parses the JSON */ },
    }),
  ]),
);

return new ToolLoopAgent({
  model: "anthropic/claude-sonnet-4-6",
  instructions: SYSTEM_PROMPT,
  tools: aiTools,
  stopWhen: stepCountIs(10),
});

The system prompt has exactly one rule I care about deeply, and it is capitalized in the code: NEVER invent projects, courses, skills, or experience. Use the tools to ground every claim. An assistant that oversells my CV is worse than no assistant.

The connection to the MCP server does not use HTTP. It uses a linked pair of in-memory transports:

// apps/web/lib/mcp-client.ts
export async function connectInProcessPortfolio(name: string): Promise<NamedServer> {
  const [clientTransport, serverTransport] = InMemoryTransport.createLinkedPair();
  const { server } = buildServer();
  await server.connect(serverTransport);
  const client = new Client(
    { name: "web-host", version: "0.0.0" },
    { capabilities: { sampling: {}, elicitation: {} } },
  );
  await client.connect(clientTransport);
  return { name, client };
}

It is full MCP: handshake, capability negotiation, tools/list, tools/call. The only thing that changes is the pipe. And since this client declares sampling and elicitation, the prompts take the good path instead of the fallback.

The widget: 14 KB and one line of HTML

The requirement was that anyone could paste it into their site with no build, no npm, and no risk to their CSS. The solution has three pieces and fits in one Preact file.

Figure 2 · The widget, in three pieces. A script that mounts a button, a button that opens an iframe, and an iframe that is a whole app: that is how the host page cannot break anything.

Preact instead of React for weight: the final bundle is 14,553 minified bytes, IIFE, no external dependencies.

// packages/widget/esbuild.config.mjs
await build({
  entryPoints: ["src/index.tsx"],
  bundle: true,
  outfile: "../../apps/web/public/widget.js",
  format: "iife",
  globalName: "PortfolioWidget",
  target: "es2020",
  minify: true,
  jsx: "automatic",
  jsxImportSource: "preact",
  alias: { react: "preact/compat", "react-dom": "preact/compat" },
});

The widget does not render the chat: it opens an iframe to /widget. It looks like a detour and it is the cheapest decision in the project. The iframe isolates styles in both directions, so the host's Bootstrap cannot break my chat and my Tailwind cannot break their page.

One detail falls out of that decision: inside the script the colors are hardcoded, with a comment explaining why.

// Portfolio palette (neryc.github.io/nery-cano-portfolio):
//   --accent      #2E75B6 (primary blue)
// Keeping these as literals (not CSS vars) because this script runs in the
// HOST page where those vars don't exist.
const COLOR_ACCENT = "#2E75B6";

And the iframe host is derived from the <script> tag itself, so the same bundle works locally and in production with no configuration:

const WIDGET_HOST =
  (document.currentScript as HTMLScriptElement | null)?.src?.replace(/\/widget\.js.*$/, "") ??
  "https://talk-to-my-portfolio.vercel.app";

The six bugs that broke production

Everything above worked on localhost on the first try. Then I deployed it. All six, in order of appearance, with the fix that is in the repo today.

BUG 1
ENOENT: cannot open /var/task/packages/mcp-server/package.json
The server read its own package.json to get its name and version. Locally the file is right there; on Vercel the bundler folds the route into a chunk and the relative path points at nothing. The fix is one line, and it beats the original: an import with a type attribute, resolved by the bundler at build time instead of at runtime.
// packages/mcp-server/src/server.ts
import pkg from "../package.json" with { type: "json" };
const serverInfo = { name: "portfolio-mcp", version: pkg.version };
BUG 2
The MCP session evaporated between workers
The first cut of /api/agent called itself over HTTP against /api/mcp. Locally it is all one process and it works. On Vercel they are two serverless functions: initialize creates the session on worker A, tools/list lands on worker B, which never saw that ID. The fix was to stop using the network.

Figure 3 · Bug 2, drawn. On the left the red dot reaches a worker that never saw that session; on the right client and server share a process and there is nothing to lose.

BUG 3
Invalid prompt: messages do not match the ModelMessage[] schema
The client uses useChat from @ai-sdk/react, which sends UIMessage[] (parts-based). ToolLoopAgent.stream() wants ModelMessage[]. AI SDK v6 ships the bridge and I had forgotten to call it.
// apps/web/app/api/agent/route.ts
const { messages } = (await req.json()) as { messages: UIMessage[] };
const modelMessages = await convertToModelMessages(messages);
const result = await agent.stream({ messages: modelMessages });
BUG 4
TypeError: e is not a function during tool execution
The MCP server hands back schemas as JSON Schema. The AI SDK's tool() expects either a Zod schema (which has .parse) or a Schema wrapped in its jsonSchema() helper. Passing the raw JSON Schema compiles perfectly and explodes at runtime when the SDK tries to validate. Compiling is not working.
// BEFORE — compiles, explodes at runtime
inputSchema: t.inputSchema as never,

// AFTER — the SDK helper wraps the JSON Schema
inputSchema: jsonSchema(t.inputSchema as Parameters<typeof jsonSchema>[0]) as never,
BUG 5
The model demanded a key that did not exist
The agent pinned its model with anthropic('claude-sonnet-4-6'), which calls Anthropic directly and needs ANTHROPIC_API_KEY. The whole portfolio runs through Vercel's AI Gateway (one key, provider failover, one usage dashboard). Passing the model as a string lets the SDK route through the Gateway.
// BEFORE: model: anthropic("claude-sonnet-4-6")
model: "anthropic/claude-sonnet-4-6",
BUG 6
ENOENT: cannot open packages/shared/data/projects/_index.json
The loaders read JSON and Markdown with readFileSync. Next.js's dependency tracer follows static import statements, so it ships every imported JS file; a readFileSync with a path assembled at runtime is invisible to it, and the data directory never made it into the deploy. You have to tell it by hand, by route.
// apps/web/next.config.ts
const nextConfig: NextConfig = {
  outputFileTracingRoot: path.join(process.cwd(), "..", ".."),
  outputFileTracingIncludes: {
    "/api/mcp/[[...path]]": ["../../packages/shared/data/**/*"],
    "/api/agent": ["../../packages/shared/data/**/*"],
  },
};
Takeaway

Five of the six are the same bug wearing six faces: on localhost there is one process, one filesystem and no boundaries, and on serverless none of that is guaranteed. No unit test would have caught any of them, since they all live on the seam between my code and the runtime. The only cheap defense is to deploy early and deploy often.

Tests and CI

There are 38 test files holding 117 cases, 2 of them behind flags (@live hits the real Cal.com, @manual stands up the HTTP transport). The part worth stealing is how CI runs:

- name: Critical tests (sampling/elicitation/capabilities)
  run: pnpm test:critical
- name: Test
  run: pnpm test

The tests tagged @critical cover sampling, elicitation and capability negotiation, and they run before the full suite. If the MCP contract breaks I want to know at step 4 of the pipeline, not buried under 117 green results. The failure mode is "critical tests failed and the rest never ran", which is exactly the message I want to read.

What I would do differently

  1. The /readme resource is a lie, and it is still live. Writing this post I found that portfolio://projects/{slug}/readme and portfolio://projects/{slug}/case-study both call loadProjectCaseStudy(): they return the same file. There is no loadProjectReadme. That is 2 of my 171 resources per project, and half of it is a duplicate. Nobody complained because nobody asked for them, which is precisely what makes the bug dangerous: an unused public API ships all the same, and silence from it measures attention rather than correctness.

  2. The bridges are module-global state. bridge-state.ts holds the capabilities of the last client to run initialize, at module level rather than per session. With two clients of different capabilities on the same warm worker, last one wins. It is documented in the code and acceptable for a portfolio with one session at a time; in anything multi-user it is a bug waiting for its turn. The right shape is to hang the state off the session, not off the module.

  3. The server does not know itself. projects/_index.json lists three projects: research-agent, multi-agent-code-reviewer and rag-agent-memory. talk-to-my-portfolio is missing. Ask the widget what I built with MCP and it answers with what it knows, and it does not know it exists. It is the most interesting of the four projects and the only one absent from the data: I added it last and forgot the file that feeds it.

The pattern, generalized

Strip the CV away and a reusable shape is left: write the capability once as an MCP server, then decide separately who consumes it and through which pipe. The same logic gets published over HTTP for the world and consumed in memory from inside. That is not a trick of this project: it is what separating protocol from transport buys you.

It applies whenever you have domain logic that several fronts want to consume. A product catalog wanted by the support chat, the internal assistant, and a partner's agent. A knowledge base wanted by your bot and by your employees' Claude Desktop. The usual alternative is a REST API plus three tool adapters, one per consumer, and those three adapters are the debt.

When not to: if the only consumer is you and that will stay true, this is an extra process, a handshake and one more package in the monorepo in exchange for nothing. Part 1 has the alternatives table and the line where MCP starts to pay.

What I take from the six bugs is not a list of Vercel gotchas. It is that the distance between localhost and production is measured in assumptions: one process, one disk, one shared memory. Each of the six was an assumption localhost let me hold for free and production charged me for.
