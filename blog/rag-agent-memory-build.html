How I built a RAG agent with memory

Part 1 was the theory. This is the real code: the PDF parser that survives serverless, the pgvector schema, the agent's tools and the memory loop, all open source.

Nery Cano
Senior Full-Stack Engineer · AI/LLM Systems · github.com/NeryC

This is part 2. In part 1 I covered what RAG is and every strategy behind each step. Here I get into the mud: the real code of an agent where you upload PDFs, chat with citations, search the web when needed, and it remembers you across sessions. All verifiable in the repo.

The system does four things. You upload a PDF and in a few seconds it is queryable. You ask and it answers citing document and page, like [manual.pdf p.3]. If the topic is not in your documents, it searches the web on its own. And at the end of each chat, it learns facts about you and remembers them next time. The stack: Next.js 16 with the Vercel AI SDK v6, chat with Claude Sonnet 4.6, Voyage embeddings, and Supabase pgvector as the vector base.

The architecture, at a glance

Two paths that meet at the vector base. Ingestion takes a PDF from raw to queryable fragments. Chat takes a question and answers it with evidence. Both share pgvector and the same per-session isolation.

Figure 1 · The real architecture. Synchronous ingestion on top, streaming chat below; both sides meet at pgvector, isolated by session.

The schema: four tables and two searches

The whole system rests on four Postgres tables and two search functions. The chunks store the PDF fragments with their vector; the memories, the learned facts, also with a vector. Notice both have the same shape: that is what lets you do RAG over documents and over memories with the same mechanism.

-- document fragments: text + vector + where it came from
CREATE TABLE chunks (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id uuid REFERENCES documents(id) ON DELETE CASCADE,
  session_id text NOT NULL,        -- isolation between users lives here
  content text NOT NULL,
  embedding vector(1536),          -- the Voyage vector
  page_number int DEFAULT 1
);
CREATE INDEX ON chunks USING hnsw (embedding vector_cosine_ops);  -- fast search

-- memories: user facts, with the SAME shape as a chunk
CREATE TABLE memories (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id text NOT NULL,
  content text NOT NULL,
  embedding vector(1536),
  confidence float NOT NULL        -- how sure the fact is
);

There is also a documents table (one row per PDF, with its status) and a conversations one (each chat's history in JSON). The two searches, match_chunks and match_memories, are twins: they take a query vector, filter by session, and return what is most similar by cosine. I cover them in detail in the retrieval section.

Ingestion, and the PDF parser that survives serverless

Ingestion runs synchronously inside the upload route: I upload the PDF to Vercel Blob, extract the text, split into 500-token fragments with 50 of overlap, embed in batches of 100, and store, all before responding. It is on purpose: on Vercel's Hobby plan the function window is short, so I put the work in a single invocation instead of building a queue. It works for small PDFs, which is the case for a demo.

The only step that got ugly was extracting the text. I started with pdf-parse, which under the hood uses pdfjs-dist, and I crashed the moment I deployed.

BUG 1
DOMMatrix is not defined
It worked on my laptop; on Vercel, instant crash. pdfjs-dist was designed for the browser and depends on DOMMatrix, canvas and other DOM APIs that never existed in Node. The serverless runtime has no DOM and no shim. The only real way out was to stop depending on that library.
BUG 2
The text comes out as binary
A PDF is a tree of objects, and the text lives in content streams that are almost always compressed with FlateDecode (zlib). Without decompressing, all you see is garbage. I detect the filter in the dictionary before the stream and inflate it with Node's built-in zlib, no dependencies.
BUG 3
Bytes corrupted on read
Reading the PDF as UTF-8 destroys any non-ASCII byte. The correct read is latin1, which maps every byte 0 to 255 to a unique character, so the round-trip loses nothing. Then two regexes over the Tj and TJ operators pull out the text.

The parser ended up around 80 lines of TypeScript with not a single npm dependency. The core is this:

function extractPdfText(buffer: Buffer): string {
  const raw = buffer.toString("latin1")           // lossless read
  const texts: string[] = []
  // 1. find each stream...endstream block
  for (const m of raw.matchAll(/stream\r?\n([\s\S]*?)\r?\nendstream/g)) {
    let content = m[1]
    // 2. inflate if the preceding dictionary says FlateDecode
    const dict = raw.slice(Math.max(0, m.index - 500), m.index)
    if (dict.includes("FlateDecode")) {
      try { content = inflateSync(Buffer.from(m[1], "latin1")).toString("utf8") }
      catch { continue }                            // broken stream: skip it
    }
    // 3. pull the text from the Tj operators
    for (const t of content.matchAll(/\(([^)\\]*(?:\\.[^)\\]*)*)\)\s*Tj/g)) {
      if (t[1].trim()) texts.push(t[1])
    }
  }
  return texts.join(" ").replace(/\s+/g, " ").trim()
}

It does not replace a serious PDF library: it loses images and odd layouts. But it is small, has no DOM dependency, and it deploys. The last nails were fine format details: escaped parentheses inside the strings and octal escapes like \101 that have to be converted to their character.

The pattern

When an "easy" dependency ties you to an environment your deploy does not have, 80 lines that do 90% of the work sometimes beat the 200 KB that do 100%. The key is knowing which 10% you can give up: here, PDFs with images, which a demo does not need.

Retrieval: pgvector, cosine, per session

Each user has their own document space, so retrieving is not just "search for the most similar," it is "search for the most similar from this session." A Postgres function solves that by filtering on session_id before ordering by cosine:

-- match_chunks: nearest neighbors, but only from the user's session
SELECT c.content, d.filename, c.page_number,
       1 - (c.embedding <=> query_embedding) AS similarity   -- <=> = cosine distance
FROM chunks c
JOIN documents d ON d.id = c.document_id
WHERE c.session_id = p_session_id                              -- isolation goes first
  AND 1 - (c.embedding <=> query_embedding) > match_threshold  -- drop noise
ORDER BY c.embedding <=> query_embedding
LIMIT match_count;                                            -- top-k

In the chat I search chunks and memories in parallel: documents with threshold 0.25 and top-5, memories with threshold 0.3 and top-3. The HNSW index makes both searches near-instant even with many vectors.

const q = await embed(question)                    // one embedding for both
const [docs, mems] = await Promise.all([
  supabase.rpc("match_chunks",   { query_embedding: q, match_count: 5, match_threshold: 0.25, p_session_id }),
  supabase.rpc("match_memories", { query_embedding: q, match_count: 3, match_threshold: 0.30, p_session_id }),
])

That per-session filter is the same multi-tenant isolation discipline I worked through in the harness engineering series: one user's data can never appear in another's answer, and the barrier lives in the database, not in the code above it.

The agent: three tools and a step limit

It is not a one-shot RAG. It is an agent: Claude gets the question with the context already retrieved, but it can also go fetch more if it needs to. With the Vercel AI SDK v6, defining that is a few lines.

const tools = {
  search_documents: tool({    // search the PDFs again, with a different phrasing
    inputSchema: z.object({ query: z.string() }),
    execute: async ({ query }) => searchChunks(query, sessionId),
  }),
  search_web: tool({          // Exa, when the documents are not enough (3 results)
    inputSchema: z.object({ query: z.string() }),
    execute: async ({ query }) => searchWeb(query),
  }),
  save_memory: tool({         // save a fact about the user on purpose
    inputSchema: z.object({ fact: z.string() }),
    execute: async ({ fact }) => saveMemory(fact, sessionId, 0.9),
  }),
}

streamText({
  model: gateway("anthropic/claude-sonnet-4.6"),
  system: promptWithContextAndMemories,
  messages,
  tools,
  stopWhen: stepCountIs(6),   // the brake: at most 6 reason-and-act steps
})

The stepCountIs(6) is the equivalent of the autonomy principle from the harness series: an agent with no step ceiling is one that can loop and burn tokens. Six is enough for "search the documents, if it's not there search the web, then answer," and it puts a hard limit.

The answer streams token by token over SSE (server-sent events), not all at once. The server emits a sequence of events the UI consumes live: tool_start and tool_done (to show "searching the web…"), text for each little piece of answer, memories_used with what it recalled, citations with the sources, and done at the end. That is what makes it feel like an agent thinking, not a box that stalls and spits out a block.

Memory: two ways to remember

Here is the part that gives the project its name. The agent remembers things about the user across conversations, and it does so with the same RAG mechanism: memories are text embedded in the same vector base, in their own table. Remembering is, literally, doing RAG over your memories.

There are two ways to write a memory and one to read it. The automatic one is the interesting one: after each answer, a small, cheap model (Claude Haiku 4.5) reads the conversation and extracts durable facts with a confidence score, using structured output. Only those above 0.7 are stored:

const { object } = await generateObject({
  model: gateway("anthropic/claude-haiku-4.5"),
  schema: z.object({
    facts: z.array(z.object({
      content: z.string(),
      confidence: z.number().min(0).max(1),
    })),
  }),
  prompt: `Extract durable facts about the user from this conversation.
Only high-confidence facts (over 0.7) useful in the future.\n\n${conversation}`,
})
for (const f of object.facts.filter(f => f.confidence > 0.7)) {
  await saveMemory(f.content, sessionId, f.confidence)   // embeds and stores
}

The explicit one is the save_memory tool we already saw: the agent itself can decide to save something when it spots it as important. And to read, on every question the most similar memories are retrieved (threshold 0.3, top-3) and injected into the system prompt under "what I remember about this user." The user never repeats that they are coeliac or that they work in finance: the agent already knows.

Figure 2 · The memory loop. Facts are extracted, embedded and stored; on the next conversation they are retrieved and returned to the prompt. RAG over your own memories.

Per-session isolation: the thread that stitches it all

One detail shows up in every table, every search and every tool: the session_id. Each visitor gets a 24-hour httpOnly cookie, and that session tags their documents, their fragments and their memories. The two search functions filter on it before comparing vectors. The result is that two people can upload different PDFs to the same system and neither ever sees the other's, with no login and no accounts. It is multi-tenancy in the lightest possible form, and the barrier lives in the database, which is where it belongs.

What I'd do differently

  1. Split by page for real. Today the parser joins the whole PDF into one text and splits over that, so citations always say p.1. Tracking the real page number during extraction would make citations precise, which is half the value of RAG.
  2. Split better. I use length over 4 as a token proxy and cut by paragraphs. It works, but a real token counter and structure-aware splitting would cut the split fragments down.
  3. Add reranking. I retrieve by pure similarity. Pulling 20 candidates and reordering them with a dedicated model, as I covered in part 1, is the highest-return improvement I have pending.
  4. Asynchronous ingestion for large PDFs. Synchronous ingestion fits the serverless window for small documents. For large files, the right thing is a queue: respond fast and process in the background.

The pattern, generalized

If you look at the whole system, there is a single mechanism repeated three times. Documents are RAG. Memory is RAG over what the agent learned about you. Web search is RAG with an index you do not control. Embed, search by closeness, inject into the prompt: always the same. Once you see RAG as "bring the right evidence at the right time," it stops being a technique and becomes a way to think about any system that has to answer over data the model did not see.

And the hard part is always the same, the one I hammered in part 1: retrieving well. If the answers are bad, it is almost never the model doing the generating; it is the wrong context. Fix retrieval and you fix RAG.
