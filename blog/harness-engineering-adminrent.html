Harness engineering on a real SaaS: I measured, wrote the rules, measured again

I gave an agent five real commits from my multi-tenant SaaS and told it nothing about the domain. It invented a permission policy, got it wrong, and wrote six tests that armored it in green.

Nery Cano
Senior Full-Stack Engineer · AI/LLM Systems · github.com/NeryC

In Part 1 I explained what a harness is and how you build one. Here I put it to the test on AdminRent, a multi-tenant SaaS in production: I measured a baseline with no harness, wrote down the rules that lived in my head, and measured again. What I found surprised me twice over: once about the agent, once about myself.

The task was textbook: "add GET /units/<id>/members returning the unit's members with their email." The agent came back with code I'd have approved in a PR without blinking. RBAC via decorator, per-unit access check, correct multi-tenant filter, logic in the service and not in the endpoint. Six tests, including the negative ones.

@require_role("admin", "owner", "tenant")
def get_unit_members(unit_id):
    ...

The portero, the doorman, is missing. In AdminRent the doorman needs to see a unit's members. That is his job: identifying who he lets in. The agent, reasoning about privacy, decided a guard shouldn't see residents' emails. That's a sensible conclusion. It's also exactly the opposite of what my product decided two years ago.

And then it wrote this:

def test_portero_is_forbidden(client, portero_token):
    r = client.get(f"/units/{unit_id}/members", headers=portero_token)
    assert r.status_code == 403

A test that asserts its own wrong decision as if it were the spec. If that had merged: doorman broken in production, and six green tests swearing everything is fine. The agent didn't fail as a programmer. It failed because a rule that lives in my head didn't live in my repo.

What this is, in one paragraph

I took AdminRent-backend, a multi-tenant Flask API for managing residential communities, in production, with 458 tests, and ran an experiment in three acts. Act 1: measure how an agent handles five real tasks with no help at all. Act 2: write an AGENTS.md that attacks exactly the failures act 1 surfaced. Act 3: re-run the same tasks and look at the delta. The conclusion, up front: the agent already is a good programmer. What it lacks is context. And twice, with the context in place, it wrote better security policy than what's in my main today.

Figure 1 · Commit replay. For each task, a worktree is created at the commit before the change, the agent gets the intent without the solution, and its diff is compared against the commit I merged.

Commit replay: why I didn't invent the tasks

The temptation is to write ten synthetic tasks and measure on those. It's a mistake: invented tasks look like what you imagine the agent does, not like what your team does. And you have no correct answer to compare against.

Instead I replayed commits I'd already merged. For each task: the base is the parent commit of the change; the prompt is the intent phrased the way you'd ask a colleague; the oracle is the real commit. Ground truth comes free, the tasks are representative by construction, and the same set works for re-measuring at every phase. The delta is fair because nothing else changed.

The detail that was hardest to respect: the Phase 0 prompt is minimal. No "respect the layers," no "filter by residential_id." Putting those rules in the prompt is doing the harness's job by hand, and it guarantees a flat delta. If you want to measure what writing the context is worth, you first have to measure the world without it.

The baseline: five tasks, no help

Five tasks, each probing a different dimension. One run per task. Fixed rubric: compiles, suite green, covered by a test, meets the intent, no scope creep, with at most one nudge from me.

T01
Add two fields to a search match · success
Four lines. Identical to the oracle, two new asserts in the test. With no domain ambiguity, the agent nails it. This matters: it confirms the harness adds nothing where there's no implicit decision.
T02
Unit members endpoint + RBAC · failure
Senior-quality code. Invented policy, and wrong: it excluded the portero. Six tests armoring the mistake. The acceptance criterion said "restricted to whoever it should be," and "whoever it should be" was written down nowhere.
T03
units-lookup endpoint "for admin and portero" · success
Same dimension as T02, except this time the prompt named the policy. It wrote @require_role("admin","portero") and three tests. The rule, written down, gets followed.
T07
Hide target_unit_ids from non-admins · with reservations
Functionally equivalent to the oracle, but it chose default=True (fail-open) and dropped the key instead of returning it empty, breaking the API contract. We come back to this below. It's the most uncomfortable finding of the baseline.
T08
Double-counted fines in the account statement · success (with a hint)
The subtlest bug, and the agent understood it: it explained that a fine already creates a mirror Transaction, so subtracting pending_amount double-counted it. It reached the exact fix. Honest caveat: my prompt named net_balance, which was part of the solution. I contaminated the task.

Three successes, one with reservations, one failure. But the number matters far less than the pattern: the technical stuff comes out on its own; the implicit domain and security stuff does not.

The two failure modes

Mode 1 · implicit domain policy. Facing an ambiguous criterion, the agent invents the business rule. It invents it reasonably, but reasonable ≠ what your product decided. And because it writes tests that assert its own decision, the error is armored in green. Green doesn't protect you from judgment.

Mode 2 · implicit security posture. This one is worse, because it reveals an incentive. In T07 the agent chose default=True:

def to_dict(self, include_targets: bool = True):
    ...

My real fix had chosen default=False: fail-closed, with a comment explaining why. Afterward I asked the agent why it had picked the insecure default. The answer, paraphrased: default=False broke an existing model test.

The agent optimizes for easy green, never for safe. Facing an unspecified security choice, it takes the path of least friction, and the path of least friction is almost never the defensive one. My fix chose the safe option and paid the price of updating that test. The agent had no way of knowing that price was worth paying, because nobody had written it down.

Pattern

The agent is already a good programmer. What you owe it is the implicit made explicit: RBAC policies, security defaults, domain semantics. Everything you know and never wrote down is exactly what the agent will invent.

The experiment that controls the variable

With both modes identified, I wrote a hundred-line AGENTS.md. Not answers to the tasks: principles. The who-sees-what table. The note that portero is a lateral role that inherits nothing and must be included on purpose. The fail-closed posture. The rule about never changing the response shape to hide a field.

Then I re-ran T02 and T07 in fresh worktrees, with the same agent and exactly the same minimal prompt. The only variable that changed was a text file at the root of the repo.

Figure 2 · The delta. Both baseline failure modes turned into successes. Nothing changed in the agent. What changed is what it could see.

The detail I liked most in T07: the agent chose default=False and updated the model test that decision broke. The written policy beat the easy-green incentive. It's the same model, with the same bias toward least friction; what changed is that the friction now had a stated reason.

I threw a third task at it that the AGENTS.md didn't anticipate: search a vehicle by license plate and return its owner. The guide says nothing about plate search. The agent derived a coherent policy from the principles: doorman yes (access control), owner/tenant no (privacy), with multi-tenant isolation tested. The harness doesn't just cure what failed: it applies its principles to tasks it never saw.

When the harness beat my own code

Here the experiment turned on me.

In T02, the guided agent added a restriction my oracle didn't have: owners and tenants can only see the members of their own unit. I went and looked at main. At that moment, in production, any owner could list the members, with email, of any unit in their building. A privacy leak between neighbors, sitting there since I wrote the endpoint.

On the plate-search task, the same thing: the agent gave access to admin / super_admin / portero. My real code gave it to owners and tenants. When I sat down to decide which of the two policies was correct, the answer was uncomfortable: the agent's.

Twice, the guided agent wrote better security policy than what was in my main. And that breaks my own rubric, which defined success as "equivalent to the oracle commit." When the agent beats the oracle, the rubric scores it as a failure. The oracle is a reference, not a verdict. I had to redefine the metric mid-experiment, which is exactly what a good experiment shouldn't need, and exactly the kind of thing you only discover by measuring.

Both bugs are fixed now. The fix went past the patch: list_members is fail-closed, and its docstring cites the line of the AGENTS.md that justifies it. The plate search ended up somewhere neither the agent nor I had proposed: a per-building privacy gate, where residents can only search when the building has privacy mode off, and the doorman is exempt because access control is his job. The written policy didn't win; it forced a decision that had never been made.

Both leaks now have a test with their name on it:

test_owner_cannot_list_members_of_other_unit       # the T02 bug
test_comunicado_target_unit_ids_hidden_by_default  # the T07 fail-closed

That's the whole cycle: the harness found the bug, the bug got fixed, and a sensor now stops it from coming back. Policy → code → test, traceable end to end.

Figure 3 · The closed cycle. The harness found the bug, the fix cites the policy line that justifies it, and a test now stops it from coming back.

From guides to sensors: what earned its place

A guide tells the agent what to do. A sensor makes the repo enforce it without a human looking at every PR. The rule I set for myself was that a sensor only ships if it attacks a failure the baseline produced, and if, run against the baseline's own bad code, it fails.

Four made the cut. A pre-commit hook (real lint errors, secret scanning, syntax) because it's the cheapest thing that exists and it prevents rather than diagnoses. An import-linter contract for the layers, written against the architecture that exists (services never imports Flask; api may read models) rather than an idealized one, because a sensor that reports two hundred pre-existing violations is noise, not a signal. The domain guardian, which is the two tests above. And a CI gate so all of it runs on every PR instead of on my laptop when I remember.

The fifth was never on the list, and it changed the most. The backend and the frontend used to be two repos, and every contract change was two commits that drifted apart. That was a class of bug the agent produced reliably, because the type it was coding against was a lie half the time. I consolidated them into a monorepo where the schema generates the contract and the contract generates the frontend types, with a CI check that fails if you forgot to regenerate. A contract change is now one atomic commit.

Pattern

The most valuable sensor turns a convention into a rule that fails on its own. And sometimes the cheapest way to build it is to change the shape of the repo so the mistake becomes impossible to express.

The meta-finding: my instrument lied to me three times

I wasn't planning to write this part.

Three times during the measurement, a sensor of the measurement, not the agent, produced a false reading that nearly caused a wrong verdict. All three were mine.

SENSOR 1
Desynchronized read
A view of the worktree showed the file broken with an IndentationError. Preliminary verdict: "the agent broke everything." On the disk the file was fine. The git diff, the source of truth, contradicted it.
SENSOR 2
Late comparison
I called T02 a "resounding success" before comparing against the oracle. The computational rubric was entirely green (compiles, tests pass, coverage exists) and the task had failed on the only dimension that mattered.
SENSOR 3
The cure worse than the disease
I "hardened" the scoring script to solve a false-green that didn't exist (it was problem 1). The hardened script introduced a false-red: it reported 1 failed once; three clean runs gave 458 passed. I fixed a misdiagnosed problem and created a new one.

A harness is built on its sensors. If the sensors aren't reliable and deterministic, the whole building rests on sand, and worse, it gives you false confidence. It is, ironically, the same principle the harness applies to the agent, applied to the instrument that measures the agent.

Warning

Don't call a verdict without all three: score.sh (what it delivered) + git diff (what it did) + git show <oracle> (what was expected). Any sensor alone lied. And never believe the agent when it says "the tests pass." In T08 that didn't match a clean run.

The autonomy ladder, and the rung I skipped

With the sensors in place, the obvious next move was to climb. Level 1 is: the agent opens a PR, I read it, I merge it. Level 2 is: the agent opens a PR, the gate reviews it, and it merges without me. That's a real threshold: the moment you take your own eyes off the path between a change and your users' data.

So I wrote the promotion criterion first, which is the right thing to do. Six weeks of observation. A log where every real catch by a sensor gets one line. Climb only if there are zero surprises on the sensitive surfaces (RBAC, multi-tenant isolation, the backend↔frontend type contract) because in a multi-tenant SaaS the expensive class of bug is irreversible: one customer's data seen by another. And a scope: auto-merge only for peripheral diffs. Never for anything touching the endpoints, the business logic, the models, or the migrations. Never for the surface where I'd already bled.

Then, the same day, I enabled auto-merge on every pull request. The observation window had zero entries in it. The scope existed only as a paragraph in a markdown file; the workflow that merges things never read that paragraph, because paragraphs don't execute.

Nothing bad happened. The pull requests that came through touched documentation and tests, so none of them landed on the surface the criterion was protecting. But that's the draw of the deck, not the guardrail. If the next PR had touched services/, it would have shipped to production, unread, and the policy I'd written three hours earlier would have watched it go by.

Takeaway

I wrote, in this very post, that writing a criterion and not applying it is worse than not having one. I wrote it about a success rate I never computed. It turns out it also applies to the merge path of a production system holding my neighbors' data. There, "the feeling of rigor without the rigor" is an incident waiting for the right pull request.

The fix took fifteen minutes: the workflow now reads the diff and refuses to auto-merge anything that touches the endpoints, the services, the models, the migrations, or the CI itself. Everything peripheral still merges on green. The interesting part is that I'd built four sensors to stop the agent from doing exactly this, and none of them were pointed at me.

Figure 4 · The merge gate, after the fix. Peripheral diffs merge on green without me. Anything touching the surface where I already bled waits for a human.

What this experiment doesn't prove

If you got this far expecting a success rate, you won't find one, and I'd rather say it myself before a comment does.

  • n = 5, one run per task, one model. Good for discovering failure modes: it found two real ones. Not good for asserting a percentage. Agents have high run-to-run variance; with n=1, T02 could have been bad luck.
  • I only re-ran the tasks that had failed. That guarantees a positive delta by construction. A harness can cure T02 and break T01, and I wouldn't have seen it. The regression check, re-running the successes with the AGENTS.md in place, is the run that's missing, and it's cheap.
  • The T02/T03 pair is not a control. I sold it as "the only thing that changed was whether the rule was written down." That's false: the endpoint changed, the prompt changed, the whole task changed. It's the signal that made me form the hypothesis, not the one that proves it. The clean proof is T02 without the harness against T02 with it: same task, same prompt, one variable.
  • I contaminated T08 by naming net_balance in the prompt, and counted it as a success anyway.
  • I was the judge and I knew the oracles. Mitigated by running the agent in cold, separate sessions, and always using commands on the machine as the source of truth. Mitigated, not eliminated.
  • The autonomy window is still open. The six weeks of observation that were supposed to justify level 2 haven't elapsed. I'm running with auto-merge on peripheral surfaces and an empty log. Ask me in August.

None of this invalidates the qualitative finding. The T02 and T07 delta is sharp and directional, but it bounds how much you can lean on it. An experiment that only confirms what you wanted to find is a well-dressed anecdote.

What I'd do differently

  1. Define the number before starting, and then compute it. I wrote in the plan that the success rate was "the number the harness has to move," with a kill criterion of 10 percentage points. Then I never computed it. Writing a criterion and not applying it is worse than not having one: it gives you the feeling of rigor without the rigor. I then did the same thing with the autonomy ladder, which suggests the lesson didn't take the first time.
  2. Include negative controls. Two of the ten tasks in the original plan (pagination, writing tests) have no implicit domain decision at all. If the harness "improves" those too, then the one measuring badly is me. A harness that improves everything is probably measuring my expectations, not the code.
  3. Fix the sensors before adding sensors. The Phase 2 plan had four new tools in it. The lesson from the baseline is that none of the four is worth anything until the suite stops lying. Determinism first, coverage second.

The pattern, generalized

If you have a repo with business rules that live only in two or three people's heads, and every repo older than a year does, then you already know what happens when an agent touches it. It will fail on the rule nobody wrote down, with impeccable code and green tests.

The experiment suggests three things, none surprising in hindsight and all invisible before measuring. First: the harness budget goes to domain and posture, not to technical scaffolding. Second: encoding principles generalizes; encoding answers does not. And third, the one that took me longest to accept: writing the rules for the agent is writing them for yourself. Two real privacy bugs in production didn't surface because the agent was brilliant. They surfaced because, for the first time in two years, something forced me to write down in a table who can see what.

The harness doesn't replace your judgment. It forces you to have it in writing. That, and not productivity, turned out to be the return.

With one asterisk I earned the hard way: writing it down isn't the same as enforcing it. A policy in a markdown file governs the agent, who reads it. It does not govern you, who wrote it. Whatever rule you care about (the promotion criterion, the merge scope, the success rate) has to live somewhere that runs.
