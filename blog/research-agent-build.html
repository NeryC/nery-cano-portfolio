What it is, in one paragraph

A Next.js 16 app that takes a research question, lets an LLM agent decide what to search for and which pages to read, and streams back a cited Markdown answer in roughly 25 seconds. The agent is a ToolLoopAgent with three tools (searchWeb, readPage, finalAnswer) and a single safety cap of stepCountIs(8). Termination is structural: when the model calls finalAnswer, the loop stops, because that tool has no executor. The whole thing, UI included, is about 700 lines.

Figure 1 · The research agent, end to end. The query enters the loop; searchWeb and readPage feed results back; finalAnswer ends it, and the answer streams to the browser.

How I built it, step by step

The three tools

The use case is intentionally simple, so the pattern stays visible. Three tools:

  1. searchWeb({ query }): calls Exa's /search, returns 5 results with title, URL, and pre-extracted highlights.
  2. readPage({ url }): calls Exa's /contents, returns up to 8,000 characters of clean page text.
  3. finalAnswer({ answer, sources }): the terminal tool, no executor, emits the cited response.

The agent itself is one declaration:

export const researchAgent = new ToolLoopAgent({
  model: 'anthropic/claude-sonnet-4.6',
  instructions: SYSTEM_PROMPT,
  tools: { searchWeb: searchWebTool, readPage: readPageTool, finalAnswer: finalAnswerTool },
  stopWhen: stepCountIs(8),
});

That's the whole agent. No while, no "did it say it's done?" parsing. A typical run: searchWeb → readPage → searchWeb → readPage → finalAnswer, and the loop terminates on that last call.

Streaming the agent's reasoning to the browser

ToolLoopAgent.stream() returns a UI Message Stream that AI SDK v6's useChat consumes natively:

// Server: Route Handler
export async function POST(req: Request) {
  const { messages }: { messages: UIMessage[] } = await req.json();
  const result = await researchAgent.stream({
    messages: await convertToModelMessages(messages),
  });
  return result.toUIMessageStreamResponse();
}

Each tool call streams as it happens (searchWeb start → result → readPage start → …) and the UI renders a card for each in real time. No polling, no manual SSE on the client.

Vercel AI Gateway over the direct Anthropic SDK

I route the model through Vercel AI Gateway instead of the Anthropic SDK directly. One API key for every provider, automatic failover if Anthropic has an outage, and a cost dashboard per request. Swapping claude-sonnet-4.6 for claude-haiku-4.5 is a one-string change:

export const RESEARCH_MODEL = 'anthropic/claude-sonnet-4.6';

Exa instead of scraping Google

Exa returns clean text content directly, no raw HTML, no anti-bot walls, no JavaScript-heavy pages. For an LLM that needs to read content, that's a big time-saver; the whole lib/exa.ts is about 50 lines.

Rate limiting per IP

The agent makes a Sonnet call per step plus an Exa call per search or read, which gets expensive fast on a public demo. A small in-memory Map<string, Bucket> keyed by IP caps it at 5 queries per IP per hour:

export function rateLimit(key: string, opts: { max: number; windowMs: number }) {
  const now = Date.now();
  const existing = buckets.get(key);
  if (!existing || existing.resetAt <= now) {
    buckets.set(key, { count: 1, resetAt: now + opts.windowMs });
    return { allowed: true, remaining: opts.max - 1 };
  }
  if (existing.count >= opts.max) return { allowed: false };
  existing.count += 1;
  return { allowed: true };
}

What I'd do differently

  1. Put the rate limiter in Upstash Redis from day one. The in-memory Map is fine for one instance, but it leaks the budget the moment Vercel spins up a second function: each instance has its own bucket, so the global limit becomes 5 × instanceCount. Cheaper to start with Upstash than to migrate under load.
  2. Eval the trajectory, not just the final text. The interesting failures are "it searched for the wrong thing" or "it called finalAnswer before reading enough", not "the Markdown is malformed". I'd score trajectories (number of searches, query relevance, sources cited) instead of only judging the output.
  3. Ship a stdio mode. Now that I've built three of these, I'd give each one an HTTP route and a thin CLI that runs the agent in a terminal. It would have caught two bugs faster than the deployed UI did, and it is the same shape as the MCP server in the next project.

The pattern, generalized

Strip the web search away and what's left is the reusable core: a declarative agent with a typed terminal tool. Swap the two work tools and the terminal tool, and the same 700-line skeleton becomes a coding agent, a support agent, or a data-extraction agent. The loop, the step cap, the streaming, and the deploy don't change.

What I keep from this build: make termination a tool call, validate everything with a schema, and prefer a declarative agent over a hand-rolled loop. Part 1 has the full case for why. The repo walks through every file, so steal it for your own project.
