Cómo un agente de IA usa tools en un loop, y la forma más limpia de dejarlo decidir cuándo terminó: una tool sin executor. Explicado desde cero, y aplicado a un agente de investigación real en la Parte 2.

Cada post sobre "agentes LLM autónomos" en 2024 tenía más o menos la misma forma: un loop escrito a mano que llama al modelo, corre la tool que pidió, y chequea alguna condición para decidir si parar.

let messages = initialMessages;
for (let i = 0; i < MAX_STEPS; i++) {
  const result = await streamText({ model, messages, tools });
  messages = [...messages, ...result.responseMessages];
  if (result.text.includes('FINAL ANSWER')) break;   // ← la parte frágil
}

Funciona. La parte frágil es esa última línea: decidir cuándo el agente terminó escaneando su texto por una frase mágica. Esta guía construye hasta una respuesta más limpia, la que hoy traen casi todos los frameworks de agentes, y al final vas a saber exactamente por qué le gana al loop artesanal.

Seis términos, primero

Seis palabras sostienen toda la idea. Definilas una vez y el resto se lee limpio.

LLM. El modelo (Claude, GPT). Predice texto, y se le puede contar de tools que puede llamar.

Tool. Una función que le exponés al modelo con un nombre, una descripción y un esquema de entrada tipado. El modelo puede pedir llamarla.

Execute. La función que corre cuando se llama una tool. Casi todas tienen una; el truco de esta guía es una tool que no tiene ninguna.

Loop de agente. El modelo llama una tool, ve el resultado, llama otra, y así hasta que decide que terminó. No un prompt: un loop de llamadas a tools.

Paso (step). Una vuelta de ese loop (una llamada a tool). Los agentes ponen un tope de pasos como red de seguridad.

Tool terminal. La tool cuya llamada significa "terminé". El corazón del patrón.

Qué es un loop de agente, en un párrafo

Un loop de agente es lo que convierte a un predictor de texto en algo que actúa. Le das al modelo un conjunto de tools y un objetivo. Elige una tool, la corrés, le devolvés el resultado, y vuelve a elegir. El loop se repite hasta que el modelo avisa que terminó. Todo lo interesante de los agentes (investigar, programar, dar soporte) es una variación de esta única forma. La única pregunta difícil es la que los loops de 2024 respondían mal: ¿cómo dice el modelo "terminé", y cómo se entera tu código?

El problema: ¿cómo sabe el loop cuándo parar?

En el loop artesanal, "parar" es una adivinanza sobre el texto del modelo. Dos versiones comunes, las dos frágiles:

  • Buscar una cadena mágica. Cortar cuando la salida contiene "FINAL ANSWER". Falla cuando el modelo reformula, o escribe la frase mientras todavía piensa en voz alta.
  • Chequear que no haya tool calls. Cortar cuando el último mensaje no llamó ninguna tool. Falla cuando el modelo llama una tool y escribe prosa que parece final en el mismo turno.

Las dos comparten una falla de raíz: la señal de parada vive en texto libre, así que estás parseando prosa para tomar una decisión de control de flujo. Y una vez que decidiste que paró, todavía tenés un segundo problema, sacar la respuesta final: escanear los mensajes, cortar el JSON, parsearlo, y manejar el caso en que el modelo escribió prosa en vez de JSON. Cada uno de esos pasos es un lugar donde romperse.

La anatomía: una tool que corta el loop

El arreglo es hacer que "terminé" sea una llamada a tool como cualquier otra. Definís una tool terminal, digamos finalAnswer, con un esquema tipado y sin executor. El framework de agentes sabe qué hacer: cuando el modelo llama una tool que no tiene execute, no hay nada que correr, así que el loop corta. La llamada a la tool es el resultado.

Y los dos problemas desaparecen de golpe. La terminación la decide qué función llamó el modelo, un campo estructurado, no el parseo de un texto. Y la respuesta final es la entrada validada de la tool, ya tipada, sin paso de extracción.

Figura 1 · El loop de un agente. Las tools de trabajo tienen executor y devuelven su resultado al loop. La tool terminal no tiene, así que llamarla corta el loop con un payload tipado.

Las piezas, en detalle

Una tool con executor, y una sin

En el AI SDK, una tool normal junta un esquema con un execute que hace el trabajo:

export const searchWebTool = tool({
  description: 'Search the web for relevant pages on a topic.',
  inputSchema: z.object({ query: z.string().describe('The search query') }),
  execute: async ({ query }) => {
    const results = await searchExa(query, 5);
    return results.map(r => ({ title: r.title, url: r.url, highlights: r.highlights }));
  },
});

La tool terminal es la misma forma con el executor sacado:

export const finalAnswerTool = tool({
  description: 'Submit the final answer to the user with structured citations.',
  inputSchema: z.object({
    answer: z.string(),
    sources: z.array(z.object({ url: z.url(), title: z.string() })),
  }),
  // ← sin execute a propósito
});

Sin execute. Solo un esquema. Cuando el modelo la llama, el loop corta y el { answer, sources } validado es lo que recibe tu código.

Patrón · Tool terminal

Usá una tool sin execute cada vez que quieras que el modelo elija entre "seguir trabajando" y "terminé". La decisión es el mismo mecanismo que elegir cualquier otra tool, mucho más confiable que parsear texto libre, y la entrada validada se vuelve tu valor de retorno tipado gratis.

Por qué le gana al loop artesanal

La terminación es estructural, no heurística. "¿Cuándo paramos?" lo responde qué función llamó el modelo, un campo estructurado, no una cadena que buscás. Ni reformular ni pensar en voz alta lo engaña.

La salida ya está tipada. Cuando el loop corta obtenés la entrada de la tool terminal, parseada y validada contra el esquema. No hay paso aparte de "extraer la respuesta" ni caso de JSON malformado que manejar.

El loop es declarativo. Un agente descrito como ({ model, tools, stopWhen }) se lee como una declaración completa de lo que hace. Sin cuerpo de loop escondido, sin condiciones de salida desparramadas en helpers. Volvés en un año y lo entendés en segundos.

La red de seguridad: un tope de pasos

Queda un riesgo: que el modelo nunca llame la tool terminal (se traba buscando en círculos). Así que le ponés tope al loop. En el AI SDK es stopWhen: stepCountIs(8): después de 8 iteraciones de tool call el framework frena al agente igual. Acota tanto el costo como el riesgo de loop infinito. La tool terminal es la salida normal; el tope de pasos es el freno de emergencia.

El agente mínimo, en código

Juntando todo, un agente entero es una sola declaración. Dos tools de trabajo, una tool terminal, un tope de pasos:

import { ToolLoopAgent, stepCountIs } from "ai";

export const agent = new ToolLoopAgent({
  model: "anthropic/claude-sonnet-4.6",
  instructions: SYSTEM_PROMPT,
  tools: {
    searchWeb: searchWebTool,     // tiene execute → corre, devuelve
    readPage: readPageTool,       // tiene execute → corre, devuelve
    finalAnswer: finalAnswerTool, // sin execute → corta el loop
  },
  stopWhen: stepCountIs(8),
});

const result = await agent.stream({ messages });
// result.finalToolCall.args está tipado como { answer, sources }

Sin while, sin for, sin parsear "¿dijo que terminó?", sin regex sobre la respuesta. El modelo elige entre "buscar", "leer" y "terminar" con el mismo mecanismo que usa para cualquier tool, y el loop corta en el instante en que elige "terminar".

El patrón, generalizado

La tool terminal sin execute va mucho más allá de los agentes de investigación. Donde sea que un LLM tenga que decidir cuándo terminó, esta es la señal más limpia:

  • Agentes de código: submitDiff({ files, summary }) es la tool terminal. El agente lee, edita, corre tests, y envía. El loop termina con un diff tipado, no con un regex sobre un transcript.
  • Agentes de soporte: escalateToHuman({ summary }) y resolveTicket({ answer }) son las dos terminales. El trabajo del agente es elegir una. El loop termina con una acción estructurada.
  • Agentes de extracción: emitRecord({ ...fields }) es la tool terminal. El agente crawlea, pregunta por otras tools, y emite el registro validado cuando tiene suficiente.

La señal de que querés este patrón es simple: te encontrás escribiendo código que lee la prosa del modelo para decidir qué hacer después. Convertí esa decisión en una llamada a tool y la adivinanza desaparece.
