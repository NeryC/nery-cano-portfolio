El patrón más limpio que he encontrado para agentes autónomos: una tool terminal sin execute

Por qué ToolLoopAgent + una tool finalAnswer sin execute reemplazó 80 líneas de código frágil con while-loop en mi research agent.

Nery Cano
Ingeniero Full-Stack Senior · Sistemas de IA/LLM · github.com/NeryC

Por qué ToolLoopAgent + una tool finalAnswer sin execute reemplazó 80 líneas de código frágil con while-loop en mi research agent — y por qué nunca volveré a escribir un loop de agente a mano.

Cada post que leí sobre «agentes LLM autónomos» en 2024 tenía más o menos la misma forma:

let messages = initialMessages;
for (let i = 0; i < MAX_STEPS; i++) {
  const result = await streamText({ model, messages, tools });
  messages = [...messages, ...result.responseMessages];

  if (result.text.includes('FINAL ANSWER')) break;
  if (result.finishReason === 'stop' && !result.toolCalls.length) break;
  // ...maybe other heuristics...
}

const finalText = parseFinalAnswer(messages);

Esto funciona. Lancé variantes de esto durante más de un año. Pero cada vez que volvía a uno de estos loops seis meses después para arreglar algo, me encontraba re-deduciendo:

  • ¿Cuándo se detiene realmente este loop?
  • ¿Cómo distingue el parser «ya terminé» de «estoy pensando en voz alta»?
  • ¿Qué pasa si el modelo alucina la frase mágica a mitad de un pensamiento?
  • ¿Por qué mi extracción de la respuesta final es un regex sobre el último mensaje del asistente?

Luego, a principios de 2026, construí un research agent sobre el ToolLoopAgent del Vercel AI SDK v6 y un patrón diminuto encajó en su sitio e hizo desaparecer todas esas preguntas.

El patrón es: define tu señal de «ya terminé» como una tool sin función execute. Esa frase suena aburrida. Quédate conmigo — es el truco de loop de agente más limpio que he visto.

Qué es, en un párrafo

Una app Next.js 16 que toma una pregunta de investigación, deja que un agente LLM decida qué buscar y qué páginas leer, y devuelve en streaming una respuesta en Markdown con citas en unos 25 segundos. El agente está definido como un ToolLoopAgent con tres tools — searchWeb, readPage y finalAnswer — y un único tope de seguridad de stepCountIs(8). La terminación es estructural: cuando el modelo llama a finalAnswer, el loop se detiene, porque esa tool no tiene ejecutor. Todo el agente cabe en unas 700 líneas de TypeScript incluyendo UI, tests y rate limiting.

Figura 1 · El loop del agente. El modelo elige una de tres tools en cada paso. searchWeb y readPage tienen ejecutores y realimentan su salida al loop. finalAnswer no tiene ejecutor — llamarla es el resultado, y el loop termina con el payload validado.

¿Qué es una «tool sin execute» y por qué usarla para esto?

En el AI SDK v6, cada tool se define con tool() y un input schema de Zod. La mayoría de las tools también tienen una función execute que de verdad hace el trabajo:

export const searchWebTool = tool({
  description: 'Search the web for relevant pages on a topic.',
  inputSchema: z.object({
    query: z.string().describe('The search query'),
  }),
  execute: async ({ query }) => {
    const results = await searchExa(query, 5);
    return results.map(r => ({ title: r.title, url: r.url, highlights: r.highlights }));
  },
});

Ahora mira finalAnswer:

export const finalAnswerTool = tool({
  description: 'Submit the final answer to the user with structured citations.',
  inputSchema: z.object({
    answer: z.string(),
    sources: z.array(z.object({
      url: z.url(),
      title: z.string(),
      snippet: z.string().optional(),
    })),
  }),
  // ← intentionally no execute
});

Sin execute. Solo un schema.

Dentro de ToolLoopAgent, el SDK ve que esta tool no tiene ejecutor. Cuando el modelo decide llamarla, el loop se detiene de inmediato. La llamada a la tool es el resultado. El payload validado y completamente tipado ({ answer, sources }) es lo que se devuelve a tu código.

Patrón · Tool terminal

Usa una tool sin execute siempre que quieras que el modelo elija entre «seguir trabajando» y «ya terminé». La decisión es el mismo mecanismo que elegir cualquier otra tool, lo cual es mucho más confiable que parsear texto libre buscando una frase mágica. El input validado se convierte automáticamente en tu valor de retorno tipado.

La implementación: tres tools y un ToolLoopAgent

El caso de uso es intencionalmente simple: un LLM que toma una pregunta de investigación, busca en la web, lee páginas, y sintetiza una respuesta en Markdown con citas. Tres tools:

  1. searchWeb({ query }) — llama al endpoint /search de Exa, devuelve 5 resultados con título, URL y highlights pre-extraídos.
  2. readPage({ url }) — llama al /contents de Exa, devuelve hasta 8.000 caracteres de texto limpio de la página.
  3. finalAnswer({ answer, sources }) — emite la respuesta final al usuario, con citas.

El agente en sí es una sola declaración:

export const researchAgent = new ToolLoopAgent({
  model: 'anthropic/claude-sonnet-4.6',
  instructions: SYSTEM_PROMPT,
  tools: {
    searchWeb: searchWebTool,
    readPage: readPageTool,
    finalAnswer: finalAnswerTool,
  },
  stopWhen: stepCountIs(8),
});

Ese es todo el agente. Sin while. Sin for. Sin parsear «¿dijo el modelo que terminó?». Sin regex sobre el texto de la respuesta. El modelo decide entre «buscar», «leer» y «terminar» usando el mismo mecanismo que usa para cualquier otra tool, y el loop termina en el momento en que elige «terminar».

Por qué esto es mejor que el patrón viejo

Tres razones por las que se me quedó:

1 · La terminación es estructural, no heurística

En el patrón viejo, «¿cuándo paramos?» lo decide lo que el modelo dice. Podrías matchear la cadena literal «FINAL ANSWER» o comprobar que el último mensaje del modelo no tiene llamadas a tools. Ambas son heurísticas. Fallan cuando el modelo reformula, cuando piensa en voz alta, cuando hace ambas — llama a una tool y emite algo de prosa que parece una respuesta final.

En el patrón sin execute, «¿cuándo paramos?» lo decide qué función llama el modelo. No hay ninguna cadena que parsear. La decisión vive en la salida de tool-call del modelo, que es un campo estructurado, no texto libre.

2 · La salida está completamente tipada

Cuando el loop se detiene, obtienes el input de la tool finalAnswer — ya parseado, ya validado contra el schema de Zod. No hay un paso aparte de «extraer la respuesta final».

const result = await researchAgent.stream({ messages });
// result.finalToolCall.args is typed as { answer: string, sources: Source[] }

Compáralo con el patrón viejo, donde tendrías que escanear los mensajes buscando la etiqueta, recortar el JSON, parsearlo, y manejar el caso en que el modelo escribió prosa en vez de JSON.

3 · El loop del agente es declarativo

ToolLoopAgent({ model, tools, stopWhen }) se lee como una descripción completa de lo que hace el agente. No hay un cuerpo de loop oculto, ni condiciones de salida dispersas, ni manejo de casos límite repartido entre funciones auxiliares. Si vuelvo a este código dentro de un año, sabré exactamente qué hace en cinco segundos.

Qué significa realmente stepCountIs(8)

stopWhen: stepCountIs(8) es la red de seguridad. Si el modelo nunca llama a finalAnswer (p. ej. se queda atascado en un loop de búsqueda), el SDK detendrá el agente tras 8 iteraciones de tool-call. Esto limita tanto el costo de cómputo como el riesgo de loops infinitos.

Para el research agent, 8 pasos sobran. Una sesión típica:

Step 1: searchWeb({ query: "Vercel AI Gateway vs LiteLLM" })   → 5 results
Step 2: readPage({ url: "https://vercel.com/.../ai-gateway" }) → 8000 chars
Step 3: searchWeb({ query: "LiteLLM features pricing 2024" })  → 5 results
Step 4: readPage({ url: "https://docs.litellm.ai/..." })       → 8000 chars
Step 5: finalAnswer({ answer: "...", sources: [...] })         → loop terminates

Si la pregunta es más difícil, el modelo usa más pasos. Si toca fondo, el tope te salva.

Algunos extras prácticos

Transmitir el razonamiento del agente al navegador

ToolLoopAgent.stream() devuelve un UI Message Stream que el hook useChat del AI SDK v6 consume de forma nativa:

// Server side — Route Handler
export async function POST(req: Request) {
  const { messages }: { messages: UIMessage[] } = await req.json();

  const result = await researchAgent.stream({
    messages: await convertToModelMessages(messages),
  });

  return result.toUIMessageStreamResponse();
}
// Client side
const { messages, sendMessage } = useChat({
  transport: new DefaultChatTransport({ api: '/api/research' }),
});

Cada llamada a tool se transmite al navegador a medida que ocurre — inicio de searchWeb → resultado de searchWeb → inicio de readPage → … — y la UI renderiza una tarjeta para cada una en tiempo real. Sin polling, sin fontanería SSE manual en el cliente; lo maneja el hook.

Vercel AI Gateway por encima del SDK directo de Anthropic

Enruto la llamada al modelo a través de Vercel AI Gateway en vez de usar el SDK de Anthropic directamente. Tres razones:

  1. Una API key para todo. Puedo cambiar de claude-sonnet-4.6 a claude-haiku-4.5 a gpt-5 cambiando una cadena. Sin gestión de keys nuevas.
  2. Failover automático de proveedor. Si Anthropic tiene una caída, el Gateway reenruta a Bedrock o Vertex sin que yo toque nada.
  3. Dashboard de costos. Cada petición queda registrada con costo, latencia y uso de tokens. Útil cuando intentas averiguar qué agente se está comiendo tu presupuesto.

La constante del modelo vive en lib/agent/model.ts:

export const RESEARCH_MODEL = 'anthropic/claude-sonnet-4.6';

Eso es todo. El Gateway lo toma de la variable de entorno AI_GATEWAY_API_KEY.

Por qué Exa en vez de Google/Bing

No necesito estrictamente un motor de búsqueda — podría parsear el HTML de Google o usar un proxy estilo SerpAPI. Pero Exa devuelve contenido de texto limpio directamente, sin HTML crudo, sin resistencia anti-bot, sin páginas cargadas de JavaScript. Para un LLM que necesita leer contenido, esto ahorra muchísimo tiempo. Todo el archivo lib/exa.ts tiene ~50 líneas.

Rate limiting por IP

El agente hace una llamada a Sonnet por paso más una llamada a Exa por cada searchWeb / readPage. Eso se vuelve caro rápido en una demo pública. Uso un rate limiter simple en memoria, un Map<string, Bucket> indexado por IP, limitado a 5 consultas por IP por hora. Para una demo de portfolio esto basta; en producción cambiarías a Upstash Redis para que el límite sea global entre instancias.

export function rateLimit(key: string, opts: { max: number; windowMs: number }) {
  const now = Date.now();
  const existing = buckets.get(key);
  if (!existing || existing.resetAt <= now) {
    buckets.set(key, { count: 1, resetAt: now + opts.windowMs });
    return { allowed: true, remaining: opts.max - 1 };
  }
  if (existing.count >= opts.max) return { allowed: false };
  existing.count += 1;
  return { allowed: true };
}

Lo que aprendí

Tres aprendizajes que me llevaré a todo futuro agente:

  1. Haz la terminación estructural. Si la condición de salida de tu loop es «el modelo dijo la frase mágica», tienes un agente frágil. Conviértelo en una llamada a tool. El modelo sigue decidiendo; tú obtienes una señal limpia y un payload tipado.
  2. Apóyate en ToolLoopAgent (o el equivalente en tu framework) en vez de armar loops a mano. Una definición de agente declarativa es más fácil de leer, más fácil de depurar, y más fácil de entregar a un compañero de equipo. El loop while hecho a medida casi nunca vale su costo.
  3. Schemas en todas partes. La misma lección que redescubro cada vez que trabajo con LLMs: el schema es el contrato. Una vez que interiorizas que las salidas de los LLM deberían ser datos tipados validados, no texto libre, cada agente que construyes queda más limpio.

Qué haría distinto

  1. Poner el rate limiter en Upstash Redis desde el día uno. El Map en memoria está bien para una demo de una sola instancia, pero deja escapar el presupuesto en el momento en que Vercel levanta una segunda instancia de función. Cada instancia tiene su propio bucket y el límite global se vuelve 5 × instanceCount. Más barato empezar con Upstash que migrar después bajo carga.
  2. Añadir evals para la calidad de las tool-calls, no solo para el texto final. Los fallos interesantes para este agente son «el modelo buscó la cosa equivocada» o «el modelo llamó a finalAnswer antes de leer suficientes páginas», no «el Markdown final está malformado». Construiría una pequeña suite de evals que puntúe trayectorias — número de búsquedas, relevancia de la query de búsqueda, fuentes citadas — en vez de solo juzgar la salida.
  3. Exponer un modo stdio para el agente. Ahora que he construido tres de estos agentes, estructuraría cada uno con un route handler HTTP y un binario de CLI delgado que corra el agente en una sesión de terminal. Habría cazado dos bugs más rápido de lo que lo hizo la UI desplegada, y es el mismo patrón que el servidor MCP de mi siguiente proyecto.

El patrón, generalizado

La tool terminal sin execute generaliza bien mucho más allá de los research agents. En cualquier lugar donde tengas un LLM que necesite decidir cuándo terminó, esta es la señal más limpia.

Algunas formas donde echaría mano de ella:

  • Agentes de programación — submitDiff({ files, summary }) es la tool terminal. El agente lee archivos, los edita, corre tests, y cuando está listo los envía. El loop termina con un payload de diff tipado, no con un regex sobre una transcripción de chat.
  • Agentes de atención al cliente — escalateToHuman({ summary, severity }) y resolveTicket({ answer }) son ambas tools terminales. El trabajo del agente es elegir una. El loop termina con una acción estructurada.
  • Agentes de extracción de datos — emitRecord({ ...fields }) es la tool terminal. El agente rastrea una página, hace preguntas de seguimiento a través de otras tools, y cuando tiene suficiente emite el registro validado.

Todo el research agent — UI, ruta de servidor, definición del agente, tres tools, rate limiter, 14 tests — cabe en unas 700 líneas de TypeScript. El README recorre cada archivo. Róbalo para tu propio proyecto.
