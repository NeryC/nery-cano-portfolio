La Parte 1 explica el patrón: un loop de agente que corta cuando el modelo llama una tool sin executor. Esto es la cosa real construida encima, desplegada en Vercel, con el código que importa.

🔗 github.com/NeryC/research-agent · demo

Qué es, en un párrafo

Una app Next.js 16 que toma una pregunta de investigación, deja que un agente LLM decida qué buscar y qué páginas leer, y devuelve en streaming una respuesta con citas en Markdown en unos 25 segundos. El agente es un ToolLoopAgent con tres tools (searchWeb, readPage, finalAnswer) y un único tope de seguridad de stepCountIs(8). La terminación es estructural: cuando el modelo llama finalAnswer, el loop corta, porque esa tool no tiene executor. Todo, con UI incluida, son unas 700 líneas.

Figura 1 · El agente de investigación, de punta a punta. La consulta entra al loop; searchWeb y readPage devuelven resultados; finalAnswer lo corta, y la respuesta va en streaming al browser.

Cómo lo construí, paso a paso

Los tres tools

El caso de uso es a propósito simple, para que el patrón quede visible. Tres tools:

  1. searchWeb({ query }) — llama a /search de Exa, devuelve 5 resultados con título, URL y highlights pre-extraídos.
  2. readPage({ url }) — llama a /contents de Exa, devuelve hasta 8.000 caracteres de texto limpio de la página.
  3. finalAnswer({ answer, sources }) — la tool terminal, sin executor, emite la respuesta con citas.

El agente mismo es una sola declaración:

export const researchAgent = new ToolLoopAgent({
  model: 'anthropic/claude-sonnet-4.6',
  instructions: SYSTEM_PROMPT,
  tools: { searchWeb: searchWebTool, readPage: readPageTool, finalAnswer: finalAnswerTool },
  stopWhen: stepCountIs(8),
});

Ese es el agente entero. Sin while, sin parsear "¿dijo que terminó?". Una corrida típica: searchWeb → readPage → searchWeb → readPage → finalAnswer, y el loop corta en esa última llamada.

Streaming del razonamiento del agente al browser

ToolLoopAgent.stream() devuelve un UI Message Stream que el useChat del AI SDK v6 consume de forma nativa:

// Servidor — Route Handler
export async function POST(req: Request) {
  const { messages }: { messages: UIMessage[] } = await req.json();
  const result = await researchAgent.stream({
    messages: await convertToModelMessages(messages),
  });
  return result.toUIMessageStreamResponse();
}

Cada tool call va en streaming a medida que pasa (searchWeb arranca → resultado → readPage arranca → …) y la UI renderiza una card por cada uno en tiempo real. Sin polling, sin SSE a mano en el cliente.

Vercel AI Gateway en vez del SDK de Anthropic directo

Ruteo el modelo por Vercel AI Gateway en lugar de usar el SDK de Anthropic directo. Una clave para todos los proveedores, failover automático si Anthropic tiene una caída, y un dashboard de costos por request. Cambiar claude-sonnet-4.6 por claude-haiku-4.5 es cambiar un string:

export const RESEARCH_MODEL = 'anthropic/claude-sonnet-4.6';

Exa en vez de scrapear Google

Exa devuelve texto limpio directo, sin HTML crudo, sin muros anti-bot, sin páginas pesadas de JavaScript. Para un LLM que necesita leer contenido, es un ahorro grande de tiempo; todo lib/exa.ts son unas 50 líneas.

Rate limiting por IP

El agente hace una llamada a Sonnet por paso más una llamada a Exa por búsqueda o lectura, lo que se pone caro rápido en un demo público. Un pequeño Map<string, Bucket> en memoria por IP lo topea en 5 consultas por IP por hora:

export function rateLimit(key: string, opts: { max: number; windowMs: number }) {
  const now = Date.now();
  const existing = buckets.get(key);
  if (!existing || existing.resetAt <= now) {
    buckets.set(key, { count: 1, resetAt: now + opts.windowMs });
    return { allowed: true, remaining: opts.max - 1 };
  }
  if (existing.count >= opts.max) return { allowed: false };
  existing.count += 1;
  return { allowed: true };
}

Qué haría distinto

  1. Poner el rate limiter en Upstash Redis desde el día uno. El Map en memoria está bien para una instancia, pero fuga el presupuesto apenas Vercel levanta una segunda función: cada instancia tiene su propio bucket, así que el límite global se vuelve 5 × cantidadDeInstancias. Más barato arrancar con Upstash que migrar bajo carga.
  2. Evaluar la trayectoria, no solo el texto final. Las fallas interesantes son "buscó lo que no era" o "llamó a finalAnswer antes de leer suficiente", no "el Markdown salió mal". Puntuaría trayectorias (cantidad de búsquedas, relevancia de la query, fuentes citadas) en vez de juzgar solo la salida.
  3. Shipear un modo stdio. Ahora que construí tres de estos, a cada uno le daría una ruta HTTP y un CLI fino que corre el agente en la terminal. Habría atrapado dos bugs más rápido que la UI desplegada, y es la misma forma que el servidor MCP del próximo proyecto.

El patrón, generalizado

Sacale la búsqueda web y lo que queda es el núcleo reusable: un agente declarativo con una tool terminal tipada. Cambiá las dos tools de trabajo y la terminal, y el mismo esqueleto de 700 líneas se vuelve un agente de código, uno de soporte, o uno de extracción de datos. El loop, el tope de pasos, el streaming y el deploy no cambian.

Lo que me llevo de este build: hacé que la terminación sea una llamada a tool, validá todo con un esquema, y preferí un agente declarativo antes que un loop artesanal. La Parte 1 tiene el caso completo de por qué. El repo recorre cada archivo, así que robátelo para tu propio proyecto.
