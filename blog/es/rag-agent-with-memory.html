Un parser de PDF sin dependencias para sobrevivir al serverless de Vercel

Cómo esquivé el «DOMMatrix is not defined» y lancé un agente RAG que de verdad despliega.

Nery Cano
Ingeniero Full-Stack Senior · Sistemas de IA/LLM · github.com/NeryC

Cómo esquivé el DOMMatrix is not defined y lancé un agente RAG que de verdad despliega — reemplazando un paquete npm de 200KB con 80 líneas de regex y zlib.inflateSync.

Llevaba tres horas construyendo una demo de «chatea con tus PDFs» en Next.js 16 cuando me topé con el tipo de error que te hace cuestionar tu carrera:

ReferenceError: DOMMatrix is not defined
    at /var/task/node_modules/pdfjs-dist/build/pdf.js:...

El PDF cargaba sin problemas en localhost. En el momento en que desplegué a Vercel — boom. Crash de la función. El error venía de pdfjs-dist, la librería subyacente que usa pdf-parse y prácticamente todo paquete «fácil» de extracción de texto de PDF en npm.

La razón es simple y brutal: pdfjs-dist fue diseñada para correr en el navegador. Depende de DOMMatrix, canvas y un puñado de otras APIs del DOM que siempre han sido parte de la Plataforma Web pero que nunca han existido en Node.js. En tu portátil, el dev server de Next.js es un entorno indulgente, tipo Node + JSDOM, que tapa el hueco. En el runtime serverless de Vercel no hay DOM, ni shim, ni piedad.

Necesitaba otro camino. Acabé escribiendo un parser de PDF sin dependencias usando nada más que el zlib nativo de Node y unas cuantas expresiones regulares. No es bonito. Nunca reemplazará a una librería de PDF de verdad. Pero es pequeño, es seguro en serverless, y despliega.

Qué es, en un párrafo

Un agente RAG en Next.js 16 que deja al usuario subir PDFs, los trocea en chunks, los embebe con Voyage AI, los guarda en pgvector de Supabase, y chatea sobre ellos con citas. La única pieza no obvia es el extractor de texto del PDF: ~80 líneas de TypeScript que leen los bytes crudos como latin1, inflan cualquier stream FlateDecode con el zlib nativo de Node, y sacan el texto de los operadores Tj / TJ con dos expresiones regulares. Cero dependencias npm más allá de lo que el resto de la app ya trae. Sobrevive a Vercel porque no tiene dependencias del DOM.

Figura 1 · Panorama general. El parser de PDF son tres pasos pequeños sobre zlib y regex. Todo lo que viene después es fontanería RAG estándar — trocear, embeber, guardar, chatear — cableado para correr de forma síncrona dentro de la ruta de subida, así cabe en el tope de 10 segundos de las funciones del plan Hobby de Vercel.

Una lección de anatomía de PDF en 60 segundos

Antes de que el parser tenga sentido, necesitas saber qué hay realmente dentro de un archivo .pdf. Yo sabía casi nada de esto cuando empecé; una tarde leyendo la referencia de Adobe PDF 1.7 me aclaró las cosas más rápido de lo que esperaba.

Un PDF es un árbol de objetos, codificado como una mezcla de ASCII y binario. El texto que de verdad ves en pantalla vive dentro de content streams — bloques de instrucciones tipo PostScript que dicen «mueve el cursor aquí, pon esta fuente, muestra esta cadena».

Un content stream típico se ve más o menos así:

BT
  /F1 12 Tf
  72 720 Td
  (Hello, world.) Tj
  0 -14 Td
  [(Multiple) -250 (segments)] TJ
ET

Dos operadores hacen casi todo el trabajo:

  • Tj — muestra una sola cadena. Sintaxis: (some text) Tj.
  • TJ — muestra un array de cadenas con ajustes de kerning. Sintaxis: [(Multiple) -250 (segments)] TJ.

Si puedes extraer las cadenas que hay dentro de los paréntesis de cada operador Tj y TJ en cada content stream, tienes, aproximadamente, el texto del PDF.

Hay una trampa. Los PDFs modernos casi siempre comprimen sus content streams con zlib (el filtro FlateDecode en la terminología de PDF). Un byte-stream crudo de PDF contiene largas tiradas de basura binaria que se descomprimen en el PostScript legible de arriba.

Así que el parser necesita hacer tres cosas:

  1. Encontrar cada bloque stream ... endstream.
  2. Detectar si está comprimido con FlateDecode e inflarlo si es así.
  3. Sacar el texto de los operadores Tj / TJ con regex.

Ese es todo el algoritmo.

Paso 1 — Encontrar cada stream

Un stream de PDF está delimitado por las palabras clave literales stream y endstream, con saltos de línea en medio. El contenido entre ellas son bytes opacos — o binario (comprimido) o PostScript (crudo).

La forma robusta de leer un PDF en Node sin perder bytes es leerlo como latin1. Latin-1 mapea cada byte 0–255 a un carácter único, así que el round-trip es sin pérdida. (UTF-8 corrompería cualquier cosa no-ASCII; la API Buffer es excesiva para escanear texto.)

import { inflateSync } from 'zlib'; // built-in, no npm install

function extractPdfText(buffer: Buffer): string {
  const raw = buffer.toString('latin1');
  const texts: string[] = [];

  const streamRegex = /stream\r?\n([\s\S]*?)\r?\nendstream/g;
  let m: RegExpExecArray | null;

  while ((m = streamRegex.exec(raw)) !== null) {
    const streamData = m[1];
    // ... step 2 below
  }

  return texts.join(' ').replace(/\s+/g, ' ').trim();
}

[\s\S]*? es el idioma perezoso de «matchea cualquier cosa incluyendo saltos de línea» — JavaScript no tiene re.DOTALL como Python, así que este es el workaround estándar.

Paso 2 — Descomprimir si es FlateDecode

Cada stream de PDF va precedido por un diccionario que te dice qué filtro (si hay alguno) se usó para codificar los datos. El diccionario vive en los bytes justo antes de la palabra clave stream. No necesitamos parsearlo por completo — comprobar si FlateDecode aparece en los ~500 bytes que preceden al stream es suficiente para el 99% de los PDFs del mundo real.

const streamStart = m.index;
const preceding = raw.slice(Math.max(0, streamStart - 500), streamStart);
const isFlate = preceding.includes('FlateDecode');

let content: string;
if (isFlate) {
  try {
    const compressed = Buffer.from(streamData, 'latin1');
    const decompressed = inflateSync(compressed);
    content = decompressed.toString('utf8');
  } catch {
    continue; // corrupted stream, skip
  }
} else {
  content = streamData;
}

inflateSync es parte del módulo zlib nativo de Node — sin npm install, sin bundling. Lanza una excepción ante input malformado, lo cual está bien porque los PDFs a veces contienen streams parciales o corruptos por los que no vale la pena tirar abajo toda la subida.

Tras inflarlo, content es el texto tipo PostScript que de verdad queremos.

Paso 3 — Sacar el texto de Tj y TJ

Dos regexes hacen la mayor parte del trabajo.

// Skip streams that don't have any text operators
if (!content.includes('Tj') && !content.includes('TJ')) continue;

// (text) Tj
for (const t of content.matchAll(/\(([^)\\]*(?:\\.[^)\\]*)*)\)\s*Tj/g)) {
  const s = t[1]
    .replace(/\\n/g, '\n')
    .replace(/\\(\d{3})/g, (_, oct) => String.fromCharCode(parseInt(oct, 8)));
  if (s.trim()) texts.push(s);
}

// [(text) -250 (more) ...] TJ
for (const t of content.matchAll(/\[([^\]]+)\]\s*TJ/g)) {
  for (const p of t[1].matchAll(/\(([^)\\]*(?:\\.[^)\\]*)*)\)/g)) {
    if (p[1].trim()) texts.push(p[1]);
  }
}

El lío de (?:\\.[^)\\]*)* está ahí para manejar caracteres escapados dentro de paréntesis — los PDFs permiten \) y \\ dentro de literales de cadena, que de otro modo terminarían el match antes de tiempo. La pasada \\(\d{3}) convierte los escapes octales (\101 → A) que aparecen en algunos PDFs.

Esto basta para extraer el texto de un paper clínico, un capítulo de libro de texto, o prácticamente cualquier PDF basado en texto que le metas a un pipeline RAG.

Lo que esto NO maneja

Quiero ser honesto: este parser es suficiente para RAG, no suficiente para todo.

EDGE 1
PDFs escaneados (solo imagen, sin streams de texto)
No hay contenido Tj/TJ que extraer — las páginas son imágenes aplanadas. Para esos necesitas OCR (Tesseract, AWS Textract, Google Document AI). El parser devolverá una cadena vacía y la ruta de subida puede manejarlo con elegancia.
EDGE 2
Codificaciones de fuente complejas (CMaps ToUnicode)
Algunos PDFs codifican caracteres con IDs de glifo no estándar que necesitan una búsqueda en el CMap para mapearlos de vuelta a caracteres legibles. La mayoría de los PDFs generados por consumidores (Word, Pages, LaTeX) están bien; los PDFs editoriales raros a veces devuelven basura.
EDGE 3
Estructura de tablas
El texto sale en orden de lectura, pero pierdes las relaciones de fila/columna. Bien para trocear y búsqueda semántica, mal para «extrae esta tabla como JSON».
EDGE 4
PDFs cifrados
Simplemente se verán como basura por el paso de descifrado que falta. Necesitas una librería de PDF de verdad si los PDFs cifrados entran en el alcance.

Para RAG, ninguno de estos suele importar. De todos modos vas a trocear el texto, embeberlo, y dejar que la búsqueda semántica haga el trabajo pesado.

Conectándolo al pipeline de ingesta

Una vez que el texto está fuera, el resto de la ingesta es estándar:

async function processDocument(blobUrl: string, sessionId: string, documentId: string) {
  const buffer = await fetch(blobUrl).then(r => r.arrayBuffer());
  const text = extractPdfText(Buffer.from(buffer));

  const chunks = chunkText(text);              // ~500 tokens each, 50-token overlap
  const embeddings = await embedBatch(chunks); // Voyage AI batch endpoint

  await supabase.from('chunks').insert(
    chunks.map((c, i) => ({
      document_id: documentId,
      session_id: sessionId,
      content: c.content,
      embedding: embeddings[i],
      chunk_index: i,
      page_number: c.pageNumber,
    }))
  );

  await supabase
    .from('documents')
    .update({ status: 'ready', chunk_count: chunks.length })
    .eq('id', documentId);
}

Un detalle sutil que me mordió: originalmente corría processDocument en segundo plano usando el after() de Vercel. Eso muere en el plan Hobby porque after() mantiene la función viva más allá del tope duro de 10 segundos. El arreglo fue poco romántico — correr la ingesta de forma síncrona dentro del handler de la ruta de subida, lo bastante rápido como para que un PDF de 1–10 páginas termine en 2–5 segundos. El usuario recibe status: 'ready' directamente en la respuesta y nunca ve un spinner de «procesando».

Para llevar

Las dependencias solo-navegador son una bandera roja en serverless. Si un paquete se construye sobre pdfjs-dist, canvas, jsdom, o cualquier API del DOM, asume que se romperá en serverless y echa mano de una alternativa nativa de Node — aunque eso signifique escribir tú mismo una herramienta más pequeña y tonta. El costo de mantenimiento de un «crash raro e irreparable en producción un viernes por la tarde» pesa más que la comodidad de npm install.

Lo que aprendí

Dos aprendizajes que me llevaré a todo futuro proyecto de LLM en serverless:

  1. Trata las dependencias solo-navegador como una bandera roja en serverless. Si un paquete se construye sobre pdfjs-dist, canvas, jsdom, o cualquier API del DOM, asume que se romperá en serverless y busca una alternativa nativa de Node — aunque signifique escribir tú mismo un parser peor y más simple. El costo de mantenimiento de un «crash raro e irreparable en producción un viernes por la tarde» pesa más que la comodidad de npm install pdf-parse.
  2. Un parser peor, más pequeño y propio le gana a uno más sofisticado que no puedes depurar. Este parser tiene ~80 líneas. Entiendo cada byte de él. Cuando algo sale mal, el stack trace apunta a un regex que yo escribí, no a tres capas de abstracción dentro de una librería. Esa propiedad vale mucho cuando estás entregando contra un plazo.

Qué haría distinto

  1. Mostrar «no se extrajo texto» como un error de primera clase en la UI. Ahora mismo, si un usuario sube un PDF escaneado, el parser devuelve una cadena vacía, el chunker produce cero chunks, y el documento se queda en status: 'ready' sin nada sobre lo que chatear. Una comprobación simple tras la extracción («el texto extraído estaba vacío — esto parece un PDF escaneado; el OCR aún no está soportado») evitaría confusión y abriría la puerta a conectar un fallback de OCR más adelante.
  2. Detectar CMaps ToUnicode y avisar antes de trocear. El parser actual devuelve en silencio galimatías para PDFs con codificaciones de glifo personalizadas. Una comprobación rápida de cordura (p. ej. la proporción de caracteres ASCII imprimibles en el texto extraído) podría señalarlos y decirle al usuario «este PDF usa fuentes no estándar; los resultados pueden salir corruptos».
  3. Sacar la ingesta de la ruta de subida una vez que salga de Hobby. La ingesta síncrona es la decisión correcta dentro de un tope de 10 segundos, pero limita el tamaño de PDF que puedes manejar y hace que la subida se sienta lenta con archivos más grandes. En Pro, pondría la ingesta detrás de Inngest o QStash con eventos de progreso apropiados transmitidos por SSE.

El patrón, generalizado

Este proyecto trata en realidad de una decisión más general: cuándo escribir tu propia herramienta de 80 líneas frente a instalar una librería de 200KB. La regla a la que he llegado es, más o menos:

  • Gana la librería cuando la spec es grande, los casos límite importan, y el entorno de runtime es generoso. ¿Renderizado de PDFs a imágenes en el servidor? Usa pdfjs-dist en un worker que tenga un polyfill del DOM.
  • Gana el código propio cuando solo necesitas una rebanada de la spec, tu runtime es restrictivo, y el costo de fallo de «la librería no despliega» es alto. ¿Extraer texto para RAG? 80 líneas le ganan a 200 KB.

El mismo patrón aplica mucho más allá del parseo de PDF — miniaturas de imágenes (Sharp vs hacer a mano un Resize vía node:zlib+chunks de PNG), parseo de CSV (Papa Parse vs un tokenizador de 20 líneas), renderizado de Markdown (un pipeline completo de unified vs una sola pasada de regex). Echa mano de la librería por defecto, pero reconoce el momento de escribir tú mismo la versión pequeña.
