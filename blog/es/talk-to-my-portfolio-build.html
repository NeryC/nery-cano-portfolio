Un servidor MCP sobre mi CV, mis proyectos y mis 164 certificados de Platzi, embebido en cualquier página con una línea de HTML. Acá está el código real, las decisiones y los seis bugs que rompieron producción antes de que algo funcionara.

La Parte 1 explica qué es MCP y por qué existe. Esta es la otra mitad: qué pasa cuando lo llevás a producción en un runtime serverless que no perdona nada.

La idea era simple de enunciar. Un reclutador abre mi portfolio y en vez de leer una lista de bullets pregunta lo que le importa: "¿tiene experiencia con RAG?", "¿qué construyó con TypeScript?", "¿está libre el jueves?". Y las respuestas salen de datos verificables, con el link al diploma, no de un modelo inventando adjetivos sobre mí.

Lo que no era simple: que ese mismo servidor sirviera para el chat de mi web y para Claude Desktop y para cualquier agente ajeno, sin escribir la lógica dos veces. Eso es exactamente lo que MCP resuelve, y es la razón por la que el proyecto existe.

La arquitectura, de punta a punta

Es un monorepo con pnpm. La regla que ordena todo: el servidor MCP no sabe que existe Next.js. Es un paquete de TypeScript plano, con su propio package.json y su binario, que podría publicarse en npm y correr por stdio. Next.js es solo uno de sus dos consumidores.

Figura 1 · La arquitectura. Un solo paquete define el servidor MCP; la ruta HTTP lo publica al mundo y el agente del demo lo consume sin salir del proceso.

Los cuatro paquetes, y qué hace cada uno:

PaqueteQué esDepende de
packages/sharedLos datos (JSON + Markdown), sus esquemas de Zod y los loaders con cachénada
packages/mcp-serverEl servidor: tools, resources, prompts, integraciones con Cal.com y Resendshared
apps/webNext.js 16.2.6: la ruta MCP, la ruta del agente, el chat y la página del widgetmcp-server, shared
packages/widgetEl botón flotante embebible, en Preact, bundleado con esbuildnada

Las dos rutas de apps/web consumen el mismo servidor por caminos distintos, y esa asimetría es deliberada. /api/mcp lo publica por Streamable HTTP para el mundo. /api/agent lo consume por dentro, sin tocar la red. El bug 2 explica por qué.

Cómo lo construí, paso a paso

El servidor: 9 tools

Usé la API de bajo nivel del SDK (Server + setRequestHandler) en vez de McpServer, porque quería que cada tool fuera un objeto plano y testeable sin levantar un servidor. Una tool es su esquema, su descripción y su función:

// packages/mcp-server/src/tools/types.ts
export interface Tool<S extends ZodTypeAny = ZodTypeAny> {
  name: string;
  description: string;
  inputSchema: S;
  execute: (args: z.infer<S>) => Promise<unknown>;
}

Y el registro es una lista. Nueve tools, tres familias: leer el portfolio, buscar en él, y agendar una llamada.

// packages/mcp-server/src/server.ts
const tools: readonly AnyTool[] = [
  listProjectsTool, getProjectTool, searchByTechTool,
  getSkillsTool, getExperienceTool, searchCoursesTool,
  getCourseTool, getAvailabilityTool, bookCallTool,
];

server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: tools.map((t) => ({
    name: t.name,
    description: t.description,
    inputSchema: zodToJsonSchema(t.inputSchema),
  })),
}));

El zodToJsonSchema de esa línea es el puente entre los dos mundos: yo escribo Zod porque me da tipos en execute, y el protocolo pide JSON Schema. El bug 4 nace justo acá.

La descripción de cada tool la escribí como si fuera prompt, porque lo es. Esta es la de listProjects, textual:

description:
  "List Nery's portfolio projects. Use when the recruiter asks 'what have you built?' " +
  "or wants an overview. Optionally filter by tag (e.g. 'rag', 'ai', 'react').",

Nombra al lector ("the recruiter"), da ejemplos de la pregunta que la dispara y aclara el filtro opcional. Con una descripción tipo "lista proyectos" el modelo la llamaba para todo.

Los resources: 4 formas en el código, 171 en el cable

Acá está la corrección más incómoda de este post. Durante meses mi README dijo "resources (4 URIs)" y el post viejo decía "4 resources". Es falso, y lo descubrí releyendo mi propio código para escribir esto.

En el código hay cuatro formas de URI. Pero listResources() no devuelve formas: devuelve la lista expandida, una entrada por cada cosa que exista.

// packages/mcp-server/src/resources/index.ts
const resources: Resource[] = [
  { uri: "portfolio://cv/full", name: "Full CV (markdown)", mimeType: "text/markdown", ... },
];
for (const p of projects) {
  resources.push({ uri: `portfolio://projects/${p.slug}/readme`, ... });
  resources.push({ uri: `portfolio://projects/${p.slug}/case-study`, ... });
}
for (const c of courses) {
  resources.push({ uri: `portfolio://courses/${c.slug}/certificate`, ... });
}

La cuenta real: 1 CV + 2 × 3 proyectos + 164 cursos = 171 resources. Con un ResourceTemplate el número en el cable sería otro, pero yo expando a mano, así que 171 es lo que un cliente recibe cuando pide el catálogo. La lección no es que 171 esté mal: es que escribí un número en el README sin ejecutar la cuenta, y sobrevivió meses de copiar y pegar.

Takeaway

Un número en un README es una afirmación sin test. Los tres números de este proyecto (9 tools, 3 prompts, 171 resources) salen de contar la estructura de datos, no de mi memoria; los dos primeros coincidían, el tercero no. Si un número te importa lo suficiente para publicarlo, te importa lo suficiente para asertarlo.

Los prompts: pedir inteligencia prestada, con plan B

Los tres prompts (pitch-for-role, compare-with-jd, tech-deep-dive) usan sampling: en vez de llamar a Anthropic con mi clave, le piden al cliente que corra la completación con su modelo. Si el cliente no soporta sampling, el prompt tiene que seguir funcionando igual. Eso significa escribir dos implementaciones de cada uno.

compare-with-jd es el más completo, porque usa las dos capacidades del cliente. Si la descripción del puesto que pegaste es muy corta, pide más con elicitation:

// packages/mcp-server/src/prompts/compare-with-jd.ts
if (parsed.jobDescription.trim().length < MIN_JD_CHARS) {
  if (opts.clientSupportsElicitation && opts.elicitationBridge) {
    const r = await opts.elicitationBridge({
      message: "The JD you pasted is very short. Could you paste the full role description?",
      requestedSchema: {
        type: "object",
        properties: { jobDescription: { type: "string", minLength: MIN_JD_CHARS } },
        required: ["jobDescription"],
      },
    });
    if (r.action !== "accept" || typeof r.content?.jobDescription !== "string") {
      return { description: "compare-with-jd (elicitation cancelled)", messages: [ /* ... */ ] };
    }
    parsed = { jobDescription: r.content.jobDescription };
  } else {
    // El cliente no sabe preguntar: devolvemos un mensaje que le pide al modelo que pregunte.
    return { description: "compare-with-jd (no elicitation support)", messages: [ /* ... */ ] };
  }
}

Fijate el r.action !== "accept". La persona puede aceptar, declinar o cancelar, y las tres son respuestas legítimas que tenés que manejar. Cerrar el diálogo no es un error.

Y si no hay sampling, el fallback no es un mensaje de disculpa: es una comparación determinística por palabras clave contra los tags de mis proyectos, las skills de mis cursos y el stack de mi experiencia. Peor que un LLM, mucho mejor que nada.

Las capacidades del cliente se descubren en el handshake, así que hay que reconfigurar los bridges cada vez que alguien se conecta:

// SDK 1.29.0 expone `oninitialized` (i minúscula) como callback público.
if (typeof (server as any).oninitialized !== "undefined") {
  (server as any).oninitialized = configure;
}

Ese as any con un comentario al lado es de las cosas que prefiero mostrar y no esconder. El callback existe, el tipo no lo declaraba, y la alternativa era peor.

El agente del demo: MCP sin red

El chat de /demo corre un ToolLoopAgent del AI SDK v6 que descubre las tools por MCP en el arranque y las mapea a tools del SDK. La conversión de esquemas es la parte que importa:

// apps/web/lib/agent.ts
import { ToolLoopAgent, jsonSchema, stepCountIs, tool } from "ai";

const aiTools = Object.fromEntries(
  discovered.map((t) => [
    t.name,
    tool({
      description: t.description,
      inputSchema: jsonSchema(t.inputSchema as Parameters<typeof jsonSchema>[0]) as never,
      execute: async (args) => { /* llama al server MCP y parsea el JSON */ },
    }),
  ]),
);

return new ToolLoopAgent({
  model: "anthropic/claude-sonnet-4-6",
  instructions: SYSTEM_PROMPT,
  tools: aiTools,
  stopWhen: stepCountIs(10),
});

El system prompt tiene una sola regla que me importa, y está en mayúsculas en el código: NEVER invent projects, courses, skills, or experience. Use the tools to ground every claim. Un asistente que exagera mi CV es peor que no tener asistente.

La conexión al servidor MCP no usa HTTP. Usa un par de transportes en memoria:

// apps/web/lib/mcp-client.ts
export async function connectInProcessPortfolio(name: string): Promise<NamedServer> {
  const [clientTransport, serverTransport] = InMemoryTransport.createLinkedPair();
  const { server } = buildServer();
  await server.connect(serverTransport);
  const client = new Client(
    { name: "web-host", version: "0.0.0" },
    { capabilities: { sampling: {}, elicitation: {} } },
  );
  await client.connect(clientTransport);
  return { name, client };
}

Es MCP completo: handshake, negociación de capacidades, tools/list, tools/call. Lo único que cambia es el caño. Y como el cliente declara sampling y elicitation, los prompts toman el camino bueno en vez del fallback.

El widget: 14 KB y una línea de HTML

El requisito era que cualquiera pudiera pegarlo en su sitio sin build, sin npm y sin que le rompa el CSS. La solución tiene tres piezas y está entera en un archivo de Preact.

Figura 2 · El widget, en tres piezas. Un script que monta un botón, un botón que abre un iframe, y un iframe que es una app entera: así el estilo del anfitrión no puede romper nada.

Preact en vez de React por peso: el bundle final son 14.553 bytes minificados, IIFE, sin dependencias externas.

// packages/widget/esbuild.config.mjs
await build({
  entryPoints: ["src/index.tsx"],
  bundle: true,
  outfile: "../../apps/web/public/widget.js",
  format: "iife",
  globalName: "PortfolioWidget",
  target: "es2020",
  minify: true,
  jsx: "automatic",
  jsxImportSource: "preact",
  alias: { react: "preact/compat", "react-dom": "preact/compat" },
});

El widget no renderiza el chat: abre un iframe a /widget. Parece un rodeo y es la decisión más barata del proyecto. El iframe aísla estilos en las dos direcciones, así que ni el Bootstrap del anfitrión me rompe el chat ni mi Tailwind le rompe la página.

Un detalle que sale de esa decisión: adentro del script los colores van hardcodeados, con un comentario que explica por qué.

// Portfolio palette (neryc.github.io/nery-cano-portfolio):
//   --accent      #2E75B6 (primary blue)
// Keeping these as literals (not CSS vars) because this script runs in the
// HOST page where those vars don't exist.
const COLOR_ACCENT = "#2E75B6";

Y el host del iframe se deduce del propio <script>, así el mismo bundle sirve en local y en producción sin configuración:

const WIDGET_HOST =
  (document.currentScript as HTMLScriptElement | null)?.src?.replace(/\/widget\.js.*$/, "") ??
  "https://talk-to-my-portfolio.vercel.app";

Los seis bugs que rompieron producción

Todo lo de arriba anduvo en localhost al primer intento. Después lo desplegué. Los seis, en orden de aparición, con el fix que quedó en el repo.

BUG 1
ENOENT: cannot open /var/task/packages/mcp-server/package.json
El servidor leía su propio package.json para sacar nombre y versión. En local el archivo está ahí; en Vercel el bundler mete la ruta en un chunk y el path relativo apunta a la nada. El fix es una línea, y es mejor que la original: un import con atributo de tipo, que el bundler resuelve en build y no en runtime.
// packages/mcp-server/src/server.ts
import pkg from "../package.json" with { type: "json" };
const serverInfo = { name: "portfolio-mcp", version: pkg.version };
BUG 2
La sesión MCP se evaporaba entre workers
La primera versión de /api/agent se llamaba a sí misma por HTTP contra /api/mcp. En local todo es un proceso y anda. En Vercel son dos funciones serverless: el initialize crea la sesión en el worker A, el tools/list cae en el worker B, que nunca vio ese ID. El fix fue dejar de usar la red.

Figura 3 · El bug 2, dibujado. A la izquierda el punto rojo llega a un worker que nunca vio esa sesión; a la derecha cliente y servidor comparten proceso y no hay nada que perder.

BUG 3
Invalid prompt: messages do not match the ModelMessage[] schema
El cliente usa useChat de @ai-sdk/react, que manda UIMessage[] (con partes). ToolLoopAgent.stream() quiere ModelMessage[]. El AI SDK v6 trae el puente y yo me había olvidado de llamarlo.
// apps/web/app/api/agent/route.ts
const { messages } = (await req.json()) as { messages: UIMessage[] };
const modelMessages = await convertToModelMessages(messages);
const result = await agent.stream({ messages: modelMessages });
BUG 4
TypeError: e is not a function al ejecutar una tool
El servidor MCP devuelve los esquemas como JSON Schema. El tool() del AI SDK espera un Zod (que tiene .parse) o un Schema envuelto con su helper jsonSchema(). Pasarle el JSON Schema crudo compila perfecto y explota en runtime cuando el SDK intenta validar. Compila no significa anda.
// ANTES — compila, explota en runtime
inputSchema: t.inputSchema as never,

// DESPUÉS — el helper del SDK envuelve el JSON Schema
inputSchema: jsonSchema(t.inputSchema as Parameters<typeof jsonSchema>[0]) as never,
BUG 5
El modelo pedía una clave que no existía
El agente fijaba el modelo con anthropic('claude-sonnet-4-6'), que llama a Anthropic directo y necesita ANTHROPIC_API_KEY. Todo el portfolio va por el AI Gateway de Vercel (una clave, failover entre proveedores, un dashboard). Pasar el modelo como string deja que el SDK rutee por el Gateway.
// ANTES: model: anthropic("claude-sonnet-4-6")
model: "anthropic/claude-sonnet-4-6",
BUG 6
ENOENT: cannot open packages/shared/data/projects/_index.json
Los loaders leen JSON y Markdown con readFileSync. El tracer de dependencias de Next.js sigue los import estáticos, así que empaqueta todo el JS; un readFileSync con una ruta armada en runtime le es invisible, y la carpeta de datos quedó afuera del deploy. Hay que decírselo a mano, por ruta.
// apps/web/next.config.ts
const nextConfig: NextConfig = {
  outputFileTracingRoot: path.join(process.cwd(), "..", ".."),
  outputFileTracingIncludes: {
    "/api/mcp/[[...path]]": ["../../packages/shared/data/**/*"],
    "/api/agent": ["../../packages/shared/data/**/*"],
  },
};
Takeaway

Cinco de los seis son el mismo bug con seis caras: en localhost hay un proceso, un sistema de archivos y ninguna frontera, y en serverless no hay nada de eso garantizado. Ninguno lo hubiera encontrado un test unitario, porque todos viven en el borde entre mi código y el runtime. La única defensa barata es desplegar temprano y desplegar seguido.

Tests y CI

Hay 38 archivos de test con 117 casos, de los cuales 2 están detrás de flags (@live pega contra el Cal.com real, @manual levanta el transporte HTTP). Lo que me parece que vale la pena robar es cómo corre el CI:

- name: Critical tests (sampling/elicitation/capabilities)
  run: pnpm test:critical
- name: Test
  run: pnpm test

Los tests marcados @critical cubren sampling, elicitation y negociación de capacidades, y corren antes que la suite completa. Si se rompe el contrato MCP quiero saberlo en el paso 4 del pipeline, no enterrado entre 117 resultados verdes. El modo de falla es "los críticos fallaron y el resto ni corrió", que es exactamente el mensaje que quiero leer.

Qué haría distinto

  1. El resource /readme es una mentira, y sigue viva. Escribiendo este post encontré que portfolio://projects/{slug}/readme y portfolio://projects/{slug}/case-study llaman los dos a loadProjectCaseStudy(): devuelven el mismo archivo. No existe un loadProjectReadme. Son 2 de mis 171 resources por proyecto y la mitad es una copia. Nadie se quejó porque nadie los pidió, que es justo lo que hace peligroso al bug: una API pública sin uso se despliega igual, y su silencio mide atención, no corrección.

  2. Los bridges son estado global de módulo. bridge-state.ts guarda las capacidades del último cliente que hizo initialize, a nivel módulo y no por sesión. Con dos clientes de capacidades distintas en el mismo worker caliente, gana el último. Está documentado en el código y es aceptable para un portfolio con una sesión a la vez; en cualquier cosa multiusuario es un bug esperando su turno. La forma correcta es colgar el estado de la sesión, no del módulo.

  3. El servidor no se conoce a sí mismo. projects/_index.json tiene tres proyectos: research-agent, multi-agent-code-reviewer y rag-agent-memory. Falta talk-to-my-portfolio. Si le preguntás al widget qué construí con MCP, te contesta con lo que sabe, y no sabe que él mismo existe. Es el proyecto más interesante de los cuatro y es el único que no está en los datos: lo agregué al final y me olvidé del archivo que le da de comer.

El patrón, generalizado

Sacale el CV y lo que queda es una forma reusable: escribí la capacidad una vez como servidor MCP, y después decidí por separado quién la consume y por qué caño. La misma lógica se publica por HTTP para el mundo y se consume en memoria desde adentro. Eso no es un truco de este proyecto: es lo que te compra separar el protocolo del transporte.

Aplica cada vez que tenés lógica de dominio que quieren consumir varios frentes. Un catálogo de productos que necesitan el chat de soporte, el asistente interno y un agente de un partner. Una base de conocimiento que quieren tu bot y el Claude Desktop de tus empleados. La alternativa habitual es una API REST más tres adaptadores de tools, uno por consumidor, y esos tres adaptadores son la deuda.

Cuándo no: si el único consumidor sos vos y va a seguir siendo así, esto es un proceso extra, un handshake y un paquete más en el monorepo a cambio de nada. La Parte 1 tiene la tabla de alternativas y la línea donde MCP empieza a pagar.

Lo que me llevo de los seis bugs no es una lista de gotchas de Vercel. Es que la distancia entre localhost y producción se mide en supuestos: un proceso, un disco, una memoria compartida. Cada uno de los seis fue un supuesto que localhost me dejó tener gratis y que producción me cobró.
