La Parte 1 explicó el patrón fan-out + supervisor en abstracto. Esta es la versión con código: un revisor de código donde tres agentes especializados miran el mismo archivo a la vez y un cuarto junta sus hallazgos en un informe con puntaje.

La arquitectura, de punta a punta

El flujo entero cabe en una línea: pegás código (o una URL de GitHub), y el sistema saca metadatos, dispara tres revisores en paralelo, y un supervisor consolida todo en un informe con puntaje 0-100, todo transmitido en vivo por streaming. Nada de esto necesita una base de datos: es una sola ruta de Next.js que emite eventos SSE.

El stack real: Next.js 16.2.5 (App Router), AI SDK v6 con generateObject, Zod 4 para la salida estructurada, y los modelos por el AI Gateway de Vercel: claude-sonnet-4.6 para los tres especialistas y claude-haiku-4.5, más barato, para el supervisor.

Figura 1 · El flujo completo. Entrada y metadatos, tres especialistas en paralelo, un supervisor que consolida y un informe, todo emitido en vivo por un solo endpoint SSE.

Cómo lo construí, paso a paso

Los modelos, en un solo lugar. Todo pasa por el AI Gateway de Vercel con una sola clave. Dos modelos: el bueno para encontrar problemas, el barato para consolidar.

import { createGateway } from '@ai-sdk/gateway';

const gateway = createGateway({ apiKey: process.env.AI_GATEWAY_API_KEY! });

// Los 3 especialistas: mejor razonamiento para encontrar problemas
export const REVIEW_MODEL = gateway('anthropic/claude-sonnet-4.6');
// El supervisor: síntesis + puntaje, más barato
export const SUPERVISOR_MODEL = gateway('anthropic/claude-haiku-4.5');

Un especialista = un rol + un esquema. Los tres agentes son la misma función con distinto prompt de sistema. El de seguridad busca inyección, secretos, XSS; devuelve una lista de hallazgos que Zod valida. Fijate que el prompt limita el largo (título máx. 10 palabras) para que la salida sea comparable entre agentes.

const SYSTEM = `Sos un revisor de seguridad. Analizá el código y encontrá
vulnerabilidades: inyección SQL/comandos, XSS, secretos hardcodeados,
deserialización insegura, validación faltante en los bordes.
Si el código es seguro, devolvé un array vacío.
Sé conciso: título máx 10 palabras, descripción máx 50.`;

export async function runSecurityAgent(code: string, meta: CodeMetadata) {
  const { object } = await generateObject({
    model: REVIEW_MODEL,
    schema: FindingArraySchema,        // Zod: fuerza la forma de la salida
    system: SYSTEM,
    prompt: `Lenguaje: ${meta.lang}

Código:
${code}`,
  });
  return object;                        // Finding[] — ya validado
}

Los esquemas Zod son el contrato entre agentes. Sin un esquema, el supervisor tendría que parsear texto libre de tres agentes distintos. Con generateObject + Zod, cada especialista devuelve exactamente la misma forma, y el supervisor recibe datos, no prosa.

export const FindingSchema = z.object({
  severity: z.enum(['critical', 'high', 'medium', 'low', 'info']),
  category: z.enum(['security', 'performance', 'maintainability']),
  title: z.string(),
  line: z.number().int().positive().optional(),
  description: z.string(),
  suggestion: z.string(),
  codeExample: z.object({ before: z.string(), after: z.string() }).optional(),
});

export const ReviewReportSchema = z.object({
  score: z.number().int().min(0).max(100),
  summary: z.string(),
  recommendation: z.string(),
  findings: z.array(FindingSchema),
});

La orquestación: el fan-out. Acá está el corazón. Los tres agentes se disparan con Promise.all, así que corren a la vez y el tiempo total es el del más lento, no la suma. Un detalle que parece menor y no lo es: anuncio los tres como «corriendo» antes del await, para que el timeline en vivo muestre paralelismo real y no una fila.

// Los 3 se anuncian ANTES de esperar — si no, el timeline miente
send({ type: 'agent', name: 'security', status: 'running' });
send({ type: 'agent', name: 'performance', status: 'running' });
send({ type: 'agent', name: 'maintainability', status: 'running' });

const [sec, perf, maint] = await Promise.all([
  runSecurityAgent(code, meta).then((f) => { send({ type: 'agent', name: 'security', status: 'done', findings: f }); return f; }),
  runPerformanceAgent(code, meta).then((f) => { send({ type: 'agent', name: 'performance', status: 'done', findings: f }); return f; }),
  runMaintainabilityAgent(code, meta).then((f) => { send({ type: 'agent', name: 'maintainability', status: 'done', findings: f }); return f; }),
]);

// El supervisor recibe las 3 listas juntas y devuelve UN informe
const report = await runSupervisorAgent([...sec, ...perf, ...maint], meta);

El supervisor consolida. Una sola llamada con el modelo barato. Recibe todos los hallazgos crudos y devuelve un ReviewReport: deduplica, reprioriza (seguridad > rendimiento > mantenibilidad a igual severidad), escribe el resumen y asigna el puntaje.

export async function runSupervisorAgent(findings: Finding[], meta: CodeMetadata) {
  const { object } = await generateObject({
    model: SUPERVISOR_MODEL,
    schema: ReviewReportSchema,
    system: SYSTEM,   // «deduplicá, reprioriza, resumí, puntuá 0-100»
    prompt: `Hallazgos crudos de 3 agentes:
${JSON.stringify(findings, null, 2)}`,
  });
  return object;
}

El transporte: SSE, sin cola. La ruta abre un ReadableStream y le pasa a orchestrate una función send que empuja cada evento como data: {...}. El navegador lo lee con EventSource. Sin Redis, sin cola de trabajos: un solo request que va escupiendo el progreso. Un rate limit de 3 por hora por IP evita que se me vaya la factura.

export async function POST(req: Request) {
  const ip = req.headers.get('x-forwarded-for')?.split(',')[0]?.trim() ?? 'unknown';
  if (!rateLimit(ip, { max: 3, windowMs: 3_600_000 }).allowed)
    return new Response('Rate limit exceeded.', { status: 429 });

  const stream = new ReadableStream({
    async start(controller) {
      const send = (e: object) => controller.enqueue(
        new TextEncoder().encode(`data: ${JSON.stringify(e)}\n\n`));
      try { await orchestrate(send, body); }
      finally { controller.close(); }
    },
  });
  return new Response(stream, { headers: { 'Content-Type': 'text/event-stream' } });
}

Qué se rompió, y cómo lo arreglé

La parte honesta. Nada de esto salió a la primera; varios se rompieron en silencio, que es lo peor de los sistemas multi-agente.

Bug · El supervisor deduplica de más

En el ejemplo de inyección SQL, los tres agentes producían 16 hallazgos crudos y el informe final mostraba uno solo. Funcionaba, pero desperdiciaba toda la historia multi-agente. La causa está en el prompt del supervisor: «deduplicá hallazgos que cubren el mismo problema subyacente» es demasiado amplio y colapsa hallazgos distintos-pero-parecidos. El arreglo son cinco líneas: ajustar «deduplicá» a «solo el mismo problema exacto» y exigir al menos un hallazgo por categoría no vacía.

Bug · El timeline mentía con await secuencial

La primera versión esperaba los agentes uno tras otro y recién ahí actualizaba la UI. El resultado: el timeline mostraba «seguridad… después rendimiento… después mantenibilidad», como si fueran en fila, aunque el objetivo era el paralelo. La corrección fue emitir los tres eventos running antes del Promise.all. El paralelismo tiene que verse, no solo pasar.

Bug · El rate limit se resetea en cold start

El límite de 3 por hora vive en un Map en memoria. En serverless, cada arranque en frío lo borra, así que un usuario decidido puede pasarlo. Para una demo de portafolio alcanza: es un freno de costos, no una cuota real. Una versión seria usaría Upstash o KV, y lo dejé anotado en vez de fingir que estaba resuelto.

Qué haría distinto

Transmitir los hallazgos a medida que llegan. Hoy la UI muestra que cada agente terminó, pero el detalle recién aparece cuando el supervisor cierra. Cada agente ya emite sus hallazgos; me faltó pintarlos apenas llegan, para que la espera se sienta más corta.

Arreglar la deduplicación antes de mostrarlo. Es el defecto que más resta a la demo, porque justo esconde el trabajo de los tres agentes. Es un cambio de cinco líneas que debí hacer antes de publicar.

Un paso de routing para lo trivial. Un snippet de diez líneas no necesita tres agentes y un supervisor. Un router liviano que decida «esto es chico, con un agente alcanza» ahorraría tres llamadas de cada cuatro en los casos fáciles.

El patrón, generalizado

Sacando el tema «código», lo que queda es el patrón de la Parte 1: especialistas en paralelo + un supervisor que consolida. Tres cosas lo hicieron funcionar acá y se repiten en cualquier caso: cada agente tiene un rol único (y así rinde mejor que un prompt que hace de todo), Zod convierte las salidas en un contrato (el supervisor recibe datos, no prosa que adivinar), y el Promise.all compra paralelismo casi gratis. El costo es real (cuatro llamadas por corrida, y coordinación que un solo agente no tenía), pero cuando el problema se parte en puntos de vista, se paga solo.

Si tenés un due diligence, un triage de soporte o una moderación donde varios expertos miran lo mismo y alguien tiene que sacar una conclusión, esta es la forma. El revisor de código es solo la excusa para mostrarla con código real.
