Tres agentes de IA en paralelo + un supervisor

Cómo Promise.all, Zod y Server-Sent Events se combinan en un revisor de código multi-agente.

Nery Cano
Ingeniero Full-Stack Senior · Sistemas de IA/LLM · github.com/NeryC

Cómo Promise.all, Zod y Server-Sent Events se combinan en un revisor de código multi-agente que es más fácil de razonar que un único LLM grande — y notablemente más rápido.

Un único LLM haciendo «revisión de código general» es mediocre.

No porque los LLMs sean malos — son geniales. Pero pídele a un modelo que encuentre bugs de seguridad, de rendimiento y de mantenibilidad todos a la vez, y verás aparecer los mismos modos de fallo una y otra vez: se inclina hacia la categoría que se ve más obvia en el snippet, deja caer hallazgos cuando el código se alarga, y produce reportes desbalanceados — cinco nimiedades de estilo, cero banderas de seguridad, sobre una inyección SQL que está clarísima.

Construí un revisor de código que no tiene ese problema. El truco fue dejar de pedirle a un único agente que fuera generalista y, en cambio, orquestar tres especialistas en paralelo con un supervisor que consolida sus hallazgos.

Este post recorre la arquitectura, las tres decisiones de ingeniería que defendería en cualquier review, y las sorpresas pequeñas pero importantes que surgieron al construirlo.

Qué es, en un párrafo

Una app Next.js 16 donde pegas un snippet de código (o una URL de GitHub) y tres agentes LLM especializados — Seguridad, Rendimiento, Mantenibilidad — corren cada uno su propia pasada sobre el código en paralelo. Un cuarto agente, el Supervisor, deduplica y puntúa los hallazgos combinados en un reporte de revisión de 0–100. Cada paso se transmite de vuelta al navegador como Server-Sent Events, así que el usuario ve una línea de tiempo en vivo en vez de un único spinner de 30 segundos. Todo el pipeline corre en una sola invocación de función de Vercel, sin Redis, sin cola de trabajos.

Figura 1 · La orquestación. Tres especialistas Sonnet corren dentro de un único Promise.all; cada uno emite su propio evento SSE en el momento en que termina. El supervisor Haiku consume el Finding[] fusionado y emite un ReviewReport tipado con una puntuación de 0–100.

Decisión 1 · Promise.all, no secuencial

La forma ingenua de correr tres agentes es esperarlos uno tras otro:

const sec = await runSecurityAgent(code, meta);
const perf = await runPerformanceAgent(code, meta);
const maint = await runMaintainabilityAgent(code, meta);

Eso funciona, pero es 3× más lento de lo que tiene que ser. Cada agente hace una llamada al LLM independiente; ninguno necesita la salida de los otros. Una latencia de 15 segundos por agente suma ~45 segundos de «cargando…».

Promise.all paraleliza las tres llamadas de forma trivial:

const [secFindings, perfFindings, maintFindings] = await Promise.all([
  runSecurityAgent(code, meta).then((findings) => {
    send({ type: 'agent', name: 'security', status: 'done', findings });
    return findings;
  }),
  runPerformanceAgent(code, meta).then((findings) => {
    send({ type: 'agent', name: 'performance', status: 'done', findings });
    return findings;
  }),
  runMaintainabilityAgent(code, meta).then((findings) => {
    send({ type: 'agent', name: 'maintainability', status: 'done', findings });
    return findings;
  }),
]);

// Supervisor needs all three — runs after.
const report = await runSupervisorAgent(
  [...secFindings, ...perfFindings, ...maintFindings],
  meta,
);

Tres cosas para notar:

  1. La latencia total ahora es max(t_sec, t_perf, t_maint) ≈ 15s, no su suma.
  2. Las cadenas .then(...) me dejan enviar un evento SSE en el momento en que cada agente termina, en el orden en que terminen. La UI muestra «✓ Mantenibilidad lista · 🔄 Seguridad aún corriendo» sin que yo tenga que cablear nada especial.
  3. El supervisor corre naturalmente después de que Promise.all resuelve — sin awaits extra; la dependencia de datos la impone el sitio de la llamada.

Si tus agentes sí dependen entre sí (p. ej. el agente B necesita la salida del agente A para planear su consulta), Promise.all no ayudará y deberías mantener el patrón secuencial. Pero para revisión de código, donde cada especialista tiene un lente distinto sobre el mismo input, el paralelo es latencia gratis.

Decisión 2 · Schemas de Zod como contratos de agente

Este es el patrón más importante que he adoptado en 2026 para trabajar con LLMs.

La forma vieja: pedirle amablemente al modelo que devuelva JSON, luego JSON.parse() y esperar que no pase nada raro. Esto falla de tres maneras:

  • El modelo devuelve una salida envuelta en markdown ```json ... ``` y tu parser se rompe.
  • El modelo inventa un campo que no existe en tu tipo de TypeScript.
  • El modelo omite un campo requerido; no lo notas hasta producción.

La forma nueva: pasarle un schema de Zod al generateObject del AI SDK v6, y dejar que el SDK imponga la forma a la salida del modelo.

const FindingSchema = z.object({
  severity: z.enum(['critical', 'high', 'medium', 'low', 'info']),
  category: z.enum(['security', 'performance', 'maintainability']),
  title: z.string(),
  line: z.number().int().positive().optional(),
  description: z.string(),
  suggestion: z.string(),
  codeExample: z.object({
    before: z.string(),
    after: z.string(),
  }).optional(),
});

const FindingArraySchema = z.array(FindingSchema);

export async function runSecurityAgent(code: string, meta: CodeMetadata): Promise<Finding[]> {
  const { object } = await generateObject({
    model: REVIEW_MODEL,         // claude-sonnet-4.6
    schema: FindingArraySchema,  // ← the contract
    system: SYSTEM_PROMPT_SECURITY,
    prompt: `Language: ${meta.lang}\nLines: ${meta.lines}\n\nCode:\n\`\`\`${meta.lang}\n${code}\n\`\`\``,
  });
  return object; // already typed as Finding[]
}

Tres cosas para notar:

  1. Sin parseo. Sin regex. Sin try { JSON.parse(text) } catch. Lo hace el SDK.
  2. Sin casteo. object está tipado como Finding[], fin de la historia.
  3. Sin la ansiedad de «¿alucinó el modelo un campo?». Si el modelo devuelve algo que no coincide con el schema, el SDK reintenta o lanza una excepción. De cualquier modo, el bug nunca se escapa a tu lógica de negocio.
Patrón · Schema como contrato

Trata el schema de Zod como una frontera dura entre «texto no confiable del LLM» y «datos tipados confiables». Tu lógica de negocio nunca debería tocar una respuesta cruda del modelo — solo la salida validada y parseada. Una vez que interiorizas esto, cada componente aguas abajo se vuelve más simple porque puede confiar en que sus inputs están bien formados.

El mismo patrón funciona para el supervisor:

const ReviewReportSchema = z.object({
  score: z.number().int().min(0).max(100),
  summary: z.string(),
  recommendation: z.string(),
  findings: FindingArraySchema,
});

export async function runSupervisorAgent(findings: Finding[], meta: CodeMetadata) {
  const { object } = await generateObject({
    model: SUPERVISOR_MODEL,    // claude-haiku-4.5
    schema: ReviewReportSchema,
    system: SUPERVISOR_PROMPT,
    prompt: `Code metadata: ${meta.lang}, ${meta.lines} lines\n\nRaw findings:\n${JSON.stringify(findings, null, 2)}`,
  });
  return object;
}

Misma idea, distinto schema. El prompt del supervisor son unas pocas frases que le dicen que deduplique, priorice por severidad (seguridad > rendimiento > mantenibilidad cuando la severidad empata), y asigne una puntuación de 0–100 con cortes (90+ = listo para producción, <50 = problemas significativos). El schema impone que la puntuación sea un entero 0–100, que el resumen sea una cadena, y que los hallazgos sigan siendo un Finding[] válido.

Decisión 3 · Un solo endpoint SSE, sin Redis

La mala arquitectura más tentadora para «trabajos multi-paso de larga duración» es:

  1. El cliente hace POST /jobs/start → devuelve un jobId.
  2. El cliente hace polling a GET /jobs/:id/status cada segundo.
  3. El servidor escribe el estado del trabajo en Redis o KV entre llamadas.

Esto añade toda una dependencia externa (Redis), introduce bugs de estado obsoleto, y le da al usuario una peor UX (los intervalos de polling de 1 segundo se ven a tirones).

El patrón más limpio: la propia petición POST es el stream. El navegador abre la conexión, el servidor empuja eventos a medida que el trabajo avanza, y cuando el trabajo termina la conexión se cierra.

Lado servidor:

export async function POST(req: Request) {
  const body = await req.json();

  const encoder = new TextEncoder();
  const stream = new ReadableStream({
    async start(controller) {
      const send = (event: object) => {
        controller.enqueue(encoder.encode(`data: ${JSON.stringify(event)}\n\n`));
      };

      try {
        await orchestrate(send, body);
      } catch (err) {
        send({ type: 'error', message: String(err) });
      } finally {
        controller.close();
      }
    },
  });

  return new Response(stream, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache',
      'Connection': 'keep-alive',
    },
  });
}

La función send es un closure simple que emite un Server-Sent Event. Dentro de orchestrate, cada paso llama a send() cuando empieza, cuando produce resultados, cuando da error:

send({ type: 'started', jobId });
send({ type: 'step', name: 'parseInput', status: 'done' });
send({ type: 'agent', name: 'security', status: 'running' });
// ... agent runs in parallel ...
send({ type: 'agent', name: 'security', status: 'done', findings });
// ... supervisor runs ...
send({ type: 'report', report });
send({ type: 'done' });

Lado cliente, el navegador lee el stream:

const res = await fetch('/api/review', { method: 'POST', body: JSON.stringify({ ... }) });
const reader = res.body!.getReader();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  buffer += decoder.decode(value, { stream: true });
  const events = buffer.split('\n\n');
  buffer = events.pop() ?? '';
  for (const event of events) {
    if (!event.startsWith('data: ')) continue;
    const parsed = JSON.parse(event.slice(6));
    setEvents(prev => [...prev, parsed]);
  }
}

Eso es todo. Sin setInterval. Sin un segundo endpoint. Sin Redis. El estado vive en la respuesta en streaming, el trabajo muere cuando la conexión se cierra, y el cliente recibe actualizaciones en tiempo real con cero sobrecarga de polling.

La única restricción: esto funciona dentro de una sola invocación de función. Si tu trabajo es demasiado lento para la duración máxima de función de la plataforma, sí necesitas una cola de trabajos de verdad. Pero para «tres agentes + un supervisor en 30 segundos en total», es perfecto.

Por qué Haiku para el supervisor

Una elección pequeña pero relevante para el costo: uso Sonnet para los tres especialistas y Haiku para el supervisor.

Los especialistas hacen razonamiento de verdad: parsear código, identificar vulnerabilidades, sugerir arreglos con diffs. Se benefician de la profundidad de Sonnet.

El supervisor hace trabajo mecánico: ordenar, deduplicar, priorizar, resumir, puntuar. Nada de eso requiere razonamiento profundo sobre el código en sí — el supervisor nunca ve el código, solo los hallazgos estructurados. Haiku sobra.

Concretamente:

  • Una revisión típica dispara 4 llamadas al LLM (3 especialistas + 1 supervisor).
  • Tres de esas son Sonnet, una es Haiku — significativamente más barato que cuatro llamadas a Sonnet, e indistinguible en calidad de salida para el trabajo específico del supervisor.

Esta es una lección generalizable: el modelo que eliges debería coincidir con la complejidad cognitiva de la tarea, no con lo impresionante del sistema en general. Un sistema multi-agente con un supervisor impulsado por Haiku sigue siendo un sistema sofisticado. No ganas nada pagando tarifas de Sonnet por una tarea que Haiku hace bien.

Lo que aprendí

Tres cosas que me llevaré hacia adelante:

  1. Especialistas + supervisor le gana a un único agente grande. Partir una tarea difusa en sub-tareas enfocadas le da al modelo menos sobrecarga cognitiva por llamada y produce una salida más balanceada. El modelo mental se parece más a un equipo humano que a un cerebro-en-una-cubeta.
  2. Los schemas de Zod son innegociables para las salidas de tools de LLM. Trata el schema como el contrato. Tu lógica de negocio nunca debería ver texto libre del modelo — solo datos tipados validados.
  3. Los streams le ganan al polling para cualquier trabajo que tarde 5–60 segundos. SSE dentro de un único POST es el patrón más limpio que he encontrado. Úsalo por defecto; echa mano de una cola de trabajos de verdad solo cuando de verdad excedas la duración de función.

Qué haría distinto

  1. Añadir una dimensión de agentes por «lenguaje» en vez de un set fijo de tres. La división actual de Seguridad / Rendimiento / Mantenibilidad es genérica; un revisor específico de Python o uno específico de SQL encontraría una clase de problema distinta (y probablemente mejor). La arquitectura lo soporta gratis — añadir agentes es una entrada en el array de Promise.all — simplemente no generalicé el orquestador para aceptar una lista de especialistas en runtime.
  2. Persistir los reportes en la base de datos para que sean compartibles por URL. Ahora mismo una revisión vive solo en la memoria del cliente; refresca la página y se va. Una pequeña tabla reviews con el snippet de entrada y el JSON del reporte dejaría a un usuario compartir /review/{id} y además produciría un gran dataset para evaluar a los agentes entre sí a lo largo del tiempo.
  3. Transmitir el razonamiento del supervisor, no solo su reporte final. El usuario ve una línea de tiempo en vivo hasta el paso del supervisor, y luego un bloque de salida repentino. Cambiar el supervisor a streamObject dejaría que la puntuación y el resumen aparecieran progresivamente, lo cual es una UX notablemente mejor para una cola de 5–8 segundos.

El patrón, generalizado

«Especialistas en paralelo + un supervisor que consolida» es un patrón que rinde cada vez que el trabajo se descompone en ejes independientes. Otros lugares donde encaja:

  • Revisión de documentos — especialistas legal, factual, estilístico; un supervisor-editor que produce la lista de ediciones consolidada.
  • Reranking de búsqueda — especialistas en relevancia, frescura, diversidad; un supervisor que elige el top-10 final.
  • Extracción de datos multi-fuente — un agente por formato de fuente (PDF, HTML, imagen); un supervisor que fusiona los registros extraídos y resuelve conflictos.

La forma es siempre la misma: sub-tareas acotadas corriendo en concurrencia, un modelo más barato encima consolidando las salidas estructuradas, cada paso transmitido de vuelta al usuario para que la latencia se sienta como ~max(individual) en vez de ~sum(individual).
