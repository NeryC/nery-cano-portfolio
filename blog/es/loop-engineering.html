En algún momento de principios de 2026 la palanca que le saca más a un modelo dejó de ser el prompt. Pasó a ser el loop dentro del que corre el modelo. Esto es qué es loop engineering, los cinco diales a los que se reduce, y cómo saber cuál le falta a tu agente que falla.

Durante casi todo 2023 y 2024, lo más inteligente que podías hacer con un LLM era aprender a hablarle. Funcionaba porque los modelos respondían de una sola vez. Después empezaron a correr en loops, a llamar tools, a actuar en muchos pasos, y el prompt dejó de ser el cuello de botella.

Solomon Hykes, que fundó Docker, resumió el peligro nuevo en una frase que todo el campo cita: "un agente de IA es un LLM destrozando su entorno en un loop". El reencuadre que se esconde en el chiste es lo importante. Un modelo más potente en un loop mal diseñado es más peligroso, no menos, porque ejecuta sus malas ideas con más competencia. Cuando el modelo es fuerte, el loop es donde vive la ingeniería.

Seis términos, primero

Seis palabras sostienen toda la idea. Definilas una vez y el resto se lee limpio.

Loop de agente. Un LLM corriendo tools en un ciclo (percibir, razonar, actuar, observar) hasta llegar a un objetivo o topar un límite. Lo único que separa un agente de una llamada suelta al modelo.

Tool / ACI. Una función que el agente puede llamar para hacer algo que el modelo solo no puede. Anthropic llama a toda la superficie de tools la agent-computer interface, el equivalente de una UI para el agente.

Condición de parada. La regla que decide cuándo corta el loop: éxito, un tope de iteraciones, un presupuesto, o detección de no-progreso.

Contexto. El conjunto de tokens que ve el modelo en un turno dado. Finito, y se degrada a medida que se llena (context rot).

Verificación. Cómo el loop prueba que hizo la cosa, en vez de declararse victorioso. Casi siempre un test suite o un grader.

Guardrail. Lo que evita que el loop haga daño: un sandbox, credenciales acotadas, un tope de presupuesto, un humano en el camino de revisión.

Qué es loop engineering, en un párrafo

Loop engineering es la disciplina de diseñar el loop dentro del que corre un agente, en vez de pulir una sola instrucción. Simon Willison lo nombró en septiembre de 2025: "mi definición preferida de un agente LLM es algo que corre tools en un loop para lograr un objetivo. El arte de usarlos bien está en diseñar con cuidado las tools y el loop que van a usar." La unidad de trabajo se movió de "escribir un buen prompt" a "diseñar un buen loop", y el loop se descompone en un puñado de diales que girás: las tools que el agente puede llamar, cuándo se le permite parar, qué guarda en contexto, cómo chequea su propio trabajo, y los guardrails que evitan que un loop descontrolado haga daño.

Por qué el loop pasó a ser la palanca, no el prompt

La forma más clara de ubicar loop engineering es como el piso más alto de un edificio de tres. Anthropic encuadra el piso del medio, context engineering, como "la progresión natural del prompt engineering", y la misma lógica sube un escalón más.

Figura 1 · Tres capas anidadas. El prompt optimiza una llamada; el context optimiza lo que ve el modelo cada turno; el loop optimiza la maquinaria que decide si el agente llega a un buen estado.

La relación no es competitiva, es anidada. Lo que más probablemente rompe un agente hoy no es una instrucción mal escrita, es un loop sin interruptor de apagado, sin estrategia de memoria, o sin forma de chequear su propio trabajo. Cuando un agente falla en producción, el fix casi siempre es cambiar uno de estos diales, no reformular el prompt.

La anatomía: un ciclo, cinco diales

En el centro está el ciclo que corre todo agente: percibe el estado, razona qué hacer, actúa con una tool, y observa el resultado, una y otra vez. Loop engineering es todo lo que construís alrededor de ese ciclo. Leé a la gente que define el campo (Willison, Anthropic, LangChain, Thoughtworks) y aparecen las mismas cinco palancas siempre.

Figura 2 · Las cinco palancas. El modelo corre el ciclo del medio. Vos diseñás los cinco diales alrededor: tools, parada, contexto, verificación, guardrails. Mirá el token recorrer el loop.

Las cinco palancas, en detalle

1 · Tools (la agent-computer interface)

Lo que el agente puede hacer. Anthropic dedicó más tiempo a optimizar las tools que el prompt en su trabajo de SWE-bench y acuñó ACI (agent-computer interface) como el equivalente de una UI para el agente. La falla común es un set de tools inflado: "si un ingeniero humano no puede decir con certeza qué tool usar en una situación dada, no podés esperar que el agente lo haga mejor". Para agentes de código, Willison prefiere comandos de shell antes que MCP porque "los agentes de código son muy buenos corriendo comandos de shell", con un AGENTS.md que documenta las pocas tools extra que van a necesitar.

2 · Condiciones de parada

Cuándo se le permite al loop cortar: por éxito, por un tope de iteraciones, por límite de presupuesto, o cuando detecta que no avanza. Un loop sin lógica de terminación o se declara victorioso temprano o no para nunca. Esta es la palanca de la que trata entera la guía de research-agent: una tool terminal sin execute convierte "terminé" en una llamada a tool estructurada en vez de un string que parseás.

3 · Gestión de contexto

Qué sobrevive entre turnos. El hallazgo clave de Anthropic es el context rot: a medida que crece la cantidad de tokens, la memoria se degrada, así que el contexto es un recurso finito con un "presupuesto de atención" que hay que gastar con cuidado. Las técnicas: compaction (resumir cerca del límite de la ventana y reiniciar), notas estructuradas (un NOTES.md que el agente mantiene fuera del contexto), y sub-agentes que queman decenas de miles de tokens cada uno pero devuelven un resumen ajustado de 1.000 a 2.000 tokens. La regla guía: encontrar el menor conjunto de tokens de alta señal que consiga el resultado.

4 · Verificación

Cómo el loop prueba que hizo la cosa. Es la palanca más repetida. Willison: el valor de un agente de código "se amplifica muchísimo con un buen test suite que pasa limpio". El harness de Anthropic para agentes de larga duración usa una lista JSON de 200+ features end-to-end, cada una marcada passes: false, así el agente no puede marcar una feature como hecha sin probarlo. Un loop que actúa sin verificar es un loop que shipea sus propias alucinaciones.

5 · Guardrails

Lo que evita que el loop haga daño. Sandboxes, credenciales acotadas, presupuestos, y un humano en el camino para lo irreversible. Willison corrió un agente de código contra una org de Fly.io dedicada con un presupuesto de $5 para que un loop descontrolado no pudiera gastar plata real, y bloquea el acceso de red a una lista de hosts confiables para frenar la exfiltración. Cuanto más fuerte el modelo y más abierto el "modo YOLO", más carga esta palanca.

Diagnóstico · ¿qué palanca falta?

El agente da respuestas seguras pero incorrectas → verificación. Loopea para siempre y quema tokens → condiciones de parada. Se olvida de lo que hacía en tareas largas → contexto. Habla de la tarea pero no la hace → tools. Hizo algo destructivo → guardrails. Cuando un agente falla, el fix es una de estas cinco, casi nunca el prompt.

Una segunda lente: el stack de cuatro loops

LangChain (sobre el "loopcraft" de swyx) encuadra el mismo oficio como loops que se apilan. El loop 1 es el agente (modelo + tools). El loop 2 lo envuelve en verificación (un grader devuelve las fallas). El loop 3 es event-driven: webhooks y cron disparan el agente sin un humano. El loop 4 es hill-climbing: leés las traces y reescribís el harness mismo. El valor compone en los loops 3 y 4, y un humano queda en el camino en cada nivel. Las cinco palancas son los diales; los cuatro loops son las alturas.

El loop mínimo, con las palancas visibles

Reducilo a lo más chico que muestre las cinco. Esta es la forma debajo de todo framework:

let context = [systemPrompt, task];           // palanca 3: contexto
for (let step = 0; step < MAX_STEPS; step++) {  // palanca 2: parada (tope)
  const action = await model.decide(context, TOOLS); // palanca 1: tools (la ACI)
  if (action.type === "finish") break;          // palanca 2: parada (tool terminal)

  const result = await sandbox.run(action);     // palanca 5: guardrails (sandbox)
  const ok = await verify(result);              // palanca 4: verificación
  context = compact([...context, action, result, ok]); // palanca 3 otra vez: mantenelo ajustado
}

Todo framework (ToolLoopAgent, LangGraph, Claude Code) es este loop con los diales en distintos valores. Loop engineering es elegir esos valores a propósito.

El patrón, generalizado

Loop engineering es la hermana de runtime de harness engineering. La vista de harness diseña el repo en el que trabaja un agente (guías que lo orientan antes de actuar, sensores que lo atrapan después). La vista de loop diseña el ciclo de runtime que ejecuta un solo agente (las cinco palancas). Misma ecuación abajo, Agente = Modelo + Harness, a dos alturas, y se cruzan limpio: un test suite es un sensor a nivel repo y la palanca de verificación a nivel runtime.

La señal de que estás haciendo loop engineering, lo llames así o no: te encontrás preguntando no "qué le digo al modelo" sino "qué puede llamar, cuándo para, qué recuerda, cómo se chequea, y qué evita que haga daño". Esas cinco preguntas son el trabajo. La Parte 2 convierte cada una en una decisión real sobre un SaaS multi-tenant en producción, donde un guardrail que falta filtra los datos de un vecino y una verificación que falta cuenta una multa dos veces.
