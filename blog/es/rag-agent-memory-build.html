Cómo construí un agente RAG con memoria

La parte 1 fue la teoría. Esta es el código real: el parser de PDF que sobrevive a serverless, el esquema de pgvector, el agente con herramientas y el bucle de memoria, todo abierto.

Nery Cano
Senior Full-Stack Engineer · AI/LLM Systems · github.com/NeryC

Esta es la parte 2. En la parte 1 conté qué es RAG y todas las estrategias detrás de cada paso. Acá bajo al barro: el código real de un agente que subís PDFs, chatea con citas, busca en la web cuando hace falta, y te recuerda entre sesiones. Todo verificable en el repo.

El sistema hace cuatro cosas. Subís un PDF y en unos segundos es consultable. Preguntás y responde citando documento y página, como [manual.pdf p.3]. Si el tema no está en tus documentos, busca en la web solo. Y al terminar cada charla, aprende hechos tuyos y los recuerda en la próxima. El stack: Next.js 16 con el Vercel AI SDK v6, chat con Claude Sonnet 4.6, embeddings Voyage, y Supabase pgvector de base de vectores.

La arquitectura, de un vistazo

Dos caminos que se encuentran en la base de vectores. La ingesta lleva un PDF de crudo a fragmentos consultables. El chat toma una pregunta y la responde con evidencia. Los dos comparten pgvector y el mismo aislamiento por sesión.

Figura 1 · La arquitectura real. Ingesta síncrona arriba, chat con streaming abajo; los dos lados se encuentran en pgvector, aislados por sesión.

El esquema: cuatro tablas y dos búsquedas

Todo el sistema se apoya en cuatro tablas de Postgres y dos funciones de búsqueda. Los chunks guardan los fragmentos de PDF con su vector; las memories, los hechos aprendidos, también con vector. Fijate que las dos tienen la misma forma: es lo que permite hacer RAG sobre documentos y sobre recuerdos con el mismo mecanismo.

-- fragmentos de documentos: texto + vector + de dónde salió
CREATE TABLE chunks (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id uuid REFERENCES documents(id) ON DELETE CASCADE,
  session_id text NOT NULL,        -- el aislamiento entre usuarios vive acá
  content text NOT NULL,
  embedding vector(1536),          -- el vector de Voyage
  page_number int DEFAULT 1
);
CREATE INDEX ON chunks USING hnsw (embedding vector_cosine_ops);  -- búsqueda rápida

-- memorias: hechos del usuario, con la MISMA forma que un chunk
CREATE TABLE memories (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id text NOT NULL,
  content text NOT NULL,
  embedding vector(1536),
  confidence float NOT NULL        -- qué tan seguro es el hecho
);

Además hay una tabla documents (una fila por PDF, con su estado) y una conversations (el historial de cada charla en JSON). Las dos búsquedas, match_chunks y match_memories, son gemelas: reciben un vector de consulta, filtran por sesión, y devuelven lo más parecido por coseno. Las veo en detalle en la sección de recuperación.

La ingesta, y el parser de PDF que sobrevive a serverless

La ingesta corre síncrona dentro del route de upload: subo el PDF a Vercel Blob, extraigo el texto, troceo en fragmentos de 500 tokens con 50 de solapamiento, embebo en lotes de 100 y guardo, todo antes de responder. Es a propósito: en el plan Hobby de Vercel la ventana de función es corta, así que meto el trabajo en una sola invocación en lugar de armar una cola. Funciona para PDFs chicos, que es el caso de un demo.

El único paso que se puso feo fue extraer el texto. Empecé con pdf-parse, que por dentro usa pdfjs-dist, y me estrellé apenas desplegué.

BUG 1
DOMMatrix is not defined
En mi laptop andaba; en Vercel, crash inmediato. pdfjs-dist fue diseñado para el navegador y depende de DOMMatrix, canvas y otras APIs del DOM que nunca existieron en Node. El runtime serverless no tiene DOM ni shim. La única salida real fue dejar de depender de esa librería.
BUG 2
El texto sale en binario
Un PDF es un árbol de objetos, y el texto vive en content streams casi siempre comprimidos con FlateDecode (zlib). Sin descomprimir, solo ves basura. Detecto el filtro en el diccionario previo al stream y lo inflo con el zlib nativo de Node, sin dependencias.
BUG 3
Bytes corrompidos al leer
Leer el PDF como UTF-8 destruye cualquier byte no ASCII. La lectura correcta es latin1, que mapea cada byte 0 a 255 a un carácter único, así el ida y vuelta no pierde nada. Después, dos regex sobre los operadores Tj y TJ sacan el texto.

El parser terminó en unas 80 líneas de TypeScript sin una sola dependencia de npm. El núcleo es este:

function extractPdfText(buffer: Buffer): string {
  const raw = buffer.toString("latin1")           // lectura sin pérdida
  const texts: string[] = []
  // 1. encontrar cada bloque stream...endstream
  for (const m of raw.matchAll(/stream\r?\n([\s\S]*?)\r?\nendstream/g)) {
    let content = m[1]
    // 2. inflar si el diccionario previo dice FlateDecode
    const dict = raw.slice(Math.max(0, m.index - 500), m.index)
    if (dict.includes("FlateDecode")) {
      try { content = inflateSync(Buffer.from(m[1], "latin1")).toString("utf8") }
      catch { continue }                            // stream roto: seguir de largo
    }
    // 3. sacar el texto de los operadores Tj
    for (const t of content.matchAll(/\(([^)\\]*(?:\\.[^)\\]*)*)\)\s*Tj/g)) {
      if (t[1].trim()) texts.push(t[1])
    }
  }
  return texts.join(" ").replace(/\s+/g, " ").trim()
}

No reemplaza a una librería seria de PDF: se pierde imágenes y layouts raros. Pero es chico, no depende del DOM, y despliega. Los últimos clavos fueron detalles finos del formato: paréntesis escapados dentro de las cadenas y escapes octales como \101 que hay que convertir a su carácter.

El patrón

Cuando una dependencia "fácil" te ata a un entorno que tu deploy no tiene, 80 líneas que hacen el 90% del trabajo a veces ganan a los 200 KB que hacen el 100%. La clave es saber qué 10% podés resignar: acá, los PDFs con imágenes, que un demo no necesita.

La recuperación: pgvector, coseno, por sesión

Cada usuario tiene su propio espacio de documentos, así que recuperar no es solo "buscar lo más parecido", es "buscar lo más parecido de esta sesión". Eso lo resuelve una función en Postgres que filtra por session_id antes de ordenar por coseno:

-- match_chunks: vecinos más cercanos, pero solo de la sesión del usuario
SELECT c.content, d.filename, c.page_number,
       1 - (c.embedding <=> query_embedding) AS similarity   -- <=> = distancia coseno
FROM chunks c
JOIN documents d ON d.id = c.document_id
WHERE c.session_id = p_session_id                              -- el aislamiento va primero
  AND 1 - (c.embedding <=> query_embedding) > match_threshold  -- descarta ruido
ORDER BY c.embedding <=> query_embedding
LIMIT match_count;                                            -- top-k

En el chat busco chunks y memorias en paralelo: los documentos con umbral 0.25 y top-5, las memorias con umbral 0.3 y top-3. El índice HNSW hace que, aunque haya muchos vectores, las dos búsquedas sean casi instantáneas.

const q = await embed(pregunta)                    // un solo embedding para ambas
const [docs, mems] = await Promise.all([
  supabase.rpc("match_chunks",   { query_embedding: q, match_count: 5, match_threshold: 0.25, p_session_id }),
  supabase.rpc("match_memories", { query_embedding: q, match_count: 3, match_threshold: 0.30, p_session_id }),
])

Ese filtro por sesión es la misma disciplina de aislamiento multi-tenant que trabajé a fondo en la serie sobre harness engineering: el dato de un usuario no puede aparecer jamás en la respuesta de otro, y la barrera vive en la base, no en el código de arriba.

El agente: tres herramientas y un límite de pasos

No es un RAG de un solo tiro. Es un agente: Claude recibe la pregunta con el contexto ya recuperado, pero además puede ir a buscar más si le hace falta. Con el Vercel AI SDK v6, definir eso son unas pocas líneas.

const tools = {
  search_documents: tool({    // volver a buscar en los PDFs, con otra formulación
    inputSchema: z.object({ query: z.string() }),
    execute: async ({ query }) => buscarChunks(query, sessionId),
  }),
  search_web: tool({          // Exa, cuando los documentos no alcanzan (3 resultados)
    inputSchema: z.object({ query: z.string() }),
    execute: async ({ query }) => searchWeb(query),
  }),
  save_memory: tool({         // guardar un hecho del usuario a propósito
    inputSchema: z.object({ fact: z.string() }),
    execute: async ({ fact }) => guardarMemoria(fact, sessionId, 0.9),
  }),
}

streamText({
  model: gateway("anthropic/claude-sonnet-4.6"),
  system: promptConContextoYMemorias,
  messages,
  tools,
  stopWhen: stepCountIs(6),   // el freno: como mucho 6 pasos de razonar-y-actuar
})

El stepCountIs(6) es el equivalente al principio de autonomía de la serie harness: un agente sin techo de pasos puede entrar en loop y quemar tokens. Seis alcanzan para "buscá en los documentos, si no está buscá en la web, después respondé", y ponen un límite duro.

La respuesta se transmite token a token por SSE (server-sent events), no de una sola vez. El servidor emite una secuencia de eventos que la interfaz consume en vivo: tool_start y tool_done (para mostrar "buscando en la web…"), text por cada pedacito de respuesta, memories_used con lo que recordó, citations con las fuentes, y done al final. Es lo que hace que se sienta un agente pensando y no una caja que tarda y escupe un bloque.

La memoria: dos formas de recordar

Acá está la parte que le da el nombre al proyecto. El agente recuerda cosas del usuario entre conversaciones, y lo hace con el mismo mecanismo de RAG: las memorias son texto embebido en la misma base de vectores, en su propia tabla. Recordar es, literalmente, hacer RAG sobre tus recuerdos.

Hay dos formas de escribir una memoria y una de leerla. La automática es la interesante: después de cada respuesta, un modelo chico y barato (Claude Haiku 4.5) lee la conversación y extrae hechos durables con un puntaje de confianza, usando salida estructurada. Solo se guardan los que superan 0.7:

const { object } = await generateObject({
  model: gateway("anthropic/claude-haiku-4.5"),
  schema: z.object({
    facts: z.array(z.object({
      content: z.string(),
      confidence: z.number().min(0).max(1),
    })),
  }),
  prompt: `Extraé hechos durables sobre el usuario de esta conversación.
Solo hechos de confianza alta (mayor a 0.7) útiles a futuro.\n\n${conversacion}`,
})
for (const f of object.facts.filter(f => f.confidence > 0.7)) {
  await guardarMemoria(f.content, sessionId, f.confidence)   // embebe y guarda
}

La explícita es la herramienta save_memory que ya vimos: el propio agente puede decidir guardar algo cuando lo detecta importante. Y al leer, en cada pregunta se recuperan las memorias más parecidas (umbral 0.3, top-3) y se inyectan en el system prompt bajo "lo que recuerdo de este usuario". El usuario nunca repite que es celíaco o que trabaja en finanzas: el agente ya lo sabe.

Figura 2 · El bucle de memoria. Los hechos se extraen, se embeben y se guardan; en la próxima conversación se recuperan y vuelven al prompt. RAG sobre tus propios recuerdos.

Aislamiento por sesión: el hilo que cose todo

Un detalle que aparece en cada tabla, cada búsqueda y cada herramienta: el session_id. Cada visitante recibe una cookie httpOnly de 24 horas, y esa sesión etiqueta sus documentos, sus fragmentos y sus memorias. Las dos funciones de búsqueda filtran por ella antes de comparar vectores. El resultado es que dos personas pueden subir PDFs distintos al mismo sistema y ninguna ve jamás lo del otro, sin login ni cuentas. Es multi-tenant de la forma más liviana posible, y la barrera vive en la base de datos, que es donde tiene que estar.

Lo que haría distinto

  1. Separar por página de verdad. Hoy el parser junta todo el PDF en un texto y trocea sobre eso, así que las citas dicen siempre p.1. Rastrear el número de página real durante la extracción haría las citas precisas, que es medio punto del valor de RAG.
  2. Trocear mejor. Uso longitud sobre 4 como proxy de tokens y corto por párrafos. Funciona, pero un contador de tokens real y un troceo consciente de la estructura reducirían los fragmentos partidos.
  3. Sumar reranking. Recupero por similitud pura. Traer 20 candidatos y reordenarlos con un modelo dedicado, como conté en la parte 1, es la mejora de mayor retorno que tengo pendiente.
  4. Ingesta asíncrona para PDFs grandes. La ingesta síncrona entra en la ventana serverless para documentos chicos. Para archivos grandes, lo correcto es una cola: responder rápido y procesar en segundo plano.

El patrón, generalizado

Si mirás el sistema entero, hay un solo mecanismo repetido tres veces. Los documentos son RAG. La memoria es RAG sobre lo que el agente aprendió de vos. La búsqueda web es RAG con un índice que no controlás. Embeber, buscar por cercanía, inyectar en el prompt: siempre lo mismo. Una vez que ves RAG como "traer la evidencia justa en el momento justo", deja de ser una técnica y pasa a ser una forma de pensar cualquier sistema que tenga que responder sobre datos que el modelo no vio.

Y la parte difícil es siempre la misma, la que insistí en la parte 1: recuperar bien. Si las respuestas son malas, casi nunca es el modelo que genera; es el contexto equivocado. Arreglá la recuperación y arreglás RAG.
