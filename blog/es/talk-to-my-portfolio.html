Construyendo «Talk to My Portfolio»: un servidor MCP + widget de IA embebible

Cómo lancé un widget de chat que deja a quien me visita interrogar mi CV, mis proyectos y mis 164 certificados de Platzi — y los seis bugs de producción que tuve que arreglar para que funcionara en Vercel.

Nery Cano
Ingeniero Full-Stack Senior · Sistemas de IA/LLM · github.com/NeryC

Este es el cuarto y más ambicioso proyecto de la serie de portfolio Talent Global. Los tres anteriores (research agent, revisor multi-agente, RAG con memoria) eran apps Next.js independientes. Este es una implementación de protocolo, un monorepo multi-paquete y una pieza de JavaScript que tiene que incrustarse en la página HTML de otra persona — mi propio portfolio.

Si abriste este post desde la esquina inferior derecha de neryc.github.io/nery-cano-portfolio, ya hiciste clic en el FAB. Ese pequeño botón 💬 carga widget.js, monta un componente Preact, abre un iframe que apunta a una página Next.js y desde ahí habla con un servidor Model Context Protocol que expone una vista estructurada de toda mi trayectoria profesional.

Lo divertido: diseñarlo tomó diez minutos y depurarlo, varias horas. Seis bugs de producción separaban «pasa los 115 tests en local» de «de verdad funciona en Vercel». Esta es la versión larga de lo que hay dentro.

🔗 github.com/NeryC/talk-to-my-portfolio · demo en vivo · endpoint MCP

Qué es, en un párrafo

Un servidor MCP expone 9 tools, 4 resources y 3 prompts que leen de archivos JSON/Markdown que describen mi CV, mis proyectos, mi experiencia laboral, mis skills y 164 diplomas reales de Platzi extraídos con Playwright. Cualquier cliente compatible con MCP (Claude Desktop, ChatGPT desktop, tu propio agente) puede conectarse por Streamable HTTP y preguntar cosas como «¿Qué proyectos ha construido Nery con TypeScript y Anthropic Claude?». Las respuestas vienen de los datos estructurados, no del set de entrenamiento del modelo — así que no hay alucinaciones sobre mi perfil.

El mismo servidor también alimenta un widget de chat que embebo en este mismo portfolio. Eso le da a quien recluta una forma, a un clic, de preguntar lo que sea sobre mí sin tener que scrollear 1.500 líneas de HTML.

Figura 1 · Panorama general. El portfolio carga widget.js, que monta un FAB y un iframe apuntando a /widget en Vercel. De ahí en adelante todo es un único despliegue de Next.js — el servidor MCP corre in-process junto al agente.

El monorepo, en breve

Cuatro paquetes del workspace de pnpm:

  • apps/web — Next.js 16 (App Router, Turbopack). Aloja /widget (la UI del iframe), /demo (el chat independiente), /api/agent (la ruta del agente con streaming) y /api/mcp (el transporte MCP por Streamable HTTP).
  • packages/mcp-server — @neryc/portfolio-mcp. TypeScript puro que registra tools/resources/prompts contra el SDK de MCP. Tiene su propio binario de CLI para el modo stdio (para que Claude Desktop pueda lanzarlo en local).
  • packages/shared — schemas de Zod + cargadores de datos tipados + los archivos semilla JSON/Markdown. Tanto las apps como el servidor MCP importan de aquí.
  • packages/widget — El bundle de Preact + esbuild que se convierte en widget.js. La salida se deja en apps/web/public/ para que Vercel lo sirva desde el origen público.

Más scripts/sync-platzi — un scraper de Playwright que inicia sesión en Platzi una vez, recorre el perfil público + el dashboard de aprendizaje y vuelca 164 cursos reales con sus URLs de diploma en packages/shared/data/courses.json. Esos datos son los que devuelve el servidor MCP cuando un agente pide «muéstrame los certificados de Python de Nery».

¿Qué es MCP y por qué usarlo para esto?

El Model Context Protocol es un pequeño protocolo JSON-RPC que Anthropic publicó a finales de 2024 para estandarizar la conversación entre un cliente LLM y un backend de tools/datos. Un servidor expone:

  • Tools — funciones invocables con parámetros Zod (o JSON Schema). El mismo rol que el function calling de OpenAI, salvo que es portable entre proveedores.
  • Resources — URIs de solo lectura que el cliente puede traer. Piensa en archivos, pero direccionables por esquema (portfolio://cv/full, portfolio://projects/research-agent/case-study).
  • Prompts — plantillas de prompt parametrizadas que agrupan instrucciones + argumentos + preferencias de sampling.

También soporta sampling (el servidor le pide al cliente que ejecute una llamada al LLM) y elicitation (el servidor le pide al cliente que recoja input estructurado del usuario). Estas son las piezas que hacen que MCP se sienta distinto de «solo function calling».

Figura 2 · El handshake de MCP. El cliente inicializa con sus capabilities; el servidor responde con lo que puede hacer; las llamadas a tools van y vienen; las peticiones de sampling y elicitation van también en el otro sentido — el servidor le pregunta al cliente.

El formato de transmisión es JSON-RPC 2.0 plano. El transporte puede ser stdio (subproceso local, usado por Claude Desktop) o Streamable HTTP (un POST único que devuelve o bien JSON o un stream SSE, usado por clientes remotos). El SDK trae los transportes de servidor y de cliente para cada uno.

Entonces, ¿por qué molestarse con MCP para el chatbot de un portfolio? Podría haber escrito una simple ruta de API en Next.js. Tres respuestas:

  1. Portabilidad. El mismo servidor corre en local por stdio (para que un hiring manager pueda añadirlo a Claude Desktop y preguntar desde su herramienta de siempre) y en remoto por HTTP (para que yo pueda alimentar un widget público). Un solo camino de código, dos transportes.
  2. Es el proyecto, no solo un medio. Al rol que apunto (LLM/AI Full-Stack Engineer) le importa que sepa implementar protocolos, no solo llamarlos. MCP es lo más parecido a un «estándar» en tooling de agentes ahora mismo; hacerlo bien es la demo.
  3. Sampling y elicitation. Sin esto, «agendar una llamada» sería un stub del lado del servidor que solo manda un email a Cal.com. Con ellos, el flujo de reserva puede pedirle al cliente el nombre del usuario + su horario preferido vía un formulario estructurado, y luego pedirle al LLM del cliente que redacte un mensaje de confirmación personalizado. El servidor orquesta; el cliente renderiza.

El servidor MCP: 9 tools, 4 resources, 3 prompts

Cada tool es un objeto diminuto: un input schema de Zod y una función async execute. El mismo patrón que tool() del Vercel AI SDK, solo que agnóstico al transporte.

// packages/mcp-server/src/tools/list-projects.ts
import { z } from 'zod';
import { loadProjects } from '@neryc/portfolio-shared';

export const listProjectsTool = {
  name: 'listProjects',
  description: 'List all portfolio projects with slug, title, and tagline.',
  inputSchema: z.object({}),
  execute: async () => {
    const projects = loadProjects();
    return {
      projects: projects.map((p) => ({
        slug: p.slug,
        title: p.title,
        tagline: p.tagline,
        tags: p.tags,
        demoUrl: p.demoUrl,
        githubUrl: p.githubUrl,
      })),
    };
  },
};

Los resources exponen URIs legibles que los clientes LLM pueden traer directamente:

// portfolio://cv/full                       → text/markdown, the full CV
// portfolio://projects/{slug}/readme         → text/markdown, project README
// portfolio://projects/{slug}/case-study     → text/markdown, project case study
// portfolio://courses/{slug}/certificate     → application/json, diploma metadata

Y los prompts codifican tres flujos de agente reutilizables. Cada uno se obtiene con prompts/get y se aplica a la conversación:

// pitch-for-role          → tailored pitch for a job description (uses sampling)
// tech-deep-dive          → focused walkthrough of one technology (uses sampling)
// compare-with-jd         → side-by-side fit analysis (uses sampling + elicitation)

La anotación «uses sampling» es real: pitch-for-role primero le pide al servidor hechos respaldados por tools y luego emite una petición de sampling de vuelta al cliente para componer el pitch usando el LLM del cliente. El servidor nunca llama a Anthropic por sí mismo — solo orquesta.

Patrón · Sampling

Sampling significa que el servidor le pide al cliente que ejecute una llamada al LLM. El cliente decide qué modelo, aplica sus propios rate-limits y filtros de seguridad, y devuelve la completion. Para un servidor de portfolio esto sale gratis: yo no pago por tokens, paga el cliente (Claude Desktop, mi agente, etc.). Para tooling interno es cómo dejas que un servidor componga resúmenes sin darle su propia API key.

El agente de chat: AI SDK v6 + MCP in-process

El chat del widget (la página /widget) habla con /api/agent. Esa ruta construye un ToolLoopAgent del AI SDK v6, le pasa las tools descubiertas por MCP y devuelve la respuesta en streaming como un UI Message Stream.

Lo ingenioso es cómo descubre las tools. Hay dos clientes MCP en este codebase:

  1. Un cliente HTTP para servidores MCP externos (p. ej. GitHub MCP, si quisieras enchufarlo).
  2. Un cliente in-process para el servidor MCP del portfolio, ya que vive en el mismo proceso de Node que la ruta del agente.

Figura 3 · El arreglo que más importó. La primera versión de /api/agent hacía una auto-llamada HTTP a /api/mcp. Eso funciona en local pero en Vercel las dos funciones caen en workers distintos, el transporte MCP tiene estado, y la segunda llamada falla con «Server not initialized». El arreglo: enlazar cliente y servidor con el InMemoryTransport del SDK, todo en el mismo proceso.

El puente in-process son doce líneas:

// apps/web/lib/mcp-client.ts
import { Client } from '@modelcontextprotocol/sdk/client/index.js';
import { InMemoryTransport } from '@modelcontextprotocol/sdk/inMemory.js';
import { buildServer } from '@neryc/portfolio-mcp/server';

export async function connectInProcessPortfolio(name: string) {
  const [clientTransport, serverTransport] = InMemoryTransport.createLinkedPair();
  const { server } = buildServer();
  await server.connect(serverTransport);
  const client = new Client(
    { name: 'web-host', version: '0.0.0' },
    { capabilities: { sampling: {}, elicitation: {} } },
  );
  await client.connect(clientTransport);
  return { name, client };
}

El agente en sí es corto. Descubre las tools, las envuelve para que el AI SDK pueda llamarlas y se las pasa a ToolLoopAgent:

// apps/web/lib/agent.ts
export function buildAgent(discovered: DiscoveredTool[]) {
  const aiTools = Object.fromEntries(
    discovered.map((t) => [
      t.name,
      tool({
        description: t.description,
        // The MCP server returns JSON Schema, not Zod. Wrap with jsonSchema()
        // so the AI SDK knows it's a schema-like object, not a Zod instance.
        inputSchema: jsonSchema(t.inputSchema) as never,
        execute: async (args) => {
          const result = await t.invoke(args);
          const first = result.content[0];
          if (first?.type === 'text') {
            try { return JSON.parse(first.text); } catch { return { text: first.text }; }
          }
          return result;
        },
      }),
    ]),
  );

  return new ToolLoopAgent({
    model: 'anthropic/claude-sonnet-4-6', // routes through Vercel AI Gateway
    instructions: SYSTEM_PROMPT,
    tools: aiTools,
    stopWhen: stepCountIs(10),
  });
}

Y la ruta son veinte líneas, en su mayoría rate-limiting y conversión de mensajes:

// apps/web/app/api/agent/route.ts
export const runtime = 'nodejs';
export const maxDuration = 300;

let _ctxPromise: Promise<AgentContext> | null = null;

function getAgentContext() {
  if (_ctxPromise) return _ctxPromise;
  _ctxPromise = (async () => {
    const portfolio = await connectInProcessPortfolio('portfolio');
    const tools = await discoverToolsFromServers([portfolio]);
    return { agent: buildAgent(tools) };
  })();
  return _ctxPromise;
}

export async function POST(req: NextRequest) {
  const ip = req.headers.get('x-forwarded-for')?.split(',')[0] ?? 'anon';
  const limit = await rateLimit(ip);
  if (!limit.allowed) return new Response(JSON.stringify({ error: 'rate-limited' }), { status: 429 });

  const { messages } = await req.json() as { messages: UIMessage[] };
  const { agent } = await getAgentContext();
  const modelMessages = await convertToModelMessages(messages);
  const result = await agent.stream({ messages: modelMessages });
  return result.toUIMessageStreamResponse();
}

El contexto del agente está memoizado a nivel de módulo, así que las invocaciones en caliente reutilizan el cliente MCP in-process. En Vercel el worker se recicla cada pocos minutos; el costo de cold-start es de ~50 ms.

El widget embebible: Preact + esbuild + una sola etiqueta <script>

Para que el FAB encaje en cualquier página HTML (incluyendo mi portfolio estático en GitHub Pages), el bundle del widget tiene tres restricciones:

  1. Tiene que caber en un presupuesto. 13 KB minificado, sin React, sin Tailwind enviado al host.
  2. Tiene que aislar los estilos. Si la página host tiene resets de CSS agresivos o sus propias clases utilitarias, el widget no puede romperse. Meter la UI pesada dentro de un iframe esquiva ese problema por completo.
  3. Tiene que conocer su propio origen. El script se aloja en talk-to-my-portfolio.vercel.app/widget.js; el iframe necesita cargar desde el mismo host. Hardcodear la URL es frágil, así que el bundle lee document.currentScript.src al cargar y recorta el /widget.js.

El código completo tiene <100 líneas:

// packages/widget/src/index.tsx
import { h, render } from 'preact';
import { useState } from 'preact/hooks';

const WIDGET_HOST =
  (document.currentScript as HTMLScriptElement | null)?.src?.replace(/\/widget\.js.*$/, '') ??
  'https://talk-to-my-portfolio.vercel.app';

const COLOR_ACCENT = '#2E75B6';
const COLOR_ACCENT_HOVER = '#4FA9E8';
const COLOR_PANEL_BG = '#06101F';

function FAB() {
  const [open, setOpen] = useState(false);
  const [hover, setHover] = useState(false);
  return (
    <div style={{ position: 'fixed', bottom: 24, right: 24, zIndex: 999999, fontFamily: 'Inter, system-ui, sans-serif' }}>
      {open && (
        <div style={{
          width: 'min(440px, calc(100vw - 32px))',
          height: 'min(640px, calc(100vh - 96px))',
          marginBottom: 12, borderRadius: 16, overflow: 'hidden',
          boxShadow: '0 20px 40px -8px rgba(6,16,31,0.55), 0 8px 16px -4px rgba(46,117,182,0.25)',
          background: COLOR_PANEL_BG,
          border: '1px solid rgba(148, 163, 184, 0.14)',
        }}>
          <iframe src={`${WIDGET_HOST}/widget`} style={{ width: '100%', height: '100%', border: 0 }} />
        </div>

      <button onClick={() => setOpen(!open)} /* ...styled FAB... */>
        {open ? '×' : '💬'}

const mount = document.createElement('div');
mount.id = 'portfolio-widget-root';
document.body.appendChild(mount);
render(<FAB />, mount);

En el host (este portfolio), la integración es exactamente una línea encima de </body>:

<script src="https://talk-to-my-portfolio.vercel.app/widget.js" async defer></script>

Dos bytes de trabajo para el host, un archivo cacheado en CDN, sin lock-in de framework.

Los seis bugs que rompieron producción

Los tests locales pasaban. pnpm test reportaba 115 en verde. pnpm build compilaba. Hice push a Vercel y abrí /api/mcp desde curl — y obtuve un 500. Luego /api/agent también dio 500. Seis bugs separaban «funciona en local» de «está en vivo».

Lo que sigue es la cronología, en el orden en que aparecieron. Cada uno es del tipo de cosa que no sale en los unit tests porque tiene que ver con cómo la plataforma empaqueta y ejecuta el código, no con la lógica.

BUG 1
ENOENT: no puede abrir /var/task/packages/mcp-server/package.json
El buildServer() del servidor MCP lee su propio package.json al evaluar el módulo para obtener su nombre y versión. En local ese archivo existe. En Vercel, Turbopack empaqueta cada módulo de ruta en un único chunk en .next/server/..., y la ruta relativa original ../package.json no resuelve a ningún lado. La función serverless lanza una excepción en el import, así que la ruta ni siquiera llega a ejecutarse.
// BEFORE
const pkg = JSON.parse(
  readFileSync(join(dirname(fileURLToPath(import.meta.url)), '..', 'package.json'), 'utf-8')
);

// AFTER — JSON import that Turbopack inlines at build time
import pkg from '../package.json' with { type: 'json' };
BUG 2
La auto-llamada HTTP se cae en los workers serverless
La primera versión de /api/agent construía un cliente MCP por HTTP apuntando a su propia ruta /api/mcp. En local todo es un solo proceso de Node. En Vercel las dos rutas viven en funciones serverless separadas. La petición initialize llega al worker A y crea una sesión ahí; el tools/list siguiente llega al worker B, que no tiene sesión para ese ID y devuelve «Bad Request: Server not initialized».

El arreglo es el cliente MCP in-process que se ve en la Figura 3 de arriba: InMemoryTransport.createLinkedPair() conecta un cliente y un servidor dentro del mismo isolate de V8. Sin HTTP, sin pérdida de sesión, sin un segundo cold start.

BUG 3
Invalid prompt: messages do not match the ModelMessage[] schema
El cliente usa useChat de @ai-sdk/react, que envía UIMessage[] (basado en parts, con tokens con forma { type: 'text', text: '…' }). ToolLoopAgent.stream() quiere ModelMessage[]. El AI SDK v6 expone convertToModelMessages() como el puente de conversión — se me había olvidado llamarlo.
const modelMessages = await convertToModelMessages(messages);
const result = await agent.stream({ messages: modelMessages });
BUG 4
TypeError: e is not a function durante la ejecución de una tool
El servidor MCP devuelve los input schemas de las tools como objetos JSON Schema. El tool() del AI SDK espera o bien un schema de Zod (que tiene .parse) o un Schema creado con el wrapper jsonSchema() del SDK. Pasar el JSON Schema crudo funcionaba en tiempo de compilación pero explotaba en runtime cuando el SDK intentaba validar los inputs.
// BEFORE
inputSchema: t.inputSchema as never,

// AFTER
inputSchema: jsonSchema(t.inputSchema) as never,
BUG 5
SDK de Anthropic directo sin una API key
El agente originalmente fijaba un modelo con anthropic('claude-sonnet-4-6'), que llama a Anthropic directamente y necesita ANTHROPIC_API_KEY. Todo el portfolio corre a través de Vercel AI Gateway (una key, failover automático de proveedor, dashboard de uso). Cambiar a la forma de modelo-string deja que el AI SDK v6 enrute por el Gateway siempre que AI_GATEWAY_API_KEY esté configurada.
// BEFORE — direct Anthropic
model: anthropic('claude-sonnet-4-6'),

// AFTER — routed through Vercel AI Gateway
model: 'anthropic/claude-sonnet-4-6',
BUG 6
ENOENT: no puede abrir packages/shared/data/projects/_index.json
El cargador de datos lee archivos JSON y Markdown de packages/shared/data/ vía readFileSync. El rastreador de dependencias de Next.js sigue los imports estáticos, así que envía cada archivo JS importado — pero las llamadas crudas a readFileSync son opacas para el rastreador, y el directorio de datos quedó fuera del bundle de despliegue.
// apps/web/next.config.ts
const nextConfig: NextConfig = {
  outputFileTracingRoot: path.join(process.cwd(), '..', '..'),
  outputFileTracingIncludes: {
    '/api/mcp/[[...path]]': ['../../packages/shared/data/**/*'],
    '/api/agent': ['../../packages/shared/data/**/*'],
  },
};

Seis arreglos, seis commits, todos en vivo. El diff completo son unas 80 líneas de código repartidas entre los seis commits, pero cada línea es la diferencia entre «despliega» y «produce un 500 en la primera petición».

Para llevar

Los tests de producción no sustituyen a los unit tests; son ortogonales. De estos seis bugs, cero eran detectables desde pnpm test porque todos viven en la frontera entre el código de la aplicación y la plataforma — layout del sistema de archivos, fronteras de los workers serverless, tracing del bundler, contratos de variables de entorno. La forma más barata de cazarlos es desplegar un preview pronto y golpear los endpoints con curl.

El bucle de pulido de UX

Una vez que la API funcionaba, el widget en sí no se parecía en nada a un producto terminado. La primera versión tenía Geist como tipografía, la salida JSON cruda de las tools renderizada inline, celdas de tabla partiéndose a mitad de palabra a 360 px de ancho, y un tema claro genérico que chocaba con el portfolio oscuro. Seis iteraciones más lo llevaron a la versión embebida en esta página.

Figura 4 · Antes / después. Mismos datos, mismo modelo, mismo prompt. La izquierda es el primer despliegue: JSON crudo en el chat, tabla rota al ajustarse, cortes de palabra a mitad de carácter, y Geist cayendo al serif por defecto del navegador porque --font-sans: var(--font-sans) era una variable CSS circular. La derecha es la versión en vivo hoy: tipografía Inter, internos de las tools ocultos, tarjeta del asistente a ancho completo, tokens de tecnología inline con brillo cian.

1 · Arreglar la tipografía

globals.css tenía --font-sans: var(--font-sans); — una auto-referencia. Tailwind emitía tan campante un CSS que resolvía a nada, así que el navegador caía a Times. Cambié a Inter + JetBrains Mono vía next/font/google con stacks de fallback de fuentes de sistema apropiados. Resuelto en dos líneas.

2 · Ocultar los internos de las tools

La primera versión renderizaba cada llamada a tool como una tarjeta con la salida JSON cruda debajo. Útil para depurar, UX pésima. Ahora MessageList filtra por completo las parts dynamic-tool y tool-*. El agente sigue llamando a las tools — quien visita simplemente no ve la fontanería.

3 · Tablas que no aplastan las palabras

A 360 px de ancho, overflow-wrap: anywhere en td/th partía palabras sueltas en los límites de carácter: «Proficient» se volvía «Profic / ient». Cambié al ajuste por límite de palabra por defecto más prose-table:block prose-table:overflow-x-auto para que las tablas más anchas que la columna scrolleen en horizontal en vez de quedar aplastadas.

4 · Igualar la paleta del portfolio padre

El widget tenía que sentirse como parte del portfolio, no como un elemento ajeno. Levanté las variables CSS del portfolio tal cual — #06101F azul marino profundo, #2E75B6 azul primario, #7DD3FC brillo de acento — al tema oscuro del chat. El <code> inline ahora toma el mismo brillo cian + fondo tintado que en las tarjetas de proyecto del portfolio.

5 · Estado vacío con preguntas sugeridas

Antes, abrir el FAB te daba un textarea en blanco. Después, obtienes cuatro chips de sugerencia clicables que siembran la conversación en un toque. Quien recluta tiende a preguntar las mismas cuatro cosas con otras palabras («¿cuál es su stack?», «¿proyectos de IA?», «¿dónde trabaja?», «¿está disponible?»), así que convertirlas en botones elimina el problema del arranque en frío.

6 · Recortar el markdown incompleto durante el streaming

El agente emite tokens uno a uno a través de ReactMarkdown. Durante un frame o dos en cada ciclo, el modelo emite markdown parcial como [Ver diploma] antes de que aterrice (https://drive.google.com/...). ReactMarkdown renderiza el texto crudo y un instante después salta a un enlace con estilo — visible como un parpadeo en cada enlace. Un pequeño helper trimIncompleteMarkdown() recorta cualquier enlace/código/énfasis sin terminar al final de la última text part del mensaje del asistente que aún está en streaming. El enlace aparece de una sola vez en cuanto llega el token de cierre.

function trimIncompleteMarkdown(text: string): string {
  let result = text;
  // Unclosed link: [...] or [...](unclosed
  const lastBracket = result.lastIndexOf('[');
  if (lastBracket !== -1) {
    const tail = result.slice(lastBracket);
    if (!/^\[[^\]]*\]\([^)]+\)/.test(tail)) result = result.slice(0, lastBracket);
  }
  // Unclosed inline code: odd backtick count
  const matches = result.match(/`/g);
  if (matches && matches.length % 2 === 1) {
    result = result.slice(0, result.lastIndexOf('`'));
  }
  // Unclosed bold/italic at end
  return result.replace(/(\*{1,2})$/, '');
}

Cómo se ve la capa de datos

164 cursos de Platzi son muchos datos estructurados para mantener a mano, así que no lo hice. scripts/sync-platzi es un bot de Playwright que inicia sesión en Platzi una vez (interactivo, guarda la sesión en .platzi-session/), y en las corridas siguientes recorre el perfil público y el dashboard de aprendizaje, fusiona ambas vistas (el perfil público lista los cursos completados con diplomas; el dashboard tiene las fechas de finalización exactas) y escribe el resultado fusionado en packages/shared/data/courses.json.

pnpm --filter sync-platzi exec playwright install chromium  # first run only
pnpm --filter sync-platzi sync                              # subsequent runs are headless

La fusión es la parte interesante. Las entradas del perfil público son la autoridad sobre qué cursos he hecho; las del dashboard son la autoridad sobre cuándo. Unirlas me da un único registro por curso con ambos campos, más la URL del diploma extraída del elemento de enlace del perfil público.

Cada registro valida contra un schema de Zod en packages/shared/src/schemas/course.ts, así que el servidor MCP puede devolver datos con tipos seguros sin re-validar en cada llamada:

export const CourseSchema = z.object({
  slug: z.string(),
  title: z.string(),
  diplomaUrl: z.url().optional(),
  completedAt: z.iso.date().optional(),
  skills: z.array(z.string()),
  category: z.enum(['frontend','backend','ai','data','devops','soft-skills','other']),
});

Si Platzi cambia su HTML, el scraper se rompe ruidosamente con un error de parseo de Zod en vez de devolver datos basura en silencio. Eso ya ha sido una baranda de seguridad útil dos veces.

Tests y CI

El proyecto trae 115 tests repartidos en cuatro paquetes. La división interesante es entre la suite completa y el subconjunto @critical:

pnpm test            # full: 115 tests (3 skipped)
pnpm test:critical   # @critical only: 27 tests (sampling, elicitation, capabilities)

Los tests críticos se etiquetan envolviendo su bloque describe con @critical:

describe('@critical pitch-for-role with sampling', () => {
  // ...
});

CI ejecuta pnpm test:critical como un paso separado antes de la suite completa. Si el contrato de MCP sufre una regresión (el sampling deja de funcionar, la elicitation se rompe, la negociación de capabilities falla), el pipeline se detiene en el paso de control con un mensaje obvio — en vez de enterrar el fallo dentro de una corrida de 115 tests.

El workflow real de CI es también donde aparecieron dos bugs de tipo producción que no se veían en local:

  1. Los consumidores del workspace no pueden hacer typecheck antes del build. scripts/sync-platzi depende de @neryc/portfolio-shared, que exporta desde ./dist/index.js. Un checkout limpio de CI no tiene directorio dist/, así que tsc falla con TS2307. Reordené el workflow para correr pnpm build antes de pnpm typecheck; pnpm -r build respeta la topología de dependencias, así que shared termina antes que sus consumidores.
  2. Conflicto de versión de pnpm. El workflow tenía version: 9 en pnpm/action-setup@v4; package.json tenía packageManager: pnpm@9.12.0. Ambos puestos → ERR_PNPM_BAD_PM_VERSION. Quité la versión a nivel de action para que el campo packageManager sea la única fuente de verdad.

Qué haría distinto

Tres cosas que cambiaría si empezara de cero hoy:

  1. Desplegar en el día cero. Los seis bugs de producción habrían aparecido una semana antes si hubiera puesto un preview de Vercel detrás de un PR en borrador desde el primer commit. Local + tests es necesario, no suficiente.
  2. No leer archivos al cargar el módulo. readFileSync arriba de un módulo de servidor es un disparo al pie en serverless empaquetado. Imports de JSON + outputFileTracingIncludes para el puñado de lecturas dinámicas (archivos Markdown por slug) es más declarativo y sobrevive al tracing del bundler.
  3. Adoptar InMemoryTransport desde el principio. El patrón de auto-llamada HTTP se ve correcto en un diagrama pero pierde contra las realidades de las fronteras de los workers serverless. Los clientes co-ubicados deberían usar el transporte en memoria del SDK. El endpoint MCP alojado sigue expuesto para consumidores externos (Claude Desktop, etc.); mi propio agente simplemente no pasa por él.

El patrón, generalizado

Este proyecto es una plantilla útil para cualquier «feature de IA embebible» que quieras enchufar a un sitio estático o al host de otra persona. La forma es:

  1. El backend expone un modelo de datos estructurado vía MCP (tools / resources / prompts). Vive detrás de un único endpoint HTTP y un binario stdio; un camino de código, dos transportes.
  2. El agente corre en el mismo despliegue de Vercel, habla con el servidor MCP in-process vía InMemoryTransport, y envía las respuestas en streaming al navegador.
  3. El widget es un bundle diminuto de Preact + esbuild que carga desde tu origen, monta un FAB y abre un iframe en /widget para la UI pesada. Una etiqueta <script> en el host.

~ 2.400 líneas de TypeScript en total. Cinco tardes de trabajo. El contrato de MCP es portable — Claude Desktop puede conectarse al mismísimo servidor mañana, sin cambios.
