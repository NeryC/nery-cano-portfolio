MCP from zero: what it is, what it solves, and when to use it

One protocol, three primitives, and a mandatory greeting. Everything you need to understand about MCP before writing a line of code.

MCP is an open standard that lets an application with a language model inside it ask external programs for data and actions. This guide covers what it is, the problem it was built to solve, how it works underneath, and when it is worth using. Nothing assumed.

You have an AI assistant that answers well about the world and knows nothing about your work. You want to give it access to your database, your repository and your calendar, so you write three integrations. They work. A month later your team starts using a different assistant, and those three integrations are worthless: they were written against the first one's API. You write three more.

That is the whole problem, and it is not the one it looks like. Connecting a model to a tool is easy. The hard part is making that connection work for more than one model, and letting someone who is not you write the tool without coordinating with you at all.

MCP (Model Context Protocol) is Anthropic's answer to that. They published it in November 2024 as an open standard, and today it is implemented by Claude, several IDEs, and a long list of third-party servers. By the end of this guide you will be able to explain what it is, what it solves, which messages travel over the wire, and in what cases it is not worth it.

Eight words, first

If you have never touched this, these eight terms are enough to read everything below. None of them is explained using another one I have not defined yet.

Language model (LLM). The engine behind Claude or ChatGPT. It predicts text. It knows an enormous amount about the world up to its training date, and nothing about your private data.

Context. All the text you hand the model in one query: your question, the prior conversation, and any data you attach. It is the only thing the model can see at that moment.

Protocol. An agreement on how two programs talk: which messages exist, what shape they have, and who speaks first. HTTP is a protocol, which is why any browser understands any web server without the two ever having met.

Client and server. Two roles, not two machines. The client asks, the server answers. They can sit on different continents or be two processes on your laptop.

JSON-RPC 2.0. A convention for asking another program to run a function, written in JSON. You send a method name and its arguments, you get back a result or an error. That is all it is.

Tool. A function the model can ask to have executed: query a database, send an email, search the web. The model does not run it. It asks for it.

Host. The application you interact with, the one with the model inside: Claude Desktop, your IDE, your own app. It is in charge.

Transport. The pipe the messages travel through. It can be the standard input and output of a local process, or HTTP against a remote server.

What MCP is, in one paragraph

MCP is an open protocol that standardizes how an application with an LLM inside asks external programs for context and actions. It defines three things: the message format (JSON-RPC 2.0), a mandatory opening where both sides state what they can do, and a catalog of what a server can offer (data to read, functions to run, templates to trigger). A server you write once works with any client that speaks the protocol, and that client knows nothing about your code.

The analogy the official documentation uses is USB-C: one connector for everything, instead of a proprietary charger per device. It is useful for grasping the shape of the problem, and it should be cashed out immediately. USB-C does not teach your laptop what a monitor is: it defines the connector, the voltages, and how the two negotiate who supplies power. The monitor is still responsible for being a monitor. MCP does exactly that: it knows nothing about your database, it defines the socket and the negotiation. Your server is still responsible for knowing what an invoice is.

Figure 1 · The journey of a question. The model never touches your data directly: it asks the host, which asks a server through the MCP client.

The problem it solves, and why not the alternatives

Before MCP, every AI app had its own way of reaching external tools. If you wrote a Postgres integration, it served that app. For the next one you rewrote it against a different API, with a different schema format and a different lifecycle. With M applications and N tools you need M×N integrations, and each new app costs N.

A protocol in the middle changes the arithmetic. Each app implements the protocol once, each tool implements it once, and you are left with M+N pieces. The gap between 9 and 6 is unimpressive at small numbers. At 20 applications and 100 tools, it is the gap between 2000 integrations and 120.

Figure 2 · From M×N to M+N. On the left, every app you add costs three new integrations; on the right it costs one, and it works with every server that already exists.

The interesting part is not the saved code: it is who gets to write it. Under M×N, the Postgres integration has to be written by each app's team, since it is bolted to their internal API. Under M+N, it is written once by whoever knows Postgres best, and everyone uses it. That is what made the ecosystem grow.

Now, MCP is not the only way out, and it is not always the best one. The real alternatives:

ApproachHow it worksWhere it breaks
Paste it all in the promptYou dump your data into the context and move onIt does not fit, it costs tokens every single time, and the data is frozen at paste time. Useless for actions.
Hand-rolled function callingYou declare functions in the model's API and run them yourselfIt works fine, and it is the foundation of all of this. The catch is that you wrote it against one vendor: switching models or sharing the integration means a rewrite.
Proprietary pluginsA single vendor's format (GPT Actions, for example)Works inside that garden. Your integration does not exist outside it, and the garden's owner sets the rules.
A frameworkLangChain tools, AI SDK tools, and friendsIt abstracts the provider, not the framework. Your tool lives in Python or TypeScript, inside that ecosystem, running in your process.
MCPA separate process speaking a public protocolIt costs you an extra process, a handshake and latency. With one app and one integration, it is infrastructure you do not need.

Notice that the first four rows are not mistakes. They are reasonable, and in plenty of projects they are the right answer. MCP starts paying off once an integration has to outlive the app it was born in.

When NOT to use it

One app, one tool, one team, no plans to open it up: use direct function calling and save yourself the process. MCP only starts returning what it costs once there is more than one consumer, or once the person writing the tool is not the person writing the app.

The anatomy: host, client, server

Three roles, and it pays not to mix them up, since the names are misleading.

The host is the application you see, the one holding the model: Claude Desktop, an IDE, your own site. It is in charge. It picks which servers to connect to, what to show you, and what to authorize.

The client lives inside the host and is the piece that speaks the protocol. Here is the counterintuitive bit: the host opens one client per server, and that relationship is one to one, forever. Three connected servers, three clients. There is no single client multiplexing.

The server is a separate program exposing capabilities: it can be a binary running on your machine or a service in the cloud. It does not know who is calling, and it does not care.

Figure 3 · Who is who. The host is the application you see; inside it opens one client per server, and that relationship is always one to one.

A server can offer three things, and what separates them is not technical but who decides to use them. This table is the one that took me longest to understand and the one that orders everything else:

PrimitiveWhat it isWho triggers it
toolsFunctions with effects: write, send, bookThe model, when it decides it needs them
resourcesData to read, identified by a URIThe application, deciding what to load into context
promptsConversation templates with argumentsThe person, picking one from a menu

And three capabilities run the other way, from client to server: sampling (the server asks the client's model for a completion), elicitation (the server asks the person for a value) and roots (the client tells the server which folders it may work in). All three are covered below.

The pieces, in detail

That was the map. Now each piece on its own, in enough detail that you can read the specification without a translator.

JSON-RPC 2.0: the language

MCP did not invent a message format: it uses JSON-RPC 2.0, which has been around since 2010 and fits on one page. There are three kinds of message and none of them has a trick to it.

A request carries an id and expects an answer:

{ "jsonrpc": "2.0", "id": 7, "method": "tools/call",
  "params": { "name": "getWeather", "arguments": { "city": "Asunción" } } }

A response comes back with the same id, carrying result or error, never both:

{ "jsonrpc": "2.0", "id": 7,
  "result": { "content": [{ "type": "text", "text": "24°C and clear" }] } }

A notification has no id, so nobody answers it. It exists to inform, not to ask:

{ "jsonrpc": "2.0", "method": "notifications/tools/list_changed" }

That is the entire language. Next time you read "the server sends a notification that the tool list changed", you already know it is a one-line JSON with no id. And since this is JSON-RPC and not REST, there are no verbs and no routes: there are named methods, like tools/list, resources/read or prompts/get.

The lifecycle: nobody works before introductions

Every MCP connection starts the same way, and it is mandatory. The client sends initialize with the protocol version it speaks and its capabilities. The server replies with the version it accepts and its own. The client sends a notifications/initialized notification, and only then does real work begin.

Figure 4 · The handshake. Three messages before any useful work: watch the dots travel. If the client does not declare sampling here, the server already knows it cannot ask for it later.

This exchange looks like paperwork and it is the most important piece of the design. It is called capability negotiation, and it means both sides agree on which features exist in this session before using them. A server that wants a completion from the client's model checks whether the client declared sampling. If it did not, the server has to work anyway, down a different path. No discovery by trial and error, no failures halfway through: it is settled at message one.

It is also where the version gets agreed. The client asks for the newest one it knows; if the server does not have it, it answers with its own newest and the client decides whether to continue or hang up. Over HTTP, the agreed version then rides along in an MCP-Protocol-Version header on every later request.

Pattern · Negotiate before you use

Declaring capabilities in the greeting and assuming nothing afterwards is an old pattern (SMTP and TLS do the same) and it is free to steal. If you are designing any integration between two systems that evolve separately, this saves you half your compatibility bugs.

Tools: what the model can ask for

A tool is a function the server exposes with three things: a name, a prose description, and a JSON schema of its arguments. The client fetches the catalog with tools/list and hands it to the model. When the model decides to use one, the client sends tools/call.

The description is not documentation: it is the prompt. It is the text the model reads to decide whether this tool fits what it was asked. A vague description ("searches things") produces a model that calls it when it should not, or skips it when it should. Write it for its actual reader, an LLM with no context beyond that line.

Figure 5 · A tool call, end to end. Follow the dot: step 2 is the one most people forget, and it is the only place a human can say no.

Step 2 in that figure deserves attention. The model asks, it does not execute. Between the request and the execution sits the host, and that is where a person authorizes. The specification is explicit: the host must obtain consent before invoking a tool, and descriptions from a server you do not trust are treated as untrusted text. A tool is arbitrary code execution under a friendlier name.

Resources: data with a URI

A resource is data to read, identified by a URI you invent: portfolio://cv/full, file:///notes.md, postgres://table/customers. Clients list them with resources/list and read them with resources/read. They have no side effects: reading one twice changes nothing.

What separates a resource from a tool is who decides. A tool is picked by the model mid-reasoning. A resource is picked by the application (or by you, in the UI) to drop into the context before asking. That is why clients tend to render resources as attachments and tools as actions.

There is a trap here that almost nobody explains and that bit me in my own server. Resources come in two forms: concrete (a fixed URI, like portfolio://cv/full) and templates (a pattern with variables, like portfolio://courses/{slug}/certificate). Writing the server, you think in templates, since they are four lines of code. When a client calls resources/list, it receives the expanded list, one entry per item that exists. Four templates in the code can be 171 entries on the wire. I come back to this in Part 2, because the real number is 171 and I spent months writing "4" in my own README.

toolsresources
EffectsYes, it can write and charge moneyNo, read only
Chosen byThe modelThe application or the person
Identified byNameURI
If in doubtDoes it have effects, or does picking it require reasoning? Tool. Is it a document you read whole? Resource.

Prompts: what the person picks

A prompt is a conversation template with arguments, exposed by the server and triggered by the person. In Claude Desktop they show up as commands in a menu. The server lists them with prompts/list and returns the assembled messages with prompts/get.

They are the most ignored primitive, and that is a shame: this is where whoever wrote the server can package up how to use it well. Whoever built the Postgres server knows better than you how to ask a model to read an execution plan. A prompt is that recipe, versioned alongside the server, instead of a paragraph in a README nobody opens.

The other direction: sampling, elicitation and roots

The three primitives above run server to client. These three run the other way, and they are what makes MCP more than a catalog of functions.

Sampling. The server asks the client to run a completion with its model, by sending sampling/createMessage. It sounds strange until you see what it is for: your server needs intelligence (summarize, classify, compare) and would rather not depend on holding its own API key. With sampling, the server borrows the model the user is already paying for.

Figure 6 · Sampling flips the arrow. The server borrows intelligence from the client's model, so it needs no API key of its own and bills you no tokens.

The design has a deliberate limit: the client can inspect, edit or reject what the server asked for, and the specification says the user must approve sampling requests. A server cannot use your model behind your back.

Elicitation. The server asks the person for a value mid-operation, with a schema of what it needs. It fits when information is missing that the model must not invent: your email, which time slot you prefer, which of these three options. The answer has three possible shapes (accept, decline, cancel) and your server has to handle all three, since the person can close the dialog and get on with their day.

Roots. The client tells the server which folders or URIs it is allowed to work on. It is a limit rather than a capability: it shrinks the server's world.

The pattern behind all three is the same, and it is my favorite part of the design: a server can ask, never take. It asks for intelligence, it asks for data, it asks for permission. The client can always say no, and the server is obliged to have a plan B. A server that breaks when the client lacks sampling is a badly written server.

Transports: where the server lives

The specification defines two standard transports. The messages are identical across both; what changes is where the server runs and who can reach it.

Figure 7 · The two transports. What changes is where the server lives and who can reach it; the messages travelling inside are exactly the same.

stdio. The client launches the server as a subprocess and talks to it over standard input and output, one JSON message per line. No network, no ports, no authentication: the server runs with your permissions, on your machine. It is the transport the specification says clients should support whenever possible, and the one nearly every local server uses. A detail that bites: the server must not write anything to stdout that is not protocol, so your debug console.log goes to stderr or it corrupts the session.

Streamable HTTP. The server is an independent service with a single endpoint handling POST and GET. The client sends messages by POST; the server answers with a JSON body or opens an SSE stream if it needs to send several things. The session, when there is one, rides in an Mcp-Session-Id header. It replaces the old HTTP+SSE transport from version 2024-11-05, now deprecated and kept only for compatibility.

Security: why the host keeps asking

An MCP server is code running requests that came from a model. The specification cannot enforce security at the protocol layer, so it leans on the host and states explicit principles: the person must consent to every data access and every tool invocation; the host must not ship your data elsewhere without permission; descriptions from an untrusted server are treated as hostile text.

That last point is worth a pause. A tool's description enters the model's context. A malicious server can write a description engineered to make the model do something else. Connecting a stranger's MCP server is equivalent to installing a stranger's browser extension, except here you are also handing it an LLM with access to your other tools.

If you are exposing a server over HTTP, the specification demands two concrete things: validate the Origin header (so a random web page cannot talk to your local server through the victim's browser) and, locally, bind to 127.0.0.1 rather than 0.0.0.0.

The minimal MCP, in code

A whole working server with a single tool. This is TypeScript with the official SDK (@modelcontextprotocol/sdk, the 1.x line, which today is the production one) over stdio. Nothing is elided: this runs.

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "weather", version: "1.0.0" });

server.registerTool(
  "getWeather",
  {
    title: "Current weather",
    description:
      "Return the current weather for a city. Use it when asked about temperature or forecast for a specific place.",
    inputSchema: { city: z.string().describe("City name") },
  },
  async ({ city }) => ({
    content: [{ type: "text", text: `It is 24°C and clear in ${city}.` }],
  })
);

await server.connect(new StdioServerTransport());

Twenty lines and it is a legitimate MCP server. The SDK handles the handshake, publishes the capabilities, converts your Zod schema to JSON Schema, and answers tools/list. You write the name, the description and the function.

Wiring it into Claude Desktop takes no publishing at all: you declare it in the config file, and the host launches it as a subprocess when it needs it.

{
  "mcpServers": {
    "weather": { "command": "node", "args": ["/absolute/path/dist/server.js"] }
  }
}

On the other side, a minimal client that launches it, lists its tools and calls one:

import { Client } from "@modelcontextprotocol/sdk/client/index.js";
import { StdioClientTransport } from "@modelcontextprotocol/sdk/client/stdio.js";

const transport = new StdioClientTransport({
  command: "node",
  args: ["/absolute/path/dist/server.js"],
});

const client = new Client({ name: "my-host", version: "1.0.0" });
await client.connect(transport);          // runs the full handshake

const { tools } = await client.listTools();
console.error(tools.map((t) => t.name));  // ["getWeather"]

const r = await client.callTool({ name: "getWeather", arguments: { city: "Asunción" } });
console.error(r.content);

That client.connect() is the whole of Figure 4: it sends initialize, waits for the reply, sends notifications/initialized. The SDK hides it well, which is exactly why it helps to have seen the three messages first.

On versions

This guide goes out on July 17, 2026. The current specification is 2025-11-25, and a 2026-07-28 revision is in release candidate: it makes the protocol core stateless and lands in eleven days, alongside v2 of the SDK (which splits the package into @modelcontextprotocol/server and @modelcontextprotocol/client). v1 stays the recommended production line and keeps support for at least six more months. The concepts in this guide do not move. The import lines do.

The pattern, generalized

Strip MCP away and this is what is left: when you have M consumers and N providers talking pairwise, a protocol in the middle turns M×N into M+N, and what matters is not the saved lines but that the piece can now be written by whoever knows the domain best, without asking anyone's permission.

This is not the first time it happens. The specification itself says it draws on LSP, the protocol Microsoft shipped for editors: before it, every editor implemented autocomplete for every language (M×N); after it, each language wrote a server and each editor a client (M+N). Same movie as the printer driver, as ODBC for databases, as USB.

The signal that you are looking at this pattern is easy to spot: you open your integrations folder and see the same adapter written three times in slightly different shapes, and every new consumer means writing all the adapters again. That is where a protocol pays. The counter-signal matters just as much: if you have one consumer and that is not going to change, a protocol is a layer of indirection that buys you latency and one more process that can fall over.

The litmus test for this guide: if you can explain to someone what MCP is, what it solves and when not to use it, you are done. The rest is SDK.
