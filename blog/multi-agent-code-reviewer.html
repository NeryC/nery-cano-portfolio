Multi-agent systems from zero: how to make several AI agents work together

An agent isn't a bigger model or a longer prompt. It's a model placed inside a loop, with permission to act and a condition to stop. When one agent isn't enough, multi-agent systems appear. This guide explains both from zero.

Five words, first

Before anything else, five terms that recur through the whole text. Get these straight and the rest reads on its own.

  • Agent. A language model (LLM) placed in a loop: it looks at the state, decides an action, runs it, observes the result, and repeats until done.
  • Tool. A function the agent can call to do something the model alone can't: search the web, read a file, query a database, run code.
  • Loop (agent loop). The decide→act→observe repetition. It's what separates an agent from a single model call.
  • Orchestration. The logic that splits work across several agents and puts back together what they return. In multi-agent, it's the glue.
  • Supervisor. An agent whose job is to review and consolidate the output of other agents: deduplicate, prioritize, summarize.

What an agent is, in one paragraph

A normal LLM call is a round trip: you send text, it returns text, done. An agent breaks that shape. You give the model a goal and a set of tools, and you let it run in a loop: the model decides which tool to use, you run that tool, you hand the result back, and the model decides the next step with that new information. The loop keeps going until the model says "done" or until it hits a limit you set. That ability to act, see the result, and correct is all it takes to be an agent.

Figure 1 · The difference is the loop. A normal call ends in one step; an agent decides, acts with a tool, observes the result, and decides again until the goal is met.

The problem they solve, and why not the alternatives

The honest question is: if an LLM is already capable, why bother with loops and tools? Three concrete reasons.

A model alone can't act. It can write the text of an email but not send it; it can describe a SQL query but not run it against your database. Tools give it hands. The loop gives it the chance to see whether what it did worked, and retry.

A giant prompt doesn't scale. The temptation is to cram everything into one huge instruction: "analyze security, performance, and maintainability, and also summarize and score." It works on toy examples. As soon as the problem grows, the model spreads thin: it does all three halfway and contradicts itself. Splitting the work into steps with state is what keeps it focused.

A model isn't an expert at everything at once. Asking it in the same turn to think like a security attacker and a performance engineer lowers its quality at both. Giving it one clear role at a time raises the level of each answer. That's where the idea of several specialized agents comes from, which is the subject of Part 2.

The classic alternatives (a function with fixed rules, a data pipeline, a single model call) are still better when the problem is fixed and known. The agent wins when the path isn't known ahead of time and decisions have to be made along the way.

How it works: the anatomy

Every agent, however elaborate, is built from four pieces. It's worth seeing them separately before putting them together.

  • A goal and a role. The system prompt that tells it who it is and what it has to achieve. "You are a security reviewer; find vulnerabilities" is a role; "analyze this code" is a goal.
  • Tools. The set of functions it can call. Without tools, an agent can only think out loud. With them, it can read, write, search, and run.
  • Memory (context). What the agent "remembers" while it works: the history of what it already tried and observed. Without memory, it would repeat the same step forever.
  • A stop condition. How it knows it's done: because the model declares it, because it hit a maximum number of steps, or because it spent the budget. Without a limit, an agent can loop.

Figure 2 · The four pieces in motion. The role sets the goal; the loop decides, acts with tools, and observes; memory keeps the history; the stop condition cuts the cycle.

The pieces, in detail

With the map in view, now the slow walk. This guide assumes you already know how a single agent works: its loop, its tools, its memory, and its stop condition. If that is new, Agent loops from zero (Part 1 of the research-agent series) covers it from scratch. Here we go straight to what changes when there are several.

From one to many: multi-agent systems

A multi-agent system is, without mystery, several agents collaborating on a problem too big for one. The gain isn't "more models means better." The gain shows up when the problem splits into parts that are solved better separately: parts that run at the same time, or that need a different expert, or that are worth reviewing against each other. If your problem doesn't split like that, one well-built agent wins almost always.

There are six ways to coordinate them. They aren't exclusive (real systems mix several), but it's worth knowing them one at a time.

Figure 3 · The six patterns at a glance. They differ in who decides, who aggregates, and in what order the agents run. Most real systems combine two or more.

PatternWhen to use itWho coordinates
Orchestrator–workerA task that splits into subtasks not known ahead of timeThe orchestrator decides, delegates, and aggregates
Parallel (fan-out)Independent subtasks that can run at the same timeNobody, until the results are collected
Supervisor / aggregatorSeveral outputs that must be reconciled into oneOne agent consolidates: dedup, prioritize, summarize
Pipeline (sequential)One step's output is the next step's inputThe fixed order; no extra coordinator
Debate / critiqueRaise quality by having agents review each otherBack-and-forth rounds between agents
RoutingPick the right specialist based on the inputA router classifies and forwards

Orchestrator–worker. A central agent reads the problem, splits it into subtasks, and hands them to worker agents; then it puts together what they return. It's like a project lead. It works when the subtasks aren't known ahead of time and depend on what the orchestrator discovers when it reads the input.

Parallel (fan-out). The same input goes to several agents that work at the same time, without talking to each other. Because they don't coordinate during the run, it's the fastest pattern: total time is the slowest agent's, not the sum. It's the basis of the code reviewer in Part 2.

Supervisor / aggregator. Several agents produce partial outputs and a supervisor turns them into a single coherent answer: it removes duplicates, orders by priority, writes the summary. It almost always accompanies fan-out, because three unconsolidated lists of findings are of no use to anyone.

Pipeline (sequential). The agents chain up: the first produces something, the second takes it and transforms it, and so on. It works when each stage depends on the previous one: extract, then summarize, then translate. It's simple and predictable, but gains no speed because nothing runs in parallel.

Debate / critique. Two or more agents critique each other in rounds: one proposes, another points out flaws, the first corrects. It raises quality on reasoning tasks in exchange for more calls and more time. Useful when getting it wrong is expensive and the back-and-forth is worth it.

Routing. A lightweight agent looks at the input and decides who should handle it: "this is a billing question" → to the billing agent. It doesn't solve the problem; it picks who solves it. It's the pattern of assistants with many specialties.

Figure 4 · Fan-out + supervisor. The reviewer pattern from Part 2: the same input fans out to three specialists that run at the same time, and a supervisor merges their findings into a single report.

What a multi-agent costs

Splitting the work isn't free, and it's worth knowing the price before paying it. Money: each agent is at least one model call, so three specialists plus a supervisor are four calls per run, not one. Latency: fan-out helps because it runs in parallel, but a pipeline or a debate add up the times. Coordination: a new problem appears that one agent didn't have: merging outputs that don't match, resolving contradictions, deciding who wins. Failure modes: with more pieces there are more things that can break, and some fail silently (a supervisor that deletes too much, as you'll see in Part 2). The rule of thumb: start with one agent, and move to several only when the problem genuinely splits.

The takeaway

An agent is an LLM in a loop with tools and a brake. A multi-agent is worth it when the problem splits: into parallel tasks, into different experts, or into cross-checks. If it doesn't split, one well-built agent wins.

The minimal multi-agent, in code

The smallest version that teaches the fan-out + supervisor pattern: two specialists that run at the same time and a supervisor that consolidates. It's exactly the shape of the Part 2 project, without the infrastructure around it. It uses the AI SDK and Zod to force structured output.

import { generateObject } from 'ai';
import { z } from 'zod';

const Finding = z.object({
  title: z.string(),
  severity: z.enum(['low', 'medium', 'high']),
});

// One specialist = one call with a clear role and an output schema
async function specialist(role: string, code: string) {
  const { object } = await generateObject({
    model: REVIEW_MODEL,                 // e.g. claude-sonnet-4.6
    schema: z.array(Finding),
    system: `You are a ${role} reviewer. Return a list of findings.`,
    prompt: code,
  });
  return object;
}

// Fan-out: both run AT THE SAME TIME (Promise.all), not one after another
const [security, performance] = await Promise.all([
  specialist('security', code),
  specialist('performance', code),
]);

// Supervisor: one call that reconciles the two lists into a report
const { object: report } = await generateObject({
  model: SUPERVISOR_MODEL,               // a cheaper one is enough
  schema: z.object({ score: z.number(), findings: z.array(Finding) }),
  system: 'Consolidate the findings: remove duplicates and score 0-100.',
  prompt: JSON.stringify([...security, ...performance]),
});

That's the whole skeleton. Three things make the pattern: each specialist has a single role, the Promise.all runs them in parallel, and the supervisor gets outputs already structured by Zod, so it doesn't have to guess the format. The pattern, generalized

"Specialists in parallel + a supervisor that consolidates" isn't about reviewing code. It's the shape of any problem where several points of view look at the same thing and someone has to draw a single conclusion. A due diligence where one agent reads the legal side, another the financial, another the technical, and a supervisor writes the memo. A support triage where three agents classify by urgency, area, and sentiment, and one decides the queue. A moderation where different agents look for different kinds of harm and a supervisor merges the verdict.

The signal that this pattern applies is simple: when you catch yourself asking a single prompt to be an expert at several things at once, and the result comes out mediocre at all of them. That's when it pays to split into specialists and put someone in charge of consolidating. What that looks like in a real project (with its bugs) is Part 2.
